STANDARD SECTIONS (topics to cover in Part 1, adjust per problem)
======================================================================

--- Core Sections (always include): ---
1. Requirements Gathering       (functional + non-functional)
2. Back-of-Envelope Estimates   (throughput, storage, bandwidth)
3. High-Level Design            (architecture diagram + components)
4. Database Schema              (core tables + indexes + reasoning)
5. API Design                   (REST + real-time if applicable)

--- Problem-specific Sections (6-{N-1}): ---
   Examples per problem:
   - URL Shortener: 301 vs 302, Key Generation, Caching, Analytics, Custom Aliases
   - Chat: WebSocket Protocol, Message Ordering, Offline, Group Chat, Read Receipts, E2EE
   - Rate Limiter: Algorithms, Distributed Counting, Redis optimizations
   - News Feed: Fan-out (push/pull/hybrid), Timeline generation
   - Notification: Multi-channel delivery, Template engine, Retry logic
   - Key-Value Store: Consistent hashing, Replication, Quorum, Gossip
   - E-Commerce: Inventory, Cart, Checkout, Idempotency, Payment

--- Required Sections for 5-Year Experience (Japan) ---
   (These gaps were found in URL Shortener and Chat docs — always include)

   N-3. Monitoring & Observability
        - Key metrics table (latency p50/p99, error rates, throughput)
        - Alerting thresholds (warning/critical)
        - Dashboard essentials (real-time, infrastructure, application)
        - SRE practices: SLIs, SLOs, error budgets
        - Tools: Prometheus, Grafana, PagerDuty, Elasticsearch
        - LINE explicitly requires: "proficiency with observability tools"

   N-2. Security & Rate Limiting
        - Rate limiting strategy (Token Bucket or Sliding Window)
        - Implementation: Redis INCR + EXPIRE or sorted sets
        - Abuse protection (WebSocket abuse, DDoS, spam)
        - Content moderation considerations
        - API key management / rotation

   N-1. Disaster Recovery
        - RTO/RPO targets (table format)
        - Multi-region setup (active-active or active-passive)
        - Failover scenarios (step-by-step flow)
        - Backup strategy (retention periods, cold storage)

   N. Cost Optimization (optional but recommended for senior)
        - Instance sizing table
        - Monthly cost estimates (compute, storage, bandwidth)
        - Cost optimization strategies (3-5 bullets)

{N+1}. Interview Tips & Follow-Up Questions

DEEP DIVE TOPICS (6-7 per problem, labeled D1-D7)
======================================================================

Pick 6-7 concepts that need deeper explanation. Good candidates:
D1: Core algorithm/mechanism details (e.g., base62 encoding, consistent hashing)
D2: Distributed ID generation / scaling mechanism
D3: Caching strategy details
D4: Analytics pipeline / async processing
D5: Partitioning / sharding strategies
D6: Edge case handling / special features
D7: Advanced trade-offs (e.g., message search under E2EE, compliance implications)

======================================================================
JAPAN-SPECIFIC INTERVIEW CONSIDERATIONS
======================================================================

For 5-year experience targeting Japan (both foreign and domestic companies):

--- Evaluation Axes (5-axis rubric) ---
1. Requirements Gathering (Problem Framing)
2. Architecture Design
3. Scalability & Reliability
4. Trade-off Analysis (most weighted at senior level)
5. Communication (dialogue, not monologue)

--- Key Differences from US-style Interviews ---
- "Why you made that decision" > "What you did"
- Keigo (honorific language) matters for Japanese companies
- Reverse questions are mandatory (3 minimum)
- Practical, experience-based questions > pure algorithmic puzzles
- Punctuality, humility, prepared questions all matter

--- Common Follow-up Questions in Japan ---
- "How do you detect failures?" (monitoring)
- "What happens when X goes down?" (disaster recovery)
- "How do you handle consistency?" (trade-offs)
- "Tell me about a time..." (STAR framework, war stories)

--- Required Sections for Japan ---
- Monitoring & Observability (LINE explicitly requires this)
- Security & Rate Limiting (abuse prevention)
- Disaster Recovery (RTO/RPO, multi-region)
- Cost Optimization (cloud cost awareness)

--- Company-Specific Notes ---
- Mercari: English-first, SDI with real business problems (90 min)
- LINE/LY Corp: System design + coding,4-6 weeks process
- Rakuten: Lower pressure, more cultural alignment focus
- MUFG: Security and reliability emphasis for banking
