1. Consistent Hashing

The problem it solves

Imagine you have 10 cache servers and you want to distribute keys across them. The naive approach is hash(key) % 10. This works fine until you add or remove a server. When you go from 10 to 11 servers, the modulo changes from % 10 to % 11, and almost every key maps to a different server. You'd have to remigrate nearly all your data — a complete cache miss storm.

Consistent hashing solves this: when a node is added or removed, only K/N keys need to be remapped (where K = total keys, N = number of nodes). For 10 servers, that's only ~10% of keys affected instead of ~90%.

How it works step by step

Step 1: The ring. Imagine a circle with numbers from 0 to 2³²−1 (the range of a 32-bit hash). This is the "hash ring."

Step 2: Place nodes on the ring. Take each server's IP address or name, hash it, and place it at that position on the ring. hash("server-A") lands at some position, hash("server-B") at another, etc.

Step 3: Place keys on the ring. Take each key (e.g., "user:123"), hash it, and find its position on the ring.

Step 4: Assign keys to nodes. For each key, move clockwise from the key's position. The first node you encounter owns that key. If you wrap around the entire ring, the first node owns it.

hash("server-A") ← Node A / key1 → hash("key1") → go clockwise → A key2 → hash("key2") → go clockwise → B \ hash("server-B") ← Node B key3 → hash("key3") → go clockwise → C \ hash("server-C") ← Node C

What happens when a node fails

If Node B goes down, its keys are reassigned to the next node clockwise (Node C). Only keys that belonged to B are affected. All other keys stay where they are. In the naive % N approach, all keys would remap.

// Before: 3 nodes, key distribution hash("key1") = 10 → clockwise → A (at 15) hash("key2") = 30 → clockwise → B (at 35) hash("key3") = 60 → clockwise → C (at 70) // After B fails: only key2 moves to C hash("key1") = 10 → still A hash("key2") = 30 → clockwise → C (was B, now C) hash("key3") = 60 → still C

Virtual nodes (why you need them)

With the basic approach, if nodes aren't evenly spaced on the ring, one node might own a much larger range ("hot spot"). Virtual nodes fix this: each physical node maps to multiple positions on the ring (e.g., server-A#1, server-A#2, etc.). With enough virtual nodes, the distribution becomes statistically uniform, and when a physical node joins/leaves, the rebalancing is smoother.

Where it's used in practice

  • Redis Cluster — uses a variant (16384 hash slots, not a ring, but same idea: slot-to-node mapping)
  • DynamoDB / Cassandra — consistent hashing with virtual nodes for partitioning
  • CDNs — to assign content to edge servers
  • Load balancers — consistent session affinity (same client always hits same backend)
Interview angle: When asked "how would you scale a cache layer?" — consistent hashing is the answer. Say: "I'd use consistent hashing so adding a cache node only remaps 1/N of the keys. With virtual nodes, I avoid hot spots." That single sentence shows you understand the practical trade-off.

2. CAP Theorem & PACELC

The problem it addresses

In a distributed system, data is replicated across multiple machines. Network failures are inevitable — a cable gets cut, a switch fails, a node becomes unreachable. When this happens, you must choose: do you serve stale data but keep running, or do you refuse service until the network heals? CAP says you can't have both.

The three properties

Consistency (C) — Every read receives the most recent write or an error. All nodes see the same data at the same time. If you write "price=100" to node A, a read from node B must return 100 (not 50).

Availability (A) — Every request receives a (non-error) response — without guarantee it contains the latest write. The system stays up even if some nodes are down.

Partition Tolerance (P) — The system continues to operate despite network failures between nodes. Partitions will happen. You cannot opt out of P in a distributed system. The real choice is between CP and AP.

Working through the trade-off

Imagine two servers (A and B) that replicate data. A network partition occurs — A and B can't talk to each other. A client writes "balance=0" to server A. Another client reads from server B:

  • CP choice: Server B refuses the read (returns error) because it can't guarantee the data is fresh. Consistency wins, availability loses.
  • AP choice: Server B returns "balance=100" (stale data). The system stays up. Availability wins, consistency loses.
// Network partition between A and B Client1 → write(balance=0) → Server A (success) Client2 → read(balance) → Server B (can't reach A) CP: B returns error ("sorry, can't serve") AP: B returns 100 (stale, but system is up)

PACELC — the full picture

CAP only tells you what happens during a partition. PACELC adds what happens in normal operation (when there is no partition).

// PAC: when Partition occurs CP (Consistency + Partition tolerance) → Zookeeper, etcd, HBase AP (Availability + Partition tolerance) → Cassandra, DynamoDB // ELC: when Else (normal operation) EC (Strong Consistency) — Higher latency, synchronous replication EL (Low Latency) — Faster, async replication

Examples in the real world

SystemCategoryDuring partitionNormal operation
PostgreSQL (single master + replicas)CP/ECIf master is in minority partition, it stops accepting writesSynchronous commit + read-after-write consistency

How to use this in interviews

Start every system design by naming the consistency model. "This is a read-heavy social feed. I'll lean AP with eventual consistency — it's okay if a user's new post doesn't appear on their friend's feed for a few seconds." Compare that to a banking system where you'd say "I need CP with strong consistency — showing a stale balance is unacceptable."

How to answer "CP or AP?": Ask yourself: is it worse to serve stale data or to serve nothing at all? Social feeds → stale is fine (AP). Banking → wrong data is dangerous (CP). Most real-world systems are AP — users tolerate slight staleness better than downtime.

3. Load Balancers

What a load balancer does

A load balancer sits between clients and servers. It's the single point of contact that distributes incoming requests across a pool of backend servers. This gives you:

  • Fault tolerance — if a server fails, the LB routes around it
  • Scalability — add more servers behind the LB without clients knowing
  • Health checking — LB periodically pings servers and removes unhealthy ones

L4 vs L7 — the key distinction

Layer 4 (Transport): Works at the TCP/UDP level. The LB looks at IP address and port, then forwards the raw packet to a backend. It never inspects the HTTP request — it doesn't know about URLs, headers, or cookies. This is fast (less CPU) but dumb (can't route based on content).

Layer 7 (Application): Works at the HTTP/HTTPS level. The LB terminates the TCP connection, reads the HTTP request, inspects the URL path, headers, cookies, or even the body, and then decides where to route it. Example: route /api/users/* to the user service and /api/payments/* to the payment service. This is slower (more CPU) but smarter.

Health checks

Load balancers periodically send requests (e.g., GET /health) to each backend. If a server fails N consecutive checks, it's removed from the pool. Once it passes checks again, it's re-added. This is how LBs provide fault tolerance without manual intervention.

Your context: Your NestJS apps behind a load balancer would typically use L7 (ALB or NGINX) for path-based routing in a microservices setup. In the P2P messaging system, WebSocket connections need "sticky sessions" (IP hash or a dedicated LB config) so the same client reconnects to the same server that holds their WebSocket state.

4. Caching Strategies

Why cache?

Reading from memory is ~1000x faster than reading from disk. A typical Redis read takes ~1ms; a PostgreSQL query might take 10-100ms or more for complex joins. Caching stores frequently accessed data in fast memory so you avoid hitting the slow database.

Cache layers (from fastest to slowest)

LayerLatencyExamplesWhat it caches
CPU L1/L2 cache~1nsCPUInstructions, hot data
Application memory~0.1msIn-process Map, local LRU cacheConfig, reference data
Distributed cache~1-5msRedis, MemcachedSession data, API responses, counters
CDN~10-50msCloudFront, CloudflareStatic assets, images, video
Database buffer pool~1-10msPostgreSQL shared_buffers, MySQL InnoDB buffer poolFrequently accessed rows/indexes

Read strategy: Cache-Aside (most common)

The application code is responsible for both reading from and writing to the cache. Here's the flow for a read:

// Application reads data data = cache.get(key) if data == null: data = db.query("SELECT * FROM users WHERE id=?") cache.set(key, data, ttl=3600) // Populate cache, expire in 1 hour return data

This is simple and widely used. The cache only holds data that's actually requested (no waste). The downside: the first request after a cache miss is slow (cache stampede).

Write strategies — trade-offs explained

Write-through example

db.query("UPDATE users SET name = ? WHERE id = ?", [new_value, id])
// Both succeed or application rolls back
Write-back example (async flush)

// Worker picks up from queue, writes to DB seconds later

Eviction policies

You can't store everything in cache. When the cache is full, something must be removed:

  • LRU (Least Recently Used) — removes the item not accessed for the longest time. Most common. Works well for most workloads.
  • LFU (Least Frequently Used) — removes the item accessed least often. Better for workloads where "once-popular" data becomes cold.
  • TTL-based — remove items after a fixed time regardless of access. Guarantees freshness.
  • FIFO (First In, First Out) — removes the oldest item. Simple, but may evict hot data.

Cache stampede (thundering herd)

When a popular cache key expires, multiple requests might simultaneously find the cache empty and all hit the database at once. Solutions:

  • Mutex / lock — only one request rebuilds the cache; others wait
  • Early recomputation — rebuild the cache before it expires (probabilistic early expiration)
  • Stale data serving — serve stale data while asynchronously refreshing the cache
Your context: Your NestJS apps with TypeORM + PostgreSQL could use Redis as a cache-aside layer for frequent queries (e.g., user profiles, message history). For the real-time analytics dashboard, you'd use Redis sorted sets for leaderboards and TTL-based caching for pre-computed aggregations.

5. Database Indexing

What an index does

Think of a book's index at the back. Without it, to find a topic you'd read every page. With it, you look up the topic, get the page number, and go directly there. A database index works the same way — it's a separate data structure that lets the database find rows without scanning the entire table.

B-Tree — the default index structure

PostgreSQL, MySQL (InnoDB), and most relational databases use B+Tree as their default index type. It's a balanced tree where:

  • Data is stored in sorted order
  • Each node can have many children (high fan-out), keeping the tree shallow
  • Lookup, insert, and delete are all O(log n)
  • Range queries (WHERE age BETWEEN 20 AND 30) are efficient because leaf nodes are linked

A B-Tree with fan-out of 500 can store 500³ = 125 million entries in just 3 levels. So even for a table with 100M rows, finding any row takes about 3 index traversals.

When an index helps vs. hurts

OperationWith indexWithout index
SELECT WHERE id = 5O(log n) — ~3 lookupsO(n) — full table scan
ORDER BY created_atAlready sorted — O(log n) to traverseSorts all rows — O(n log n)
INSERTO(log n) — must update the indexO(1) — append to table
UPDATE indexed columnO(log n) — update both table and indexO(1) — update table only

Key insight: Indexes speed up reads but slow down writes. Every INSERT, UPDATE, or DELETE on an indexed column must also update the index. This is why you don't index every column — only the ones used in WHERE, JOIN, and ORDER BY.

Composite indexes — multiple columns

An index on multiple columns: CREATE INDEX ON users (country, status, created_at).

The leftmost prefix rule: The index is sorted by country first, then status, then created_at. A query can only use the index if it filters on columns from left to right:

// Can use the index: WHERE country = 'JP' // uses 1st column ✓ WHERE country = 'JP' AND status = 'active' // uses 1st + 2nd ✓ WHERE country = 'JP' AND status = 'active' AND created_at > '2024-01-01' // uses all 3 ✓ // Cannot use the index: WHERE status = 'active' // skips 1st column ✗ WHERE created_at > '2024-01-01' // skips 1st + 2nd ✗

Covering indexes — the fastest reads

A covering index contains all columns that a query needs. The database can answer the query entirely from the index without ever touching the table. This eliminates the table I/O entirely.

-- Without covering index (two I/Os: index + table) SELECT email FROM users WHERE country = 'JP' AND status = 'active'; -- With covering index (one I/O: index only, no table access) CREATE INDEX idx_covering ON users (country, status) INCLUDE (email); -- PostgreSQL: INCLUDE adds non-key columns (v11+) -- MySQL: you can use a composite index with all columns
Interview angle: When designing a database schema in an interview, say: "I'll create a composite index on (user_id, created_at DESC) to cover the 'recent messages' query." This shows you think about query patterns before writing DDL. You can also mention that TypeORM's @Index decorator makes this trivial.

6. SQL vs NoSQL

Core difference: how data is modeled

SQL: Data goes into tables with fixed schemas. Relationships are expressed through foreign keys and JOINs. You define the schema upfront — every row in a table must have the same columns.

NoSQL: Data is stored as documents (JSON), key-value pairs, wide columns, or graphs. Each record can have different fields. There's no schema enforcement — you just put data in.

Deep comparison

DimensionSQL (PostgreSQL, MySQL)NoSQL (MongoDB, DynamoDB)
SchemaFixed, predefined with migrationsFlexible, dynamic — each doc can differ
RelationshipsJOINs via foreign keys (efficient with indexes)Embed sub-docs or reference IDs (client-side join)
ACID transactionsStrong by default. Multi-row, multi-tableLimited. Single-document atomicity in most
Query flexibilityExtremely expressive — GROUP BY, HAVING, subqueries, CTEs, window functionsLimited — key lookups, simple filters, map-reduce
Horizontal scalingHarder — requires sharding + application changesBuilt-in — auto-sharding by design
Data integrityConstraints, triggers, foreign keys enforced by DBNo enforcement — handled in application code

When to pick SQL

  • Your data has clear relationships (users have orders, orders have items)
  • You need ACID transactions (financial data, inventory)
  • You run complex analytical queries (reports, aggregations, joins across tables)
  • Data integrity is critical (constraints, foreign keys)
  • You don't expect to grow beyond a single node for writes

When to pick NoSQL

  • Your schema changes frequently (startups, evolving products)
  • You need massive write throughput (millions of writes/sec)
  • Your data access is simple — "get me the user by ID" or "insert this event"
  • You need to scale horizontally across many nodes without manual sharding
  • You're storing session data, event logs, user preferences

The hybrid approach (most real-world systems)

Most production systems use both. PostgreSQL for the core transactional data (users, accounts, orders) + Redis for caching/session + Elasticsearch for search + a queue for async processing. The question isn't "SQL or NoSQL?" — it's "which tool for which job?"

Your context: Your messaging app uses PostgreSQL + TypeORM — the right choice because message relationships (sender, recipient, thread, status) benefit from relational integrity. If you needed to scale to billions of messages, you might partition by time or add a write buffer. You also used GraphQL + MongoDB-adjacent stores for e-commerce — a good example of picking NoSQL for flexible product catalogs.

7. Replication & Partitioning

Replication — copies of the same data

Replication means keeping identical copies of your data on multiple servers. The goal is fault tolerance (if one server dies, another has the data) and read scalability (spread read traffic across replicas).

Single-leader replication (most common)

One node is the leader (aka primary). All writes go to the leader. One or more followers (aka replicas) copy data from the leader asynchronously. Reads can go to any follower.

Client → WRITE → [Leader] ↘ async replication [Follower 1] [Follower 2] [Follower 3] Client → READ → (any of these)

Trade-off: If the leader fails, one follower must be promoted to leader (downtime of seconds). If a follower is lagging behind, reads from it return stale data. You can use synchronous replication (wait for all followers to ack) for stronger consistency, but that slows writes.

Multi-leader replication

Multiple nodes accept writes. Used when you have data centers in different regions and want local writes (e.g., write to Tokyo leader, which replicates to New York leader). The challenge: write conflicts. If two users edit the same document in different regions simultaneously, whose change wins? Solutions include "last write wins" (LWW), conflict-free replicated data types (CRDTs), or custom merge logic.

Partitioning (sharding) — splitting data across nodes

Replication gives you copies; partitioning gives you pieces. Each shard holds a different subset of data. Together they hold the entire dataset.

In real systems, they're combined. Each shard is replicated for fault tolerance:

For reads, you could add read replicas of PostgreSQL so analytics dashboard queries don't slow down message delivery.

8. Message Queues

The core concept

A message queue is a buffer between a producer (who sends messages) and a consumer (who processes them). The producer doesn't wait for the consumer — it just drops the message into the queue. The consumer picks it up when ready.

This decoupling is the key benefit: if the consumer crashes, messages pile up in the queue and get processed when it recovers. If the producer spikes, the queue absorbs the burst instead of overwhelming the consumer.

The three messaging patterns

Point-to-point (competing consumers): One message is delivered to exactly one consumer. Multiple consumers can pull from the same queue, but each message goes to only one. This is how you distribute work across workers. Example: your P2P texting — a message from user A to user B is picked up by one delivery worker.

Pub/Sub (broadcast): One message is delivered to all subscribers. Each subscriber gets a copy. Example: when a new user signs up, the "user.created" event goes to the email service, the analytics service, and the notification service — all independently.

Work queues: Like point-to-point but with explicit acknowledgment. The worker processes the message and sends an ack back. If the worker crashes without acking, the message is redelivered to another worker. This gives you "at-least-once" delivery.

Delivery guarantees

GuaranteeMeaningHow to achieve
At-most-onceMessage may be lost but never duplicatedSend and forget. No ack, no retry.
At-least-onceMessage is never lost, but may be delivered multiple timesConsumer acks after processing. If it crashes before acking, message is redelivered. This is the default in RabbitMQ.
Exactly-onceMessage is delivered exactly once — no loss, no duplicatesRequires idempotent consumers + distributed transactions. Very expensive. Most systems use at-least-once + idempotency.

RabbitMQ vs Kafka — the decision framework

Choose RabbitMQ when:

  • You need flexible routing (direct, topic, fanout, header exchanges)
  • You want messages removed after consumption (don't need replay)
  • You have moderate throughput (tens of thousands/sec — your 1M/week is ~1.6/sec, well within range)
  • You need RPC-style request/reply patterns
  • You want per-message acknowledgments with dead-letter queues for failures

Choose Kafka when:

  • You need extremely high throughput (millions of messages/sec)
  • You need message replay — consumers can re-read old messages
  • You're building an event sourcing / CQRS system
  • You need log compaction (keep only the latest value per key)
  • You need strong partitioning guarantees for ordered processing
Your context: Your P2P texting service uses NestJS + RabbitMQ + WebSockets. This is a textbook work-queue pattern: producers (web/mobile apps) send messages into a queue, NestJS workers consume and deliver via WebSocket. The 1M+ weekly throughput is modest for RabbitMQ. In the interview, explaining your delivery architecture (at-least-once + idempotent handlers + dead-letter queues) will impress the interviewer.

9. WebSockets vs Polling vs SSE

The problem: real-time updates

The web is built on HTTP — request/response. The client asks, the server answers. But what if the server has new data to push? A new message arrives, a stock price changes, a teammate edits a document. You need a way for the server to proactively send data to the client.

Option 1: Short polling (simplest, least efficient)

The client asks "got anything new?" every N seconds. Even when nothing changed, the client sends a full HTTP request.

// Every 5 seconds: Client → GET /new-messages?since=123 Server ← { messages: [] } // empty — wasted request // 5 seconds later: Client → GET /new-messages?since=123 Server ← { messages: [{ id: 124, text: "hello" }] }

Problem: Most polls return nothing, wasting bandwidth and server resources. Latency is bounded by the polling interval (average delay = interval/2).

Option 2: Long polling (better but hacky)

The client sends a request. The server holds it open (doesn't respond immediately). When new data arrives, the server finally responds. The client immediately sends a new long-poll request. This gives near-real-time delivery with fewer wasted requests.

Problem: Still creates many HTTP connections. Middleboxes (proxies, firewalls) may timeout idle connections. The server must manage many open connections.

Option 3: Server-Sent Events (SSE) — one-way from server

The client opens a single HTTP connection. The server sends data as "events" over this connection using a simple text protocol (event: message\ndata: hello\n\n). The connection stays open.

Pros: Simple protocol, works over standard HTTP, auto-reconnect built into browsers (EventSource API).

Cons: Server-to-client only (no client-to-server). Limited browser connections (most browsers allow 6 per domain).

Option 4: WebSockets — full-duplex (what you use)

The client and server establish a persistent TCP connection via an HTTP upgrade handshake (101 Switching Protocols). Once established, both sides can send messages any time — no request/response pattern.

Pros: True bidirectional communication. Low overhead after initial handshake. No polling waste.

Cons: Requires stateful server (can't use stateless HTTP scaling as easily). Sticky sessions on load balancers needed. More complex to manage reconnection logic.

10. REST vs GraphQL

The fundamental difference

REST exposes resources (endpoints like /users/123, /users/123/orders). Each endpoint returns a fixed structure. GraphQL exposes a single endpoint where the client describes exactly what data it wants — and gets only that.

REST — resource-oriented

Each endpoint returns a predefined shape. If the client needs user info and their orders and order items, it might need 3 separate API calls (or the server provides a custom endpoint that returns everything).

// REST: three round trips for related data GET /users/123 → { id, name, email, ...10 more fields } GET /users/123/orders → [{ id, date, total, ...5 more fields }] GET /orders/456/items → [{ productId, qty, price }]

GraphQL — client-driven queries

Single endpoint. The client specifies the exact shape:

// GraphQL: one request, exactly the fields needed POST /graphql { users(id: 123) { name email orders { date total items { productId, qty } } } }

VersioningURL-based (/v1/, /v2/)Schema evolution (deprecate fields, additive changes)

When each shines

Use REST when:

  • Your API is simple CRUD (create/read/update/delete)
  • You want simple caching (CDN, HTTP caching)
  • You serve mostly public data (same response for all clients)
  • You need file upload/download
  • Your team is small and wants simplicity

Use GraphQL when:

  • You have many different clients (web, mobile, third-party) needing different data shapes
  • You have deeply nested relational data (e.g., user → orders → items → product)
  • Your mobile clients need to minimize data transfer
  • You want strong typing and auto-generated documentation (schema introspection)
Your context: Your e-commerce stores used GraphQL — a good fit because product pages need different fields than search results than admin dashboards. Your calendar and analytics dashboards likely used REST — simpler caching, and each view has a known data shape. In interviews, mention: "I choose GraphQL when clients have heterogeneous data needs; REST when caching simplicity matters more."

11. Rate Limiting

What it is and why it matters

Rate limiting controls how many requests a user, IP, or API key can make within a time window. Without it, a single abusive user can overwhelm your servers, cause a DDoS-like effect, or rack up huge bills for you.

Rate limiters are placed before your application — at the API gateway, load balancer, or a dedicated service. They reject excess requests with HTTP 429 (Too Many Requests) so your backend never sees them.

Fixed window boundary problem

100 req/min limit. User sends 100 at :59 and 100 at :01 — 200 in 2 seconds, both windows "under limit." Sliding window prevents this by considering request history continuously.

Distributed rate limiting

When you have multiple servers, rate limit state must be shared. The standard approach: Redis.

// Simple token bucket in Redis (Lua script for atomicity): local key = KEYS[1] // "rate_limit:user:123" local limit = tonumber(ARGV[1]) // max tokens local refill = tonumber(ARGV[2]) // refill rate local current = redis.call("GET", key) or limit if tonumber(current) > 0 then redis.call("DECR", key) return 1 // allowed else return 0 // rate limited end
Interview angle: Both Mercari and Indeed ask rate limiter questions. Start with: "I'd use a token bucket algorithm — it's simple, memory efficient, and allows natural bursts. For distributed rate limiting, I'd use Redis with Lua scripting for atomic operations." Then discuss trade-offs (Redis as SPOF, using Redis Cluster with consistent hashing).

12. Idempotency & Retry Patterns

The problem

Networks fail. Timeouts happen. When a client sends a request and the connection times out, the client doesn't know if the server processed the request or not. If it retries, the operation might execute twice. In a payment system, that means double-charging the customer. In a messaging system, that means sending a duplicate message.

Idempotency is the property that an operation can be applied multiple times with the same result as applying it once.

Idempotent by nature vs. by design

OperationIdempotent?Why
SET balance = 100Yes (naturally)Setting to 100 repeatedly is the same as doing it once
SET balance = balance + 50NoEach execution increases by 50 — applying twice gives +100
DELETE /users/123Yes (naturally)Deleting a deleted user is a no-op
POST /payments (create)No (needs design)Each call creates a new payment — need idempotency key
PUT /users/123 (full update)Yes (naturally)Sending the same full representation is idempotent
PATCH /users/123 (partial)MaybeDepends on the patch: {"op": "replace", ...} is; {"op": "increment", ...} isn't

Idempotency keys — the pattern

The client generates a unique UUID (idempotency key) and sends it with the request. The server remembers the result keyed by this UUID. If the client retries with the same key, the server returns the stored result without re-executing the operation.

// First attempt: POST /payments { amount: 100, idempotency_key: "abc-123" } Server: processes payment, stores { key: "abc-123", result: "success", tx: "txn_456" } // Retry (client timeout → client resends same request): POST /payments { amount: 100, idempotency_key: "abc-123" } Server: finds existing result for "abc-123" → returns it without processing → { status: "already_processed", transaction: "txn_456" }

Retry strategies

StrategyHow it worksWhen to use
Immediate retryTry once after short delayTransient failures (DNS hiccup, connection reset)
Exponential backoffWait 1s, 2s, 4s, 8s... between retriesStandard pattern for most failures
JitterAdd ±25% randomness to backoff delayPrevent thundering herd when many clients retry
Circuit breakerAfter N failures, stop retrying. Periodically test if recovered.Prevent cascading failures

Common retry scenarios

Failure typeRetry?Strategy
HTTP 503 (Service Unavailable)YesExponential backoff + jitter, 3-5 attempts
HTTP 504 (Gateway Timeout)YesImmediate retry once, then backoff
HTTP 429 (Rate Limited)YesCheck Retry-After header, respect it
HTTP 4xx (Client Error)NoRetrying won't help (bad request, auth failure)
HTTP 5xx (Server Error)YesBackoff + jitter
Connection timeoutYesImmediate retry once, then backoff
Your context: Your messaging system with RabbitMQ uses at-least-once delivery. When a consumer processes a message and crashes before acking, the message is redelivered. If your message handler is idempotent (same message ID results in no duplicate), you get exactly-once processing semantics without exactly-once delivery guarantees. This is the standard production pattern.

13. Microservices vs Monolith

The monolith

One codebase. One deployment unit. All features — user management, payments, notifications, search — run in the same process, sharing the same database.

Advantages: Simpler to develop (no service boundaries to manage). Easier to test end-to-end. No network latency between features. Single deployment. Simple transaction management across features (DB handles it).

Disadvantages: Can't scale features independently — the entire app must scale together. One slow feature (e.g., image processing) can starve other features of resources. Large codebase becomes hard to reason about. Long CI/CD pipelines. Technology lock-in (whole app uses one stack).

Microservices

Each feature or domain is a separate service with its own codebase, deployment, database, and team. Services communicate over the network (HTTP/gRPC) or via message queues.

Advantages: Independent scaling per service (scale only the payment service, not notifications). Independent deployment (deploy one service without taking down the whole app). Technology diversity (each service can use the best language/DB for its job). Smaller, more focused codebases. Team autonomy.

Disadvantages: Complexity explosion — service discovery, API gateways, distributed tracing, circuit breakers, eventual consistency, data duplication. Network latency between services. Testing across service boundaries is hard. Debugging a transaction spanning multiple services is painful.

The comparison matrix

DimensionMonolithMicroservices
DeploymentSingle unit — one deploy affects everythingIndependent — deploy one service at a time
ScalingMust scale the entire appScale individual services by demand
Development speed (early)Fast — no cross-service orchestrationSlow — must define boundaries, contracts, infra
Development speed (late)Slow — large codebase, merge conflictsFast — teams work independently
TestingSimple E2E — one processContract testing, integration testing needed
DatabaseSingle DB, ACID transactions across featuresDatabase per service, eventual consistency across services
Operational costLow — one app to monitor, deploy, debugHigh — observability stack, service mesh, container orchestration
Team structureWorks for small teams (1-10)Designed for multiple teams (10+)

The biggest mistake: premature microservices

The most common anti-pattern is starting with microservices because "they're the modern way" or "Netflix does it." The result: you spend all your time building infrastructure (service discovery, distributed tracing, message brokers) instead of building product features.

The industry consensus: Start with a modular monolith. Use well-defined modules with clear interfaces. If and when a module needs independent scaling or a separate team owns it, extract it into a microservice. This is called the "monolith-first" or "extract microservices" approach.

Common microservice decomposition patterns

By domain (most common): Split by business domain — User Service, Order Service, Payment Service, Notification Service. Each owns its data and domain logic.

By subdomain (DDD): Following Domain-Driven Design bounded contexts. Each bounded context becomes a service.

By capability: Split along technical capabilities — Search Service, Analytics Service, Reporting Service.

Strangler fig pattern: Gradually replace monolith features with microservices one at a time. The monolith shrinks as services are extracted. Low risk, incremental migration.

Your context: Your NestJS apps follow a modular monolith pattern — @Module-decorated feature modules that could become microservices later. This is exactly the right approach. When interviewing at Mercari (which runs Go-based microservices), you can say: "My NestJS apps are modular monoliths with clear module boundaries. I'd extract the messaging module into an independent service when it needs to scale independently."

15. CDNs & Edge Caching

What a CDN does

A Content Delivery Network caches content at edge locations worldwide. Instead of every request traveling to your origin server in one region, the nearest edge node serves cached content. This reduces latency (10-50ms vs 100-300ms) and offloads your servers.

Push vs Pull CDNs

TypeHow it worksBest for
PushYou upload content to edge nodes explicitlyStatic assets, infrequent updates
PullFirst request fetches from origin, subsequent requests served from cache (TTL-based)Dynamic content, frequent updates

Cache invalidation strategies

  • TTL-based: Set Cache-Control: max-age=3600. Simple, but stale content served until expiry.
  • Versioned URLs: app.v2.js. Immutable — never stale, but requires deployment changes.
  • Purge API: Manually invalidate via CDN API. Fast (60s), but requires tooling.

What to cache at edge

CacheDon't cache
Static assets (JS, CSS, images)API responses with user-specific data
Public API responses (GET /products)Authenticated endpoints (use CDN + auth)
HTML pages (for anonymous users)Real-time data (stock prices, chat messages)
Interview angle: "I'd use a pull CDN like CloudFront for static assets with versioned filenames for cache busting. For public API responses, I'd set short TTLs (60s) with stale-while-revalidate. User-specific data bypasses the CDN."

16. Database Scaling

Read scaling: Replicas

Add read replicas to distribute read traffic. The primary handles writes; replicas handle reads. Most apps are read-heavy (100:1 read:write ratio), so replicas eliminate the bottleneck.

Client → READ → [Replica 1] [Replica 2] [Replica 3] Client → WRITE → [Primary] → replicates to all

Consistency trade-off: Async replication = eventual consistency (read after write may miss data). Solution: read-after-write consistency by routing the first read after a write to the primary.

Write scaling: Sharding

When a single primary can't handle write volume, split data across multiple primaries (shards). Each shard holds a subset of data.

StrategyHow it worksBest forTrade-off
Hash-basedhash(user_id) % NEven distributionAdding shards requires rehashing
Range-basedA-M → shard 1, N-Z → shard 2Range queriesHot spots on popular ranges
DirectoryLookup table maps key → shardFlexibilityLookup table = SPOF

Connection pooling

Opening a DB connection takes 20-100ms. With 1000 concurrent requests, you'd need 1000 connections (expensive). A connection pool maintains N pre-opened connections. Requests borrow a connection, use it, return it. Most ORMs (TypeORM, Prisma) handle this automatically.

Rule of thumb: Pool size = (number of CPU cores × 2) + disk spindles. For most apps: 10-20 connections.

Interview angle: "I'd start with read replicas for read scaling — most apps are read-heavy. If writes become the bottleneck, I'd shard by user_id using consistent hashing to minimize reshuffling. I'd use connection pooling to avoid connection exhaustion."

17. Event-Driven Architecture

Core concepts

ConceptWhat it isExample
EventAn immutable record of something that happenedUserCreated, OrderPlaced
Event producerEmits events when state changesOrder service emits OrderPlaced
Event consumerReacts to events independentlyEmail service sends confirmation on OrderPlaced
Event bus/brokerRoutes events from producers to consumersKafka, RabbitMQ, SNS/SQS

Event sourcing

Instead of storing current state, store all state-changing events. Current state = replay all events from the beginning.

// Traditional: store current state orders table: { id: 1, status: "shipped", total: 100 } // Event sourcing: store all events events table: { type: "OrderCreated", data: { id: 1, total: 100 } } { type: "PaymentReceived", data: { id: 1, amount: 100 } } { type: "OrderShipped", data: { id: 1, carrier: "UPS" } }

Benefits: Full audit trail, time-travel debugging, easy undo, multiple projections from same events.

Costs: Complexity, event versioning, larger storage, eventual consistency.

CQRS (Command Query Responsibility Segregation)

Separate the write model (commands) from the read model (queries). Write model optimizes for consistency; read model optimizes for query speed.

Commands (writes) → [Write DB] → events → [Read DB] → Queries (reads)

When to use: When read and write patterns are very different (e.g., write-heavy ingestion, complex read queries).

Interview angle: "For the order service, I'd use event sourcing for the write side — it gives us a complete audit trail. For reads, I'd project events into a denormalized read model optimized for the UI. CQRS keeps both sides independent."

18. Observability

The three pillars

PillarWhat it tells youToolsFormat
MetricsAggregated numbers over time (latency, error rate, throughput)Prometheus, Datadog, CloudWatchNumeric time series
LogsDiscrete events with context (what happened, when, why)ELK, Loki, CloudWatch LogsStructured JSON lines
TracesEnd-to-end request path across services (where time is spent)Jaeger, Zipkin, X-RaySpan tree per request

RED method (for services)

  • Rate: Requests per second
  • Errors: Errors per second (or error rate %)
  • Duration: Latency distribution (p50, p95, p99)

USE method (for resources)

  • Utilization: % of resource in use (CPU, memory, disk)
  • Saturation: Queue depth (requests waiting)
  • Errors: Error count on the resource

Structured logging best practices

{ "timestamp": "2024-01-15T10:30:00Z", "level": "error", "service": "order-service", "trace_id": "abc-123", "user_id": "u-456", "message": "Payment failed", "error": "stripe_timeout", "duration_ms": 5000 }

Always include: timestamp, level, service name, trace_id (for correlation), and context fields. Never log secrets or PII.

Alerting principles

  • Alert on symptoms (error rate, latency) not causes (CPU, memory)
  • Actionable alerts only — every alert should require human action
  • Multi-window alerting — fire only if condition persists (5 min warning, 15 min critical)

19. Security

Authentication vs Authorization

ConceptQuestion it answersImplementation
Authentication"Who are you?"JWT, session cookies, API keys, OAuth
Authorization"What can you do?"RBAC, ABAC, policy engines

JWT (JSON Web Token)

Header.Payload.Signature Header: { "alg": "RS256", "typ": "JWT" } Payload: { "sub": "user-123", "exp": 1700000000, "role": "admin" } Signature: RS256(header + payload, private_key)

Stateless: Server verifies signature without DB lookup. Token carries all claims.

Revocation problem: Can't invalidate until expiry. Solutions: short expiry (15 min) + refresh tokens, or token blacklist in Redis.

OAuth 2.0 flow (simplified)

1. User clicks "Login with Google" 2. → Redirect to Google OAuth endpoint 3. → User consents 4. → Google redirects back with authorization code 5. → Your server exchanges code for access token 6. → Use token to fetch user profile

Security checklist

  • HTTPS everywhere — encrypt in transit (TLS 1.2+)
  • Encrypt at rest — AES-256 for sensitive data (PII, financial)
  • Input validation — sanitize all user input (SQL injection, XSS)
  • Rate limiting — prevent brute force and DDoS
  • CORS — restrict which domains can call your API
  • Secrets management — never commit secrets; use vault/KMS
Interview angle: "I'd use short-lived JWTs (15 min) with refresh tokens. Sensitive data encrypted at rest with AES-256. All inter-service communication over mTLS. API keys rotated quarterly."

20. Capacity Planning

QPS estimation formula

Daily active users (DAU) × requests per user per day ÷ 86,400 (seconds/day) = average QPS Peak QPS = average QPS × peak multiplier (typically 2-4x)

Example: 10M DAU × 10 requests/day = 100M requests/day ÷ 86,400 = ~1,160 avg QPS. Peak = ~4,600 QPS.

Storage estimation

Object size × daily volume × retention days = total storage Example: 1 KB/message × 100M messages/day × 365 days = 36.5 TB/year

Bandwidth estimation

QPS × average response size = bandwidth Example: 4,600 QPS × 5 KB = 23 MB/s = 184 Mbps

Capacity planning rules of thumb

MetricRule of thumb
Database connections(CPU cores × 2) + disk spindles per DB instance
Redis memory2× the hot data size (for eviction headroom)
Kafka partitionsMax throughput per partition ≈ 10 MB/s; scale partitions for throughput
Thread poolsCPU-bound: cores + 1. I/O-bound: cores × (1 + wait_time/compute_time)

The "back-of-envelope" approach

In interviews, you don't need exact numbers. Use powers of 10:

  • 1M users → ~10 QPS average, ~40 QPS peak
  • 100M users → ~1K QPS average, ~4K QPS peak
  • 1B users → ~10K QPS average, ~40K QPS peak
Interview angle: Start every design with capacity estimation: "Let me estimate the scale. With 10M DAU and 10 requests/day, that's ~1K avg QPS, ~4K peak. Storage: ~1 KB per message, 100M messages/day = 100 GB/day. This tells me I need horizontal scaling for writes but a single PostgreSQL with replicas handles reads."
