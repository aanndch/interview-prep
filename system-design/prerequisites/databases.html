Databases: SQL vs. NoSQL

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

Choosing a database paradigm defines data integrity guarantees and system scalability boundaries. **SQL** relational databases prioritize schemas and transaction accuracy via ACID properties, while **NoSQL** databases prioritize elastic horizontal scaling and write ingestion throughput.

1. SQL & ACID Transactions

Guarantees database consistency even under hardware crashes or concurrent conflicts:

  • Atomicity: All updates in a transaction commit fully, or rollback entirely.
  • Consistency: Ensures all writes validate database constraints (foreign keys, check rules).
  • Isolation: Multi-client writes do not interfere. Realized via **Pessimistic Locking** (blocks rows, causes wait states) or **Optimistic Concurrency Control (OCC)** (versions records, aborts writes on race conflicts).
  • Durability: Once committed, updates survive system failures (written to Write-Ahead Logs - WAL).

2. NoSQL & BASE Properties

Prioritizes Availability and horizontal scale out over strict transactional isolation:

  • Basically Available: The database prioritizes accepting writes even during network partitions.
  • Soft State: Replicas can drift in data state without coordinator locks.
  • Eventual Consistency: Replicas catch up asynchronously. Readers may temporarily see stale data, but nodes eventually converge on the same state.

3. B-Trees vs. LSM-Trees

Low-level storage engines shape read and write performance trade-offs:

  • B-Tree (InnoDB, MySQL/Postgres): Keeps index nodes sorted on disk. Excellent for reads and range queries ($O(\log N)$ search steps). Writes require random page I/O, which is slow.
  • LSM-Tree (Cassandra, RocksDB): Appends writes to an in-memory buffer (MemTable) then flushes sequentially to disk files (SSTables) in background compaction cycles. Extremely fast write speeds; reads require checking multiple SSTables (optimized via Bloom Filters).

4. Connection Pooling Pitfalls

Database resources are limited. A thread connection pool (like HikariCP or PgBouncer proxy) is required to shield relational databases:

  • PgBouncer Session Mode: Keeps connection mapped to client socket for session lifetime. High overhead.
  • PgBouncer Transaction Mode: Recycles connections back to pool as soon as individual transactions commit. Drastically decreases connection counts. (Cannot support features like prepared statements or temporary tables).

How to Answer in an Interview

When selecting a database structure, frame it around transaction isolation requirements and write scaling patterns:

"For financial books, order ledgers, and core user records, I will choose a relational SQL database like PostgreSQL. Relational engines support strict schemas, complex JOIN operations, and strong ACID guarantees using row-level locking. For analytics logs, IoT event streams, or highly scaling user feeds, I will choose NoSQL wide-column systems like Cassandra or managed document systems like DynamoDB, which use LSM-Tree engines to support massive asynchronous write ingestion."
