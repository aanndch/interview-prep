Observability & Monitoring

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

**Observability** is the degree to which you can infer the internal states of a distributed microservices network based solely on its external outputs (Metrics, Logs, and Traces), enabling real-time bottleneck detection and debugging.

1. Three Pillars of Observability

The structured signals used to analyze system health and failures:

  • Metrics (Aggregations): Numerical, structured time-series data (e.g. Prometheus counters/gauges). Highly efficient for dashboards and triggering alert thresholds.
  • Logs (Discrete Events): Line-by-line event outputs containing rich debug details (JSON logs sent to ELK/Loki). Expensive to index but vital for debugging root causes.
  • Traces (Lifecycle Graph): DAG showing request propagation path through networks. Essential for diagnosing microservice cascading delays.

2. Context Propagation & Tracing

How distributed tracing engines stitch together service logs:

  • Metadata Injection: API Gateway generates a unique Trace-ID and injects it into client request HTTP headers (e.g. W3C traceparent).
  • Span Propagation: Every downstream microservice reads the parent headers, executes its sub-operations (creating a child Span-ID), and forwards correlation identifiers to databases.

3. RED & USE Frameworks

Standard architectural rules to categorize alerting alerts:

  • RED Method (For Services): Rate (requests/sec), Errors (failed HTTP/gRPC count), and Duration (latency percentiles).
  • USE Method (For Infrastructure): Utilization (percentage busy, e.g. CPU), Saturation (work backlog, e.g. disk queue lengths), and Errors (hardware level anomalies).

4. Sampling & Ingestion Costs

Storing 100% of telemetry at high traffic is financially impossible:

  • Head Sampling: Randomly drop traces at ingestion (e.g. keep only 5% of requests). Saves bandwidth but might miss rare 1-in-a-million errors.
  • Tail Sampling: Buffer active traces in memory. If a trace experiences an HTTP 500 error or slow tail latency, ingest it; otherwise, discard healthy traces.

How to Answer in an Interview

If asked: "How do you design an alerting and observability pipeline for a microservice mesh?"

Say: "I will configure OpenTelemetry across our stack to collect metrics, logs, and traces. For service-level monitoring, I will alert on RED metrics (Rate, Errors, Duration) and use Jaeger to analyze nested trace spans. We will implement head-sampling on healthy paths and tail-sampling for errors or high-latency traces. To prevent logging disk write limits from bottlenecking the system, logs will be formatted as structured JSON and buffered via Kafka before ingestion to ElasticSearch."
