Security & Authentication

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

**Authentication Systems** verify the identities of clients accessing APIs. Stateful approaches store active **Sessions** in databases like Redis to support instant token revocation, while stateless **JWTs** (JSON Web Tokens) authorize requests offline using cryptographic signatures to bypass database locks.

1. Stateful Sessions

Authentication where metadata is held on server RAM pools:

  • Mechanism: On login, server creates a session record in Redis and returns a random Session-ID cookie to the browser.
  • Pros: Security control. If a user loses their phone, you delete their record in Redis, instantly revoking their active API access.
  • Cons: Storage overhead. High traffic requires scaling a distributed Redis cluster to handle session verification read loops.

2. Stateless JWTs (JSON Web Tokens)

Authorizations encoded as base64 strings containing cryptographic signatures:

  • Mechanism: Server encodes permissions (claims) into a token, signs it using a private key, and returns it. Client sends it via HTTP Authorization: Bearer .
  • Pros: Infinite horizontal scaling. Replicas validate signature offline using the public key. Zero DB connection costs.
  • Cons: Impossible to revoke before the TTL expires, unless you maintain a blacklist (which introduces DB state checks).

3. CORS Preflights & Protection

Securing browser-based cross-origin API executions:

  • Mechanism: When executing a cross-origin request (e.g. from app.com to api.com), the browser executes an HTTP OPTIONS preflight to verify authorization scopes.
  • Headers: Server must return correct Access-Control-Allow-Origin whitelist bounds, matching credential parameters to avoid client execution blocks.

4. CSRF Attack Mitigations

Preventing attackers from exploiting cached browser session state:

  • The Vulnerability: Browsers automatically attach cookies to requests mapping target domains, allowing malicious pages to trigger actions on behalf of users.
  • SameSite Cookies: Setting cookie attributes to SameSite=Strict or Lax stops browsers from attaching authorization context on cross-site requests.
  • Double-Submit Tokens: Client injects a random cryptographically secure token into both request body parameters and cookies, forcing the server to assert they match.
Session vs. JWT Comparison
Metric Stateful Session (Redis) Stateless JWT
Database Lookup Required on every request to fetch user metadata. Zero. Validated offline using cryptography.
Revocation Speed Instant (Deleting session terminates access immediately). Delayed (Token stays valid until expiry TTL).
Storage Overhead High (Memory scales with active user count). Zero (State stored inside token string on client).
Security Vulnerability Subject to CSRF (cookies sent automatically). Subject to XSS (if stored in localStorage/JS).

How to Answer in an Interview

If asked: "How do you design a secure, scaling auth architecture for a distributed application?"

Say: "I will choose a hybrid auth architecture. We will issue short-lived JWTs (e.g. 15 minutes) containing user scopes, signed using private keys. Microservices can validate the JWT signatures statelessly offline, bypassing database check delays. For refreshing, we'll store a stateful Refresh Token inside HttpOnly, SameSite=Strict cookies. If a user logs out or is banned, we delete the refresh record from our Redis session cluster, preventing subsequent token renewals while the short-lived JWT naturally expires within 15 minutes."
