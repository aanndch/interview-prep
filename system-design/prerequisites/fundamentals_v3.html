The 6 Fundamentals

1. Consistent Hashing

Analogy

Imagine a round table with seats numbered 0 to 99. You have 3 people (servers) sitting at specific seats. A ball (data key) is rolled onto the table — it lands at some number and slides clockwise until it reaches the first person. That person catches it and owns it.

Now if one person leaves, only the balls that were between them and the next person clockwise need to move. Everything else stays put. That's consistent hashing.

WHY it exists

The naive approach is hash(key) % N where N = number of servers. This works until N changes. Going from 10 to 11 servers changes the modulo, and ~90% of keys land on different servers. That's a cache miss storm — your database gets flooded with requests it was never designed to handle.

Consistent hashing fixes this: when a node is added or removed, only K/N keys move (where K = total keys, N = nodes). For 10 servers, that's ~10% affected instead of ~90%.

How it works, step by step

Step 1 — The ring. Imagine a circle with positions 0 to 2³²−1. This is the hash ring.

Step 2 — Place nodes. Hash each server's name/IP and place it at that position on the ring.

Step 3 — Place keys. Hash each key (e.g., "user:123") and find its position on the ring.

Step 4 — Assign. For each key, walk clockwise from the key's position. The first node you encounter owns that key.

hash("server-A") ← Node A / key1 → clockwise → A key2 → clockwise → B When B fails: only key2 moves to C. \ All other keys are unaffected. hash("server-B") ← Node B \ hash("server-C") ← Node C

Virtual nodes — fixing uneven distribution

With 3 physical nodes, they might cluster on one side of the ring — one node gets 60% of keys, another gets 10%. Virtual nodes fix this: each physical node gets multiple positions on the ring (server-A#1, server-A#2, etc.). With enough virtual positions, the law of large numbers ensures even distribution.

WHY virtual nodes work

Think of it like rolling one die (highly variable) vs rolling 100 dice and averaging (very predictable). Each virtual node is like an additional die roll. With 100 virtual nodes per physical node, the average distribution across the ring becomes statistically uniform — hot spots disappear.

Where it's used

  • Redis Cluster — 16384 hash slots mapped to nodes (variant of consistent hashing)
  • Cassandra / DynamoDB — data partitioning across nodes
  • CDNs — content to edge server assignment
  • Load balancers — sticky sessions (same client → same backend)
Common Mistakes

Mistake 1: Saying "consistent hashing is the same as hash mod N." It's not — the ring structure is what makes rebalancing efficient.

Mistake 2: Forgetting to mention virtual nodes. Without them, distribution is uneven and you get hot spots.

Mistake 3: Not being able to explain what happens when a node joins vs leaves. Join: keys between new node and its counter-clockwise neighbor move to it. Leave: keys between dying node and its clockwise neighbor move there.

Interview Script

"I'd use consistent hashing so adding or removing a cache node only remaps 1/N of keys. Without it, going from 10 to 11 servers would remap ~90% of keys — a cache miss storm. I'd add virtual nodes to ensure even distribution across the ring."

Self-Check
  1. Why does hash(key) % N break when N changes? What's the percentage of keys affected?
  2. What problem do virtual nodes solve? How do they fix it?
  3. If you have 5 servers and remove one, approximately what percentage of keys move? What happens to the rest?

2. CAP Theorem & PACELC

Analogy

Imagine a restaurant with two chefs (servers) who share a recipe book. A fire cuts the intercom between them (network partition). A customer orders something new at Chef A's station. Chef A writes it in their book. Another customer asks Chef B for the new order.

CP choice: Chef B says "I don't have that order yet — I can't serve you right now." Consistent, but unavailable.
AP choice: Chef B says "I'll make what I think you want based on what I have." Available, but might be wrong.

WHY partition tolerance is non-negotiable

Networks are physical — cables get cut, routers crash, data centers lose power. No amount of engineering prevents this. Even within a single data center, packets get dropped daily. You can't "opt out" of partitions. So the real question is: when the network splits (not if), do you return stale data (AP) or refuse to answer (CP)?

The three properties

Consistency (C) — Every read receives the most recent write or an error. All nodes see the same data at the same time.

Availability (A) — Every request receives a response (non-error), but it might contain stale data.

Partition Tolerance (P) — The system continues operating despite network failures. You cannot opt out of P. The real choice is CP vs AP.

The trade-off in action

// Network partition between Server A and Server B Client1 → write(balance=0) → Server A (success) Client2 → read(balance) → Server B (can't reach A) CP: B returns error — consistent but unavailable AP: B returns 100 — available but stale

PACELC — beyond partitions

CAP only tells you what happens during a partition. PACELC adds what happens during normal operation (no partition).

// During Partition: CP → Zookeeper, etcd AP → Cassandra, DynamoDB // Normal operation (Else): EC (strong consistency) EL (low latency)

Real-world systems

SystemCategoryDuring partitionNormal operation
PostgreSQLCP/ECMinority stops writesSynchronous commit
CassandraAP/ELAny node accepts writesEventually consistent
DynamoDBAP/ELWrites succeed, eventual readsLow latency
Redis ClusterCP/ECMinority stops writesStrong consistency
MongoDBCP/ECPrimary electionPrimary handles writes
WHY most systems lean AP

The cost of being unavailable is almost always worse than being briefly wrong. If Instagram shows you a post 2 seconds late, nobody cares. If your payment gateway refuses transactions for 30 seconds, you lose revenue and trust. Partitions are usually brief (seconds to minutes), and eventual consistency catches up fast.

Common Mistakes

Mistake 1: Saying "you can pick any two of CAP." You can't — P is mandatory. The real choice is CP or AP.

Mistake 2: Not knowing when to pick CP vs AP. Rule: is it worse to serve stale data or to serve nothing? Banking → stale is dangerous (CP). Social feed → stale is fine (AP).

Mistake 3: Confusing consistency models. "Strong consistency" means every read gets the latest write. "Eventual consistency" means reads will eventually reflect writes, but there's a window of staleness.

Interview Script

"For a social feed: AP — eventual consistency is fine, staleness for a few seconds is acceptable. For banking: CP — showing a wrong balance is unacceptable. Most systems lean AP because partitions are inevitable and brief staleness is tolerable. In normal operation, I'd choose low latency (EL) over strong consistency (EC) unless the use case demands it."

Self-Check
  1. Why can't you "opt out" of partition tolerance in a distributed system?
  2. A banking app and a social media feed — which should be CP, which AP? Why?
  3. What does PACELC add that CAP doesn't cover?

3. Load Balancers

Analogy

Think of a hotel receptionist. Guests (clients) walk in and ask for rooms (servers). The receptionist knows which rooms are occupied, which are clean, and which are broken. They assign each guest to the best available room. If a room's AC breaks, the receptionist stops assigning guests there until it's fixed. That's a load balancer — the front door that distributes work and removes broken servers from rotation.

WHY not just use DNS?

DNS can point api.example.com to multiple IPs. But DNS has a fatal flaw: caching. Your DNS record has a TTL — browsers cache it for minutes or hours. If a server dies, DNS won't know until the cache expires. Meanwhile, traffic keeps hitting the dead server. A load balancer checks health in seconds and removes bad servers instantly.

L4 vs L7 — the key distinction

WHY L4 vs L7 matters

L4 just sees TCP packets — source IP, destination port. It routes fast because it doesn't look inside the packet. L7 sees the full HTTP request — URL path, headers, cookies. It can route /api/users/* to one service and /api/payments/* to another. The trade-off: L4 is faster but dumber, L7 is slower but smarter.

L4 (AWS NLB)L7 (AWS ALB, NGINX)
SeesTCP/UDP, IP, portHTTP headers, cookies, URL path
Routes byIP + portURL, host, header, cookie
SSLPasses throughTerminates (offloads from backend)
SpeedFaster (less inspection)Slower (parses HTTP)
Use whenNeed raw TCP throughputNeed path-based or header-based routing

Algorithms

AlgorithmHow it worksBest for
Round robinTakes turns sending to each serverEqual-capacity servers, uniform requests
Least connectionsSends to server with fewest active connectionsVaried request processing times
IP hashHashes client IP to pick a server (sticky)Stateful apps needing session affinity

Health checks

Load balancers periodically ping /health on each backend. Fails N consecutive checks → removed from pool. Passes again → re-added. This gives fault tolerance without manual intervention.

Common Mistakes

Mistake 1: Not mentioning sticky sessions when discussing WebSockets. WebSocket connections need to hit the same backend that holds their state.

Mistake 2: Saying "load balancers are only for scaling." They also provide fault tolerance — if a server dies, the LB routes around it.

Mistake 3: Forgetting that L7 load balancers terminate SSL — this offloads encryption work from your backends.

Interview Script

"I'd use an L7 load balancer for path-based routing — /api/users/* to the user service, /api/payments/* to the payment service. Least connections handles varied request times. Health checks every 10 seconds remove unhealthy backends within 30 seconds."

Self-Check
  1. Why is DNS not sufficient for load balancing in production?
  2. When would you choose L4 over L7? Give a concrete example.
  3. You have a WebSocket chat app. Which load balancing algorithm do you use and why?

4. Caching Strategies

Analogy

Think of a library. The main collection (database) is in a warehouse across town — it has every book ever written, but getting a book takes 30 minutes (drive there, find it, drive back). The library also has a small shelf near the entrance (cache) with the 100 most popular books — you can grab one in 10 seconds.

If someone asks for a book on the popular shelf, you hand it to them instantly. If it's not there, you go to the warehouse, get it, and put a copy on the popular shelf for next time. That's cache-aside.

WHY memory is so much faster than disk

RAM uses electrons moving through circuits — no moving parts, nanosecond access. SSDs have no moving parts either, but data travels through more controller and flash layers. The numbers: RAM reads in ~100 nanoseconds. SSD reads in ~100 microseconds. That's 1000x. At scale, that gap is the difference between handling 1,000 requests/sec and 1,000,000.

Cache layers (fastest to slowest)

LayerLatencyExamples
In-process (local)~0.1msNode.js LRU Map, Guava cache
Distributed~1-2msRedis, Memcached
CDN~10-50msCloudFront, Cloudflare
Database buffer~1-10msPostgreSQL shared_buffers

Cache-Aside (most common pattern)

// Read path data = cache.get(key) if data == null: data = db.query("SELECT * FROM users WHERE id=?") cache.set(key, data, ttl=3600) return data // Write path db.update(data) cache.invalidate(key) // Don't update — just delete
WHY cache-aside (not write-through)?

Cache-aside keeps the application in control. You decide what to cache, when to invalidate. Write-through couples the cache to every write path — if Redis is slow or down, your writes fail. With cache-aside, the cache is just an optimization. If Redis dies, your app still works — just slower. The mantra: cache is a guest, database is the source of truth.

Write strategies

StrategyOn writeTrade-off
Write-throughWrite to cache, then DBAlways fresh, but slower (two writes)
Write-aroundWrite to DB only, invalidate cacheFast write, but next read is a cache miss
Write-backWrite to cache, async flush to DBFastest, but data loss if cache crashes before flush

Cache stampede (thundering herd)

Popular key expires → 1000 requests hit DB at once. The database wasn't designed for 1000 identical heavy queries in one second.

// Before stampede: Request → cache.get("top_product") → hit → fast response Request → cache.get("top_product") → hit → fast response // After key expires: Request → cache.get("top_product") → miss → DB query (slow!) Request → cache.get("top_product") → miss → DB query (slow!) ×1000 Request → cache.get("top_product") → miss → DB query (slow!)

Fixes:

  • Mutex / lock — only one request rebuilds the cache; others wait or serve stale
  • Early recomputation — refresh cache before it expires (probabilistic early expiration)
  • Stale data serving — serve stale data while asynchronously refreshing

Eviction policies

When the cache is full, something must be removed:

  • LRU (Least Recently Used) — removes the item not accessed for the longest time. Most common.
  • LFU (Least Frequently Used) — removes the item accessed least often. Better for workloads where "once-popular" data becomes cold.
  • TTL-based — remove after fixed time. Guarantees freshness.
Common Mistakes

Mistake 1: "Cache-aside means the cache is always fresh." No — there's a window between DB write and cache invalidation where stale data is served.

Mistake 2: Forgetting the write path. Many people explain reads but skip: "On write, I update the DB then invalidate the cache — not update it."

Mistake 3: Not mentioning cache stampede. Always discuss what happens when a hot key expires.

Interview Script

"Cache-aside with Redis. On read: check cache, fall back to DB on miss. On write: update DB, invalidate cache. For critical paths: L1 in-memory (90% hit rate) + L2 Redis (9%) — only 1% of reads hit the DB. I'd use mutex locking for cache stampede prevention on hot keys."

Self-Check
  1. Why invalidate cache on write instead of updating it?
  2. What is cache stampede and how do you prevent it?
  3. You have a write-heavy system. Which write strategy would you use and why?

5. Database Indexing

Analogy

Think of a textbook with 500 pages and no table of contents. To find "photosynthesis," you'd read every page — that's a full table scan. Now imagine the back of the book has an index: "photosynthesis — page 247." You flip to 247 directly — that's an indexed lookup. The index is a separate, smaller data structure that maps topics to page numbers.

WHY an index speeds up reads

Without an index, every SELECT reads every row — O(n). On 10 million rows, that's 10 million rows checked. An index is a B-Tree (balanced sorted tree) storing just the indexed columns. Lookup is O(log n) — binary search. A B-Tree with fan-out 500 stores 500³ = 125 million entries in just 3 levels. Finding any row takes ~3 index traversals instead of 10 million comparisons.

B-Tree — the default index structure

PostgreSQL, MySQL InnoDB use B+Tree. Properties:

  • Data stored in sorted order
  • High fan-out (many children per node) — tree stays shallow
  • Lookup, insert, delete: all O(log n)
  • Range queries efficient — leaf nodes are linked for sequential access
WHY B-Tree, not hash?

Hash indexes only support equality (WHERE id=5). They can't do ranges (WHERE id > 5), ORDER BY, or prefix matching. B-Trees store data in sorted order, so they handle equality, ranges, sorting, and grouping. That's why every major database defaults to B-Tree.

The trade-off: reads vs writes

OperationWith indexWithout index
SELECT WHERE id=5O(log n) — ~3 lookupsO(n) — full scan
ORDER BY created_atPre-sorted — O(log n) traversalO(n log n) — sort all rows
INSERTO(log n) — must update indexO(1) — append to table
WHY indexes slow down writes

Every INSERT doesn't just append to the table — it inserts into every index, maintaining sorted order. One table with 5 indexes means 5 extra sorted insertions per write. Extra disk I/O, extra CPU for tree rebalancing, extra lock contention. A write-heavy table with 15 indexes can be 10x slower than one with 3.

Rule of thumb: Only index columns in WHERE, JOIN, and ORDER BY. Every index slows writes.

Composite indexes — multiple columns

Index on (country, status, created_at) — sorted left to right. Must follow the leftmost prefix rule:

// Can use the index: WHERE country = 'JP' // uses 1st column ✓ WHERE country = 'JP' AND status = 'active' // uses 1st + 2nd ✓ WHERE country = 'JP' AND status = 'active' AND created_at > '2024-01-01' // uses all 3 ✓ // Cannot use the index: WHERE status = 'active' // skips 1st column ✗ WHERE created_at > '2024-01-01' // skips 1st + 2nd ✗

Covering indexes — the fastest reads

A covering index contains all columns a query needs. The database answers entirely from the index — no table lookup:

-- Without covering index (two I/Os: index + table) SELECT email FROM users WHERE country = 'JP' AND status = 'active'; -- With covering index (one I/O: index only) CREATE INDEX idx_covering ON users (country, status) INCLUDE (email);
Common Mistakes

Mistake 1: "I'll index every column to speed up all queries." No — every index slows writes. Only index columns used in WHERE, JOIN, ORDER BY.

Mistake 2: Not knowing the leftmost prefix rule. If you index (A, B, C), a query on just B or C can't use it.

Mistake 3: Forgetting EXPLAIN/ANALYZE. Always say: "I'd verify the index is used with EXPLAIN ANALYZE."

Interview Script

"I'll create a composite index on (user_id, created_at DESC) to cover the 'recent messages' query — O(log n) instead of O(n). The leftmost prefix ensures queries filtering by user_id use the index. I'd verify with EXPLAIN ANALYZE."

Self-Check
  1. Why can't a hash index support range queries?
  2. You have a composite index on (A, B, C). Which of these queries can use it: WHERE B=1, WHERE A=1 AND B=2, WHERE A=1 AND C=3?
  3. What's a covering index and why is it faster?

6. SQL vs NoSQL

Analogy

SQL is a spreadsheet — every row has the same columns, relationships are enforced, and you can run complex formulas (queries) across rows. NoSQL is a folder of documents — each document can be different, there's no enforcement, and you can stuff whatever you want in. The spreadsheet is great when structure matters. The folder is great when flexibility matters.

WHY NoSQL even exists

SQL databases scale vertically (bigger server) — which hits a ceiling. By the late 2000s, Google and Amazon generated more data than a single server could handle. They needed to distribute data across hundreds of machines, and JOINs across network boundaries were too slow. NoSQL was born: "We need horizontal scale, and JOINs are the bottleneck." Cassandra, DynamoDB, MongoDB — all emerged from this constraint.

The deep comparison

SQL (PostgreSQL, MySQL)NoSQL (MongoDB, DynamoDB)
SchemaFixed, predefined with migrationsFlexible, dynamic — each doc can differ
RelationshipsJOINs via foreign keysEmbed docs or reference IDs (client-side join)
TransactionsACID — multi-row, multi-tableLimited — single-document atomic
QueriesGROUP BY, CTEs, window functionsKey lookups, simple filters
ScalingVertical (harder to shard)Horizontal (auto-sharding built in)
Data integrityConstraints, triggers, foreign keysNo enforcement — app code handles it

Decision framework

Pick SQL when:

  • Data has clear relationships (users have orders, orders have items)
  • You need ACID transactions (finance, inventory)
  • Complex analytical queries (reports, aggregations, JOINs)
  • Data integrity is critical (constraints, foreign keys enforced by DB)

Pick NoSQL when:

  • Schema changes frequently (startups, evolving products)
  • Massive write throughput (millions of writes/sec)
  • Simple access patterns — "get by ID" or "insert event"
  • Need horizontal scale without manual sharding
WHY hybrid is the real answer

No single database does everything well. PostgreSQL is great for transactions but slow for full-text search. Elasticsearch is great for search but bad at transactions. Redis is fast but loses data on restart. The reality: use multiple databases, each for its strength. Core data → PostgreSQL. Sessions → Redis. Search → Elasticsearch. Event logs → Kafka/S3. The skill isn't picking one — it's knowing which data goes where.

Common Mistakes

Mistake 1: "NoSQL is always faster than SQL." No — NoSQL is faster for simple key lookups. SQL is faster for complex queries with JOINs and aggregations.

Mistake 2: "We should use MongoDB because it's modern." Pick the database for your access patterns, not because it's trendy.

Mistake 3: Not mentioning the hybrid approach. Real systems use both — PostgreSQL + Redis + Elasticsearch is standard.

Interview Script

"For this system, I'd use PostgreSQL — clear relationships, need ACID for [feature]. I'd add Redis for caching hot data and session storage. If we needed full-text search, Elasticsearch. The question isn't SQL vs NoSQL — it's which tool for which job."

Self-Check
  1. Why can't SQL databases scale horizontally as easily as NoSQL?
  2. Give an example of a system that uses both SQL and NoSQL. Why?
  3. When would you pick DynamoDB over PostgreSQL for a new feature?
Quick Reference

Quick Reference Card

Review this the day before your interview. One concept per card.

1. Consistent Hashing

What: Distribute keys across nodes so adding/removing a node only moves K/N keys.

When: Cache layers, distributed databases, CDNs.

Key phrase: "Without it, going from 10 to 11 servers remaps ~90% of keys — a cache miss storm."

2. CAP Theorem

What: During network partition, choose consistency (CP) or availability (AP). P is mandatory.

When: Start of every system design — name your consistency model.

Key phrase: "Is it worse to serve stale data or serve nothing? Social → AP. Banking → CP."

3. Load Balancers

What: Distribute traffic across servers. Remove dead ones. L4 (fast/dumb) vs L7 (slow/smart).

When: Always. Every production system needs one.

Key phrase: "L7 for path-based routing, least connections for varied request times."

4. Caching

What: Store frequently accessed data in fast memory. Cache-aside: app manages cache.

When: Read-heavy workloads. Hot data accessed repeatedly.

Key phrase: "Cache-aside with Redis. On write: update DB, invalidate cache. Mutex for stampede."

5. Database Indexing

What: B-Tree data structure for O(log n) lookups. Trade write speed for read speed.

When: Columns in WHERE, JOIN, ORDER BY. Never index every column.

Key phrase: "Composite index on (user_id, created_at DESC) covers the recent-messages query."

6. SQL vs NoSQL

What: SQL for structured data with relationships. NoSQL for flexible schemas and massive writes.

When: Most systems use both — PostgreSQL + Redis + Elasticsearch.

Key phrase: "Which tool for which job — not SQL vs NoSQL."
