11. Rate Limiting

What it is and why it matters

Rate limiting controls how many requests a user, IP, or API key can make within a time window. Without it, a single abusive user can overwhelm your servers, cause a DDoS-like effect, or rack up huge bills for you.

Rate limiters are placed before your application — at the API gateway, load balancer, or a dedicated service. They reject excess requests with HTTP 429 (Too Many Requests) so your backend never sees them.

The five algorithms (know at least three)

AlgorithmHow it worksMemoryBurst supportAccuracy
Token bucketBucket holds N tokens. Refills at R tokens/sec. Each request takes 1 token. If bucket empty → reject.Low (1 counter + 1 timestamp per user)Yes (burst up to bucket size)Good
Leaky bucketRequests enter a queue (bucket). Processed at fixed rate. If queue full → reject.Low (queue + timer)No (smooths traffic)Good
Fixed windowCounter resets every N seconds (e.g., 100 requests per minute). Simple.Lowest (1 counter per window)Yes (but spike at boundary)Poor at boundaries
Sliding window logStore timestamps of all requests. Window slides continuously.High (stores all timestamps)NoBest
Sliding window counterCombine fixed window counter with weighted previous window estimate.Low (2 counters)YesGood (~2% error)

Fixed window boundary problem

100 req/min limit. User sends 100 at :59 and 100 at :01 — 200 in 2 seconds, both windows "under limit." Sliding window prevents this by considering request history continuously.

Distributed rate limiting

When you have multiple servers, rate limit state must be shared. The standard approach: Redis.

// Simple token bucket in Redis (Lua script for atomicity): local key = KEYS[1] // "rate_limit:user:123" local limit = tonumber(ARGV[1]) // max tokens local refill = tonumber(ARGV[2]) // refill rate local current = redis.call("GET", key) or limit if tonumber(current) > 0 then redis.call("DECR", key) return 1 // allowed else return 0 // rate limited end
Interview angle: Both Mercari and Indeed ask rate limiter questions. Start with: "I'd use a token bucket algorithm — it's simple, memory efficient, and allows natural bursts. For distributed rate limiting, I'd use Redis with Lua scripting for atomic operations." Then discuss trade-offs (Redis as SPOF, using Redis Cluster with consistent hashing).

12. Idempotency & Retry Patterns

The problem

Networks fail. Timeouts happen. When a client sends a request and the connection times out, the client doesn't know if the server processed the request or not. If it retries, the operation might execute twice. In a payment system, that means double-charging the customer. In a messaging system, that means sending a duplicate message.

Idempotency is the property that an operation can be applied multiple times with the same result as applying it once.

Idempotent by nature vs. by design

OperationIdempotent?Why
SET balance = 100Yes (naturally)Setting to 100 repeatedly is the same as doing it once
SET balance = balance + 50NoEach execution increases by 50 — applying twice gives +100
DELETE /users/123Yes (naturally)Deleting a deleted user is a no-op
POST /payments (create)No (needs design)Each call creates a new payment — need idempotency key
PUT /users/123 (full update)Yes (naturally)Sending the same full representation is idempotent
PATCH /users/123 (partial)MaybeDepends on the patch: {"op": "replace", ...} is; {"op": "increment", ...} isn't

Idempotency keys — the pattern

The client generates a unique UUID (idempotency key) and sends it with the request. The server remembers the result keyed by this UUID. If the client retries with the same key, the server returns the stored result without re-executing the operation.

// First attempt: POST /payments { amount: 100, idempotency_key: "abc-123" } Server: processes payment, stores { key: "abc-123", result: "success", tx: "txn_456" } // Retry (client timeout → client resends same request): POST /payments { amount: 100, idempotency_key: "abc-123" } Server: finds existing result for "abc-123" → returns it without processing → { status: "already_processed", transaction: "txn_456" }

Retry strategies

StrategyHow it worksWhen to use
Immediate retryTry once after short delayTransient failures (DNS hiccup, connection reset)
Exponential backoffWait 1s, 2s, 4s, 8s... between retriesStandard pattern for most failures
JitterAdd ±25% randomness to backoff delayPrevent thundering herd when many clients retry
Circuit breakerAfter N failures, stop retrying. Periodically test if recovered.Prevent cascading failures

Common retry scenarios

Failure typeRetry?Strategy
HTTP 503 (Service Unavailable)YesExponential backoff + jitter, 3-5 attempts
HTTP 504 (Gateway Timeout)YesImmediate retry once, then backoff
HTTP 429 (Rate Limited)YesCheck Retry-After header, respect it
HTTP 4xx (Client Error)NoRetrying won't help (bad request, auth failure)
HTTP 5xx (Server Error)YesBackoff + jitter
Connection timeoutYesImmediate retry once, then backoff
Your context: Your messaging system with RabbitMQ uses at-least-once delivery. When a consumer processes a message and crashes before acking, the message is redelivered. If your message handler is idempotent (same message ID results in no duplicate), you get exactly-once processing semantics without exactly-once delivery guarantees. This is the standard production pattern.

13. Microservices vs Monolith

The monolith

One codebase. One deployment unit. All features — user management, payments, notifications, search — run in the same process, sharing the same database.

Advantages: Simpler to develop (no service boundaries to manage). Easier to test end-to-end. No network latency between features. Single deployment. Simple transaction management across features (DB handles it).

Disadvantages: Can't scale features independently — the entire app must scale together. One slow feature (e.g., image processing) can starve other features of resources. Large codebase becomes hard to reason about. Long CI/CD pipelines. Technology lock-in (whole app uses one stack).

Microservices

Each feature or domain is a separate service with its own codebase, deployment, database, and team. Services communicate over the network (HTTP/gRPC) or via message queues.

Advantages: Independent scaling per service (scale only the payment service, not notifications). Independent deployment (deploy one service without taking down the whole app). Technology diversity (each service can use the best language/DB for its job). Smaller, more focused codebases. Team autonomy.

Disadvantages: Complexity explosion — service discovery, API gateways, distributed tracing, circuit breakers, eventual consistency, data duplication. Network latency between services. Testing across service boundaries is hard. Debugging a transaction spanning multiple services is painful.

The comparison matrix

DimensionMonolithMicroservices
DeploymentSingle unit — one deploy affects everythingIndependent — deploy one service at a time
ScalingMust scale the entire appScale individual services by demand
Development speed (early)Fast — no cross-service orchestrationSlow — must define boundaries, contracts, infra
Development speed (late)Slow — large codebase, merge conflictsFast — teams work independently
TestingSimple E2E — one processContract testing, integration testing needed
DatabaseSingle DB, ACID transactions across featuresDatabase per service, eventual consistency across services
Operational costLow — one app to monitor, deploy, debugHigh — observability stack, service mesh, container orchestration
Team structureWorks for small teams (1-10)Designed for multiple teams (10+)

The biggest mistake: premature microservices

The most common anti-pattern is starting with microservices because "they're the modern way" or "Netflix does it." The result: you spend all your time building infrastructure (service discovery, distributed tracing, message brokers) instead of building product features.

The industry consensus: Start with a modular monolith. Use well-defined modules with clear interfaces. If and when a module needs independent scaling or a separate team owns it, extract it into a microservice. This is called the "monolith-first" or "extract microservices" approach.

Common microservice decomposition patterns

By domain (most common): Split by business domain — User Service, Order Service, Payment Service, Notification Service. Each owns its data and domain logic.

By subdomain (DDD): Following Domain-Driven Design bounded contexts. Each bounded context becomes a service.

By capability: Split along technical capabilities — Search Service, Analytics Service, Reporting Service.

Strangler fig pattern: Gradually replace monolith features with microservices one at a time. The monolith shrinks as services are extracted. Low risk, incremental migration.

Your context: Your NestJS apps follow a modular monolith pattern — @Module-decorated feature modules that could become microservices later. This is exactly the right approach. When interviewing at Mercari (which runs Go-based microservices), you can say: "My NestJS apps are modular monoliths with clear module boundaries. I'd extract the messaging module into an independent service when it needs to scale independently."

Versioning strategies

StrategyExampleProsCons

Recommendation: URL path (/v1/) — most visible, easiest to route and document.

Pagination

MethodExampleBest for
Offset/users?offset=20&limit=10Simple UIs, admin panels
Cursor/users?cursor=abc123&limit=10Infinite scroll, real-time feeds

Cursor advantage: Doesn't break when items are inserted/deleted during pagination. Offset can skip or duplicate items.

Error response format

{ "error": { "code": "VALIDATION_ERROR", "message": "Email format is invalid", "details": [ { "field": "email", "issue": "must be a valid email address" } ] } }

Always return: error code (machine-readable), message (human-readable), details (field-level issues). Use proper HTTP status codes (400, 401, 403, 404, 409, 429, 500).

Interview angle: When designing an API, state your pagination strategy upfront: "I'll use cursor-based pagination for the feed — it's stable under concurrent writes." Mention versioning: "I version via URL path for simplicity."
