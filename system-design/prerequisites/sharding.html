Sharding & Partitioning

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

**Sharding** is the horizontal partitioning of database tables, splitting a single large database into smaller, faster physical nodes. Selecting a high-cardinality **Shard Key** is critical to distribute load evenly and avoid processing bottlenecks.

1. Shard Key Cardinality & Distribution

The field used to determine which database node stores a given record:

  • High Cardinality: Keys with millions of unique values (e.g. user_id). Promotes uniform write distribution across node pools.
  • Low Cardinality: Keys with few states (e.g. is_active, gender). Causes severe data skewing, piling up workload onto a single physical server (hotspotting).

2. Logical vs. Physical Sharding

Decouples database routing mapping from direct machine hardware:

  • Logical Shards: Dividing table rows into thousands of smaller logical chunks (e.g. logical partition 1 to 1024).
  • Physical Shards: actual servers running database instances. A central router maps groups of logical shards (e.g. 1-256) to Server A, and 257-512 to Server B. This abstraction makes migrating datasets during scaling simple.

3. Cross-Shard Joins (Scatter-Gather)

Queries that filter on columns other than the shard key must query every node:

  • Scatter-Gather Trap: A router broadcasts the query to all nodes in parallel, waits for all results, aggregates them, and returns to client.
  • Cost: High latency bounded by the slowest node, network packet congestion, and coordinator CPU exhaustion.

4. Resharding & Data Migration

What happens when data volume outgrows the capacity of current shards?

  • Downtime Rebalancing: Halt writes, re-hash records using new partition counts, and copy data. High operational downtime.
  • Online Migration: Adding nodes and migrating logical shards in the background using dual-writing (writing to both old and new shards during migration) and CDC (Change Data Capture) pipelines.

How to Answer in an Interview

If asked: "How do you design a partitioned database for a globally scaling service?"

Say: "First, I'll identify a high-cardinality shard key (e.g. user_id) to ensure even partitioning. Second, I will decouple logical shards from physical servers, mapping a large count of logical partitions to active hardware nodes. This allows us to rebalance data easily. Finally, I will structure query rules: queries must supply the shard key to prevent scatter-gather fanouts, while secondary lookup tables or index caches will handle secondary search fields."
