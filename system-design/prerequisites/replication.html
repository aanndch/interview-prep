Database Replication

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

**Replication** is the mechanism of copying data across multiple physical servers to guarantee durability, optimize read capacity, and survive machine crashes. The core trade-off centers on synchronous consistency vs. asynchronous replication lag.

1. Sync vs. Async Replication

Decides when client writes are acknowledged as successful:

  • Synchronous: The primary write blocks until all replicas acknowledge the update. Guarantees zero stale reads, but makes write latency compound ($O(\text{slowest replica})$). If a replica crashes, the cluster rejects writes.
  • Asynchronous: Primary writes commit immediately and replicate in the background. Extremely fast, but exposes a data loss window if the primary crashes before background flushes execute.

2. Replication Lag & Stale Reads

Occurs when read requests hit replicas that are behind the primary:

  • In asynchronous systems, network congestion or high node CPU loads slow down replication synchronization.
  • If a user performs a write and immediately triggers a read, they may pull from a lagging replica, causing **stale read anomalies** (the update appears to have vanished).

3. Read-Your-Writes Consistency

A guarantee that a client will always see updates they submitted themselves:

  • Session Stickiness: Route the client's reads directly to the primary node for a specific time window (e.g. 5 seconds) following a write operation.
  • Lag Tracking: The client stores its last transaction logical timestamp. It queries replicas but rejects reads unless the replica version meets or exceeds that timestamp.

4. Monotonic & Consistent Prefix Reads

Ensures user read continuity over distributed systems:

  • Monotonic Reads: Guarantees that if a user sees a specific state, they will never see an older, stale version on subsequent queries (prevents reading from Repl 1, then hitting lagging Repl 2).
  • Consistent Prefix Reads: Guarantees that readers see updates in the exact order they occurred at the primary, avoiding causal out-of-order anomalies.

How to Answer in an Interview

If an interviewer asks: "How do you design a replication architecture to support global scaling?"

Say: "I will set up active-passive primary replication, routing write operations to a primary instance and offloading read queries to a pool of horizontal read replicas. Because asynchronous replication introduces lag, I will mitigate stale reads for users viewing their own updates by implementing Read-Your-Writes consistency: routing a client's reads directly to the primary node for a brief period (e.g., 5 seconds) immediately after they execute a write."
