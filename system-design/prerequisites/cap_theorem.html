CAP Theorem

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

The **CAP Theorem** states that a distributed data store can simultaneously provide at most two of three guarantees: **Consistency** (every read receives the most recent write or an error), **Availability** (every non-failing node returns a non-error response), and **Partition Tolerance** (the system continues to operate despite network partitions).

1. Partition Tolerance is Mandatory

In distributed physical architectures, network drops, packet latency, and server crashes are inevitable. Therefore, you **cannot** opt out of Partition Tolerance (P).

The core trade-off during a partition is: CP (Consistency/Partition Tolerance) vs. AP (Availability/Partition Tolerance).

2. PACELC Theorem (The Extension)

CAP only dictates system behavior *during a partition*. PACELC defines normal state tradeoffs:

  • If there is a Partition, trade-off Availability vs. Consistency;
  • Else (normal state), trade-off Latency vs. Consistency.
  • Example: MongoDB defaults to CP (Consistency over Availability on partition). Normally, it enforces strong consistency, sacrificing Latency to confirm writes.

3. CP Systems (Strong Consistency)

Prioritizes data correctness. If network partitions prevent nodes from agreeing on updates, the system rejects writes and reads to avoid returning stale data.

  • Quorum Math: Requires write quorum ($W$) and read quorum ($R$) to satisfy W + R > N. Replicas in the minority partition cannot form a majority (Q = ⌊N/2⌋ + 1) and halt.
  • Odd Node Counts: We use odd node sizes (3, 5) because a 4-node cluster still requires 3 nodes for a quorum majority, meaning it has the same partition tolerance as a 3-node cluster while adding consensus overhead.
  • Best for: Financial transactions, inventory counts, banking APIs.

4. AP Systems (High Availability)

Prioritizes accessibility. Isolated partitions accept writes independently. Data is reconciled later when the network heals, leading to eventual consistency.

  • Last-Write-Wins (LWW): Resolves conflicts using physical wall-clock timestamps. It is prone to clock drift (e.g. NTP sync delay), which can silently overwrite newer writes with older ones.
  • Vector Clocks: Tracks logical causal history by keeping counters of modifications across nodes [Node_A: 1, Node_B: 3], detecting write conflicts reliably without relying on server clocks.
  • Best for: Social feeds, recommendation engines, shopping carts, search indexes.

How to Answer in an Interview

If asked: "How do you decide between a relational database and a NoSQL database for scaling?"

Say: "It comes down to CAP. For transactional domains (e.g., checkout pages or ledgers), consistency is non-negotiable. I will choose a CP model (like PostgreSQL replication with synchronous quorum commits) to prevent double-spending, even if network partitions drop client availability. For metadata, social feeds, or logs, availability is key. I will use an AP system (like DynamoDB or Cassandra) where replication is asynchronous and nodes resolve split state eventually."
