Load Balancing

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

A **Load Balancer** acts as a reverse proxy, distributing incoming network traffic and user requests across a pool of backend servers to prevent resource saturation, guarantee service availability, and maximize system throughput.

1. L4 vs. L7 Load Balancing

Decides the routing layers and intelligence of the proxies:

  • L4 (Transport Layer): Routes based on TCP/UDP ports and IP addresses. Bypasses package content inspection. Extremely fast, minimal CPU overhead, but cannot route based on HTTP headers, cookies, or path strings.
  • L7 (Application Layer): Inspects HTTP request details, URLs, headers, cookies, or payloads. Can terminate SSL handshakes. Enables path-based routing (e.g. /api/users to User-Service) and cookie sticky sessions, but incurs higher CPU overhead.

2. Session Persistence (Sticky Sessions)

Ensures subsequent requests from a client are routed to the same backend server:

  • IP Hashing: Hashes client IP to map to a specific server. Fast, but causes uneven load if a corporate proxy routes thousands of users under a single NAT IP.
  • Cookie-based Stickiness (L7): The LB injects a cookie identifying the target backend. Highly accurate, but binds client routing state.
  • Best Practice: Build stateless application servers and externalize sessions to a shared Redis store, rendering session stickiness unnecessary.

3. Passive vs. Active Health Checks

Ensures traffic is directed only to healthy nodes:

  • Active Checks: The LB proactively pings a designated endpoint (e.g. /healthz) at regular intervals (e.g. every 5s). Unhealthy endpoints are immediately taken out of rotation.
  • Passive Checks: The LB monitors inline connection responses. If a backend drops TCP connections or throws repeated 502/503 errors, it is dynamically flagged.

4. GSLB (Global Server Load Balancing)

Routes traffic across multiple geographic data centers or cloud regions:

  • Mechanism: Uses DNS queries to return IP addresses based on client location, network latency, or server capacity.
  • Issues: DNS caching at client browsers or ISPs (TTL delays) slows down failover response times when an entire region goes down.

How to Answer in an Interview

If asked: "How do you design a load balancing tier for a highly scaling app?"

Say: "I will deploy a multi-tiered load balancing architecture. At the DNS level, I will use GSLB (geographic routing) to map users to their closest region. At the edge, I'll deploy L7 load balancers (like Nginx or AWS ALB) to terminate SSL, inspect HTTP headers, and route path-specific resources to target microservice groups. For internal backend communication, I will use L4 balancers or a Service Mesh (like Envoy/Istio) for high-performance TCP routing based on Least Connections, backed by active health checks on /readyz endpoints."
