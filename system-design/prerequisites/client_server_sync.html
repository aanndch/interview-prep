Client-Server Sync & Fetching

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

How mobile and web applications query resources shapes frontend performance. While **REST** requires discrete resource endpoints that cause client-side N+1 latency cascades, **GraphQL** fetches custom nested arrays in a single round-trip, and **gRPC** leverages HTTP/2 multiplexed binary serialization to optimize server-to-server calls.

1. REST APIs & Over-fetching

Traditional resource-oriented mapping utilizing HTTP verbs (GET, POST):

  • N+1 Problem: To render a user page with posts and comments, the client fetches the user data, then triggers $N$ subsequent calls to fetch the posts, then comments.
  • Over-fetching: Endpoints return entire database columns (e.g. user address, settings) even if the UI only needs the user's name.
  • Benefit: Simple, highly cacheable at edge CDNs via static URL requests.

2. GraphQL & Under-fetching

A declarative query engine enabling clients to request nested structures:

  • Single Roundtrip: Clients submit a schema request listing exact parameters: { user { name, posts { title } } } in a single HTTP POST.
  • DataLoader Pattern: Resolves internal N+1 query fanouts in the backend by batching and caching database keys into single SQL queries.
  • Downside: Very difficult to cache at edge proxies because all requests use HTTP POST.

3. gRPC & HTTP/2 Streaming

High-performance remote procedure call framework developed by Google:

  • Serialization: Uses binary Protocol Buffers instead of textual JSON, decreasing CPU parsing limits.
  • HTTP/2 Multiplexing: Runs multiple concurrent requests over a single TCP connection socket. Supports bi-directional streaming.
  • Downside: Poor native web browser support, primarily used for microservice-to-microservice internal RPC communications.

4. Client-side Optimistic Updates

Providing the illusion of zero latency on mobile/web connections:

  • Mechanism: When a user likes a post, the UI instantly increments the count and shows a filled heart state before the network API request completes.
  • Failures: The client buffers the original state. If the API throws a 500 error, the client rolls back the UI state and displays a retry banner.

How to Answer in an Interview

If asked: "How do you design a data syncing API layer for mobile clients in poor network conditions?"

Say: "To optimize mobile clients, I will avoid N+1 REST endpoints and use a BFF (Backend-for-Frontend) or GraphQL layer to aggregate data. The mobile client submits a single query describing the nested data shape, reducing client-to-server round trips to exactly 1. Inside the backend network, we will run the gRPC protocol using Protocol Buffers over HTTP/2 multiplexed streams to retrieve downstream resources at microsecond limits."
