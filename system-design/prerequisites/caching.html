Caching

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

A **Cache** is a high-speed, temporary storage layer that keeps copies of frequently requested data in volatile memory (RAM, e.g., Redis, Memcached) to reduce database load, minimize network latencies, and speed up client response times.

1. Caching Strategies

How data is synchronized between the client, cache, and database defines the strategy:

  • Cache-Aside (Lazy Load): App queries cache. On miss, it queries database, updates cache, and returns. Simple, but tolerates stale cache data until TTL expires.
  • Write-Through: App writes to cache, which synchronously updates the DB before completing. High consistency, but adds write latency.
  • Write-Behind (Write-Back): App writes to cache, acknowledging immediately (~2ms). Cache writes to DB asynchronously in batches. Highly scalable, but carries data loss risk if the cache node crashes. Mitigations include Write-Ahead Logs (WAL) on the cache, or using Kafka as a durable write buffer.

2. Cache Eviction Policies

Since RAM is expensive and limited, caches must evict old entries when full:

  • LRU (Least Recently Used): Discards the item that hasn't been accessed for the longest time. Best for general workloads.
  • LFU (Least Frequently Used): Discards the item accessed the fewest number of times. Prevents one-hit-wonders from occupying memory.
  • TTL (Time-to-Live): Key automatically expires after a specified epoch window.

3. Cache Stampede (Thundering Herd)

Occurs when a highly popular cache key expires, causing thousands of concurrent read requests to miss the cache and hit the database simultaneously, causing a DB meltdown.

  • Mutex Locks: Only the first thread on a cache miss acquires a lock (e.g. Redis SETNX) to query the DB and update cache; other threads block, retry, or serve stale data.
  • XFetch Algorithm: A probabilistic early expiration algorithm. As keys approach their expiration threshold, read requests probabilistically trigger a background recompute before the key is deleted, avoiding synchronous misses.
  • TTL Jitter: Adding random noise to expiration times so keys do not expire simultaneously.

4. Edge Caching & CDNs

Caching static files (images, JS, CSS) at edge servers geographically close to the user (Points of Presence - PoPs) to bypass hit delays to the main origin region.

  • Origin Pull: CDN fetches assets from origin on first miss.
  • Origin Push: Uploading assets directly to the CDN in advance.

How to Answer in an Interview

If asked: "How do you design caching in a high-concurrency architecture?"

Say: "I would deploy a distributed Redis Cluster using the Cache-Aside pattern for read-heavy operations, invalidating cached entries on write events. To handle cache stampedes on hot keys, I would implement Mutex locking at the app layer or pre-warm keys prior to campaigns. For write-heavy analytics tracking, I would use the Write-Behind (Write-Back) pattern to batch and queue writes, shielding our primary relational database from ingestion spikes."
