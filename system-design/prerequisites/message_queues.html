Message Queues

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

A **Message Queue** is an asynchronous communication broker used to decouple services, buffer spikes in load, and execute background transactions. Examples include broker-oriented structures (RabbitMQ) and persistent event logs (Apache Kafka).

1. Delivery Semantics

How many times will a message reach the consumer node? Choose based on consistency requirements:

  • At-Least-Once (Most Common): Message retried until acknowledged. No data loss, but duplicate processing is possible. **Requires consumer idempotency** (e.g. checking transaction keys in DB).
  • At-Most-Once: Message sent once. Rejects retries if failures happen. Prevents duplicate logs, but tolerates packet loss.
  • Exactly-Once: Relies on transactional API commits (coordinator/Kafka transactions). High complexity and performance overhead.

2. Kafka (Log) vs. RabbitMQ (Broker)

Deciding between queues dictates system scale limitations:

  • Apache Kafka: Persistent append-only disk log. Messages are not deleted on read. Consumers keep pointer offsets and can "replay" historic streams. Outstanding throughput, great for analytics/pipelines.
  • RabbitMQ: Smart broker routing. Messages are deleted as soon as consumer acknowledges them. Advanced routing capabilities, great for simple task queues.

3. Backpressure & Overload Mitigations

What happens when downstream consumers cannot keep up with producer ingestion rates?

  • Scale Consumer Nodes: Automatically spin up more workers (horizontal scaling) to match lambda rates.
  • Rate Limiting Producers: Force producers to slow down (throw HTTP 429) to prevent queue exhaustion.
  • Buffer Storage expansion: Let queue size grow (Kafka retention), accepting processing latency degradation.

4. Poison Pills & Dead Letter Queues

A "poison pill" is a corrupted message that repeatedly crashes the consumer worker code.

  • Dead Letter Queue (DLQ): If a message fails validation or processing after maximum retries (e.g. 5 times), the broker routes it to a designated DLQ.
  • Benefit: Prevents corrupted records from blocking the entire pipeline queue while alerting developers to debug issues.

How to Answer in an Interview

If asked: "How do you design an order processing system that can handle huge traffic spikes?"

Say: "I will put a message broker (like Apache Kafka) between the order API and payment/fulfillment workers. The API acts as a producer, immediately writing orders to the log and returning Success, shielding the DB from sudden spikes. The payment workers act as consumers, pulling messages from Kafka partitions at their own pace. If a spike occurs, Kafka buffers the tasks (backpressure). If a payment transaction fails repeatedly (poison pill), we route it to a Dead Letter Queue (DLQ) after retries to prevent clogging the pipeline."
