Scalability

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

Scalability is the ability of a system to handle increasing load by adding resources without compromising performance or stability.

1. What is "Load"?

Before scaling, you must measure what kind of stress the system is experiencing. Load is multidimensional:

  • QPS (Queries Per Second): The rate of requests entering the system.
  • Concurrent Users: Active connections kept open simultaneously.
  • Read/Write Ratio: Read-heavy systems (e.g., Twitter, blogs) scale differently than write-heavy systems (e.g., IoT logs).
  • Data Volume: The storage growth rate. Storing TBs of media requires specialized blob storage (like S3) instead of databases.

2. Stateless vs. Stateful Tiers

The architecture layer determines the ease of scaling:

  • Stateless Tier (App Servers, APIs): Servers do not store session state or user data locally. Any server can handle any request. Scaled easily by adding nodes behind a Load Balancer.
  • Stateful Tier (Databases, Caches): Servers persist data. Scaling requires distributed consensus, sharding, or replication, making it significantly more complex.
  • Best Practice: Keep app tiers stateless. Offload session data to distributed caches like Redis.
Scaling Types: Vertical vs. Horizontal

In interviews, you must weigh the tradeoffs of how you choose to scale your systems. Below is a breakdown of the two paradigms.

Attribute Vertical Scaling (Scale Up) Horizontal Scaling (Scale Out)
Mechanism Adding CPU, RAM, or storage to a single server. Adding more server nodes to share the load.
Implementation Pro No code or architecture changes needed. Con Requires Load Balancers and stateless code.
Capacity Limit Con Hard hardware limits. Cannot scale infinitely. Pro Virtual infinite scale by adding nodes.
Redundancy Con Single Point of Failure (SPOF). Pro High availability. One node failure won't down the app.
Database scaling Easy. Default for relational DBs (PostgreSQL, MySQL). Complex. Requires sharding, consistent hashing, or NoSQL.

How to Answer in an Interview

Always start by assessing the load requirements (e.g., read vs. write heavy, peak traffic profiles). For stateless tiers, state: "I will scale the application layer horizontally behind a load balancer so we can scale out linearly. For stateful database nodes, I will rely on vertical scaling initially, then plan for horizontal scaling via read-replicas, caching layers, and database sharding when we hit capacity limits."
