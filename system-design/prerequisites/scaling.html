7. Replication & Partitioning

Replication — copies of the same data

Replication means keeping identical copies of your data on multiple servers. The goal is fault tolerance (if one server dies, another has the data) and read scalability (spread read traffic across replicas).

Single-leader replication (most common)

One node is the leader (aka primary). All writes go to the leader. One or more followers (aka replicas) copy data from the leader asynchronously. Reads can go to any follower.

Client → WRITE → [Leader] ↘ async replication [Follower 1] [Follower 2] [Follower 3] Client → READ → (any of these)

Trade-off: If the leader fails, one follower must be promoted to leader (downtime of seconds). If a follower is lagging behind, reads from it return stale data. You can use synchronous replication (wait for all followers to ack) for stronger consistency, but that slows writes.

Multi-leader replication

Multiple nodes accept writes. Used when you have data centers in different regions and want local writes (e.g., write to Tokyo leader, which replicates to New York leader). The challenge: write conflicts. If two users edit the same document in different regions simultaneously, whose change wins? Solutions include "last write wins" (LWW), conflict-free replicated data types (CRDTs), or custom merge logic.

Partitioning (sharding) — splitting data across nodes

Replication gives you copies; partitioning gives you pieces. Each shard holds a different subset of data. Together they hold the entire dataset.

Replication + partitioning together

In real systems, they're combined. Each shard is replicated for fault tolerance:

Your context: Your messaging system with 1M+ messages/week would benefit from time-based partitioning on the messages table (partition by month/year). Old partitions can be archived or compressed. For reads, you could add read replicas of PostgreSQL so analytics dashboard queries don't slow down message delivery.

8. Message Queues

The core concept

A message queue is a buffer between a producer (who sends messages) and a consumer (who processes them). The producer doesn't wait for the consumer — it just drops the message into the queue. The consumer picks it up when ready.

This decoupling is the key benefit: if the consumer crashes, messages pile up in the queue and get processed when it recovers. If the producer spikes, the queue absorbs the burst instead of overwhelming the consumer.

The three messaging patterns

Point-to-point (competing consumers): One message is delivered to exactly one consumer. Multiple consumers can pull from the same queue, but each message goes to only one. This is how you distribute work across workers. Example: your P2P texting — a message from user A to user B is picked up by one delivery worker.

Pub/Sub (broadcast): One message is delivered to all subscribers. Each subscriber gets a copy. Example: when a new user signs up, the "user.created" event goes to the email service, the analytics service, and the notification service — all independently.

Work queues: Like point-to-point but with explicit acknowledgment. The worker processes the message and sends an ack back. If the worker crashes without acking, the message is redelivered to another worker. This gives you "at-least-once" delivery.

Delivery guarantees

GuaranteeMeaningHow to achieve
At-most-onceMessage may be lost but never duplicatedSend and forget. No ack, no retry.
At-least-onceMessage is never lost, but may be delivered multiple timesConsumer acks after processing. If it crashes before acking, message is redelivered. This is the default in RabbitMQ.
Exactly-onceMessage is delivered exactly once — no loss, no duplicatesRequires idempotent consumers + distributed transactions. Very expensive. Most systems use at-least-once + idempotency.

RabbitMQ vs Kafka — the decision framework

Choose RabbitMQ when:

  • You need flexible routing (direct, topic, fanout, header exchanges)
  • You want messages removed after consumption (don't need replay)
  • You have moderate throughput (tens of thousands/sec — your 1M/week is ~1.6/sec, well within range)
  • You need RPC-style request/reply patterns
  • You want per-message acknowledgments with dead-letter queues for failures

Choose Kafka when:

  • You need extremely high throughput (millions of messages/sec)
  • You need message replay — consumers can re-read old messages
  • You're building an event sourcing / CQRS system
  • You need log compaction (keep only the latest value per key)
  • You need strong partitioning guarantees for ordered processing
Your context: Your P2P texting service uses NestJS + RabbitMQ + WebSockets. This is a textbook work-queue pattern: producers (web/mobile apps) send messages into a queue, NestJS workers consume and deliver via WebSocket. The 1M+ weekly throughput is modest for RabbitMQ. In the interview, explaining your delivery architecture (at-least-once + idempotent handlers + dead-letter queues) will impress the interviewer.

9. WebSockets vs Polling vs SSE

The problem: real-time updates

The web is built on HTTP — request/response. The client asks, the server answers. But what if the server has new data to push? A new message arrives, a stock price changes, a teammate edits a document. You need a way for the server to proactively send data to the client.

Option 1: Short polling (simplest, least efficient)

The client asks "got anything new?" every N seconds. Even when nothing changed, the client sends a full HTTP request.

// Every 5 seconds: Client → GET /new-messages?since=123 Server ← { messages: [] } // empty — wasted request // 5 seconds later: Client → GET /new-messages?since=123 Server ← { messages: [{ id: 124, text: "hello" }] }

Problem: Most polls return nothing, wasting bandwidth and server resources. Latency is bounded by the polling interval (average delay = interval/2).

Option 2: Long polling (better but hacky)

The client sends a request. The server holds it open (doesn't respond immediately). When new data arrives, the server finally responds. The client immediately sends a new long-poll request. This gives near-real-time delivery with fewer wasted requests.

Problem: Still creates many HTTP connections. Middleboxes (proxies, firewalls) may timeout idle connections. The server must manage many open connections.

Option 3: Server-Sent Events (SSE) — one-way from server

The client opens a single HTTP connection. The server sends data as "events" over this connection using a simple text protocol (event: message\ndata: hello\n\n). The connection stays open.

Pros: Simple protocol, works over standard HTTP, auto-reconnect built into browsers (EventSource API).

Cons: Server-to-client only (no client-to-server). Limited browser connections (most browsers allow 6 per domain).

Option 4: WebSockets — full-duplex (what you use)

The client and server establish a persistent TCP connection via an HTTP upgrade handshake (101 Switching Protocols). Once established, both sides can send messages any time — no request/response pattern.

Pros: True bidirectional communication. Low overhead after initial handshake. No polling waste.

Cons: Requires stateful server (can't use stateless HTTP scaling as easily). Sticky sessions on load balancers needed. More complex to manage reconnection logic.

10. REST vs GraphQL

The fundamental difference

REST exposes resources (endpoints like /users/123, /users/123/orders). Each endpoint returns a fixed structure. GraphQL exposes a single endpoint where the client describes exactly what data it wants — and gets only that.

REST — resource-oriented

Each endpoint returns a predefined shape. If the client needs user info and their orders and order items, it might need 3 separate API calls (or the server provides a custom endpoint that returns everything).

// REST: three round trips for related data GET /users/123 → { id, name, email, ...10 more fields } GET /users/123/orders → [{ id, date, total, ...5 more fields }] GET /orders/456/items → [{ productId, qty, price }]

GraphQL — client-driven queries

Single endpoint. The client specifies the exact shape:

// GraphQL: one request, exactly the fields needed POST /graphql { users(id: 123) { name email orders { date total items { productId, qty } } } }

When each shines

Use REST when:

  • Your API is simple CRUD (create/read/update/delete)
  • You want simple caching (CDN, HTTP caching)
  • You serve mostly public data (same response for all clients)
  • You need file upload/download
  • Your team is small and wants simplicity

Use GraphQL when:

  • You have many different clients (web, mobile, third-party) needing different data shapes
  • You have deeply nested relational data (e.g., user → orders → items → product)
  • Your mobile clients need to minimize data transfer
  • You want strong typing and auto-generated documentation (schema introspection)
Your context: Your e-commerce stores used GraphQL — a good fit because product pages need different fields than search results than admin dashboards. Your calendar and analytics dashboards likely used REST — simpler caching, and each view has a known data shape. In interviews, mention: "I choose GraphQL when clients have heterogeneous data needs; REST when caching simplicity matters more."
