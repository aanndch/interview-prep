Indexing

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

An **Index** is a specialized data structure (typically a B-Tree or LSM-Tree) that speeds up data retrieval operations on a database table at the cost of additional write overhead, storage space, and memory memory buffers.

1. Composite Indexes & Leftmost Prefix Rule

An index created on multiple columns: INDEX(city, status, date).

  • Leftmost Prefix Rule: The database can only use this index if the query filters by the leftmost columns in order. Queries filtering on city, city + status, or city + status + date will use the index.
  • Traps: Queries filtering on status or date alone, or status + date, cannot use this index and trigger slow sequential scans.

2. Covering Indexes (Index-Only Scans)

An index that contains all the columns requested by a query.

  • Mechanism: If you execute SELECT name, age FROM users WHERE age > 30 and have INDEX(age, name), the database reads values directly from the index nodes in memory, bypassing the table files on disk (no table heap lookup).
  • Benefit: Extremely fast reads. Promoted as "Index-Only Scan" in PostgreSQL explain logs.

3. Index Selectivity

The ratio of unique values to the total number of records: Selectivity = Unique / Count.

  • High selectivity columns (e.g. user_email, uuid) are highly effective index filters.
  • Low selectivity columns (e.g. is_active, gender) are ineffective. The query engine will ignore the index and run a full table scan because random disk seeks on B-trees are slower than sequential disk reads.

4. Write Overhead Penalties

Indexes are not free. Every write, update, or deletion must also maintain and rebalance the tree structures on disk.

  • A table with 5 indexes will execute writes roughly 5 times slower than an unindexed table.
  • Over-indexing must be avoided. Keep indexes aligned strictly to actual querying metrics.

How to Answer in an Interview

If asked: "A SQL query on a production database is running slowly. How do you address it?"

Say: "I will first run EXPLAIN ANALYZE on the slow query to check if it triggers a sequential scan instead of an index scan. Next, I will check the selectivity of the columns filtered. If filtering on multiple columns, I will verify if a composite index matches the leftmost prefix rule, or create a covering index to enable an index-only scan, avoiding table heap searches entirely."
