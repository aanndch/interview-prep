Performance

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

Performance in distributed systems is defined by two primary metrics: **Latency** (the time taken to process a single request, e.g., in milliseconds) and **Throughput** (the volume of requests the system handles in a given timeframe, e.g., QPS).

1. Tail Latency & Percentiles

Average latency masks issues. We measure performance using percentiles (p50, p95, p99, p99.9):

  • p50 (Median): 50% of requests are faster than this. Outlines the "typical" user experience.
  • p99: 99% of requests are faster. Critical for catching slow edge-case queries, garbage collection pauses, or database lock contentions.
  • Tail at Scale: In a microservice mesh, a parent request fanned out to 100 parallel services (each with 1% tail delay at p99) has a 1 - 0.99100 ≈ 63% chance of experiencing a slow response.
  • Hedged Requests: Fire a duplicate request to a replica if the initial request hasn't returned within the 95th percentile window, canceling the slower one once one returns. This limits tail latency with tiny (~5%) traffic overhead.

2. Little's Law & Queue Sizing

A mathematical law defining concurrency boundaries: L = λ × W

  • L: Average number of active requests in flight (concurrency).
  • λ (lambda): Arrival rate of requests (throughput/QPS).
  • W: Average time a request spends in the system (latency).
  • Database connection pool insight: If your database pool size is 20 connections ($L = 20$), and each query takes 50ms ($W = 0.05$s), your max throughput is $\lambda = L/W = 20 / 0.05 = 400$ QPS. Any higher rate will cause queueing!

3. Amdahl's Law (Parallelism Limits)

Adding more processors or servers does not scale performance linearly. It is limited by the sequential (non-parallelizable) portion of the code: Speedup = 1 / ((1 - P) + P/N).

  • P: Parallelizable proportion.
  • N: Number of processing units.
  • If 10% of a task is strictly sequential (e.g. database writing, locks), the maximum speedup is 10x, even with infinite nodes.

4. Optimization Strategies

Standard architectural patterns to enhance throughput and minimize latency:

  • Connection Pooling: Reusing database connections to save TCP handshakes.
  • Database Indexing: Speeding up searches from $O(N)$ sequential scans to $O(\log N)$ B-tree lookups.
  • Serialization: Using binary protocols like gRPC (protobuf) over plain REST JSON. JSON has high CPU parsing overhead (converting strings to floats, matching keys), whereas Protobuf uses directly byte-shifted field offsets for extremely fast serialization.
  • Caching: Offloading database read hits to local/distributed RAM.

How to Answer in an Interview

When optimizing a slow page/endpoint, use a structured checklist:

"First, I would use EXPLAIN ANALYZE to check if the database queries utilize selective indexes or trigger full table scans. Second, I would inspect the application's connection pool limits to ensure we aren't experiencing queue blockages (applying Little's Law). Third, I'd cache read-heavy entities in Redis. Finally, I would minimize network payload sizes by moving from verbose JSON serialization to binary gRPC schemas for internal service communication."
