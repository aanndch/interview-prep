Reliability & Fault Tolerance

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

Reliability is the probability that a system will perform its required function under stated conditions for a specified period. **Fault Tolerance** is the architectural ability to maintain service availability despite failures in component hardware or software.

1. High Availability (The "Nines")

Availability measures the percentage of uptime. Japanese users and companies hold systems to extreme standards, often requiring at least four nines of availability:

  • 99.9% (Three Nines): ~8.76 hours of downtime per year.
  • 99.99% (Four Nines): ~52.6 minutes of downtime per year. Standard for production APIs.
  • 99.999% (Five Nines): ~5.26 minutes of downtime per year. Reserved for payment systems and critical telecom infrastructure.

2. Essential Failure Metrics

Distributed systems must track recovery times to meet Service Level Objectives (SLOs):

  • MTBF (Mean Time Between Failures): The average time the system runs before a fault occurs. High MTBF indicates high hardware/software stability.
  • MTTR (Mean Time To Recover): The average time it takes to detect, diagnose, and fix an outage. Lowering MTTR via runbooks, monitors, and auto-failovers is key.
  • Formula: Availability = MTBF / (MTBF + MTTR)

Core Mitigation Techniques

Exponential Backoff with Jitter

When downstream service calls fail, immediately retrying will overload the failing node further (called a Thundering Herd). Use exponential delays: Delay = base * 2attempt.

Always add Jitter (random noise) to spread out retry bursts and allow the downstream node to recover.

Idempotent APIs & Keys

Retries introduce duplicate requests. Idempotency guarantees that processing the same request multiple times yields the same result without double-side-effects (e.g. charging a card twice).

Clients send an Idempotency-Key header. The server caches this key in Redis with a TTL. On duplicate detection, it immediately returns the cached response.

How to Answer in an Interview

If an interviewer asks: "How do you handle down dependencies in a highly concurrent API?"

Frame your response: "First, I'd implement a Circuit Breaker at our API Gateway layer to fail fast and prevent thread pool exhaustion. Second, we'll design retry policies with exponential backoff and random jitter. Finally, to ensure transaction safety during payment retries, I will enforce Idempotency-Keys generated by clients and tracked via Redis unique keys, returning the same response for identical tokens."
