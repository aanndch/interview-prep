Data Pipelines: Batch vs. Stream

Prerequisites / Concepts — Distributed Systems Foundations

Core Concept

**Batch Processing** maps static datasets at scheduled intervals to maximize CPU resource efficiency, while **Stream Processing** aggregates infinite, real-time events message-by-message to guarantee sub-second delivery latencies.

1. Batch Processing & MapReduce

Processes bounded historical logs in bulk workloads:

  • Tooling: Apache Spark, Apache Hadoop, AWS EMR, scheduled via Airflow DAGs.
  • Properties: High throughput, optimized disk utilization. The data is bounded (already fully written).
  • Downside: High data latency. Reports run daily or hourly, meaning decisions rely on historical states.

2. Stream Processing & Aggregations

Processes unbounded events as they occur in real time:

  • Tooling: Apache Flink, Spark Streaming, Kafka Streams.
  • Properties: Sub-second latency. Data is processed event-by-event.
  • Out-of-Order Events: Network drops force packets to arrive late. Stream engines use **Watermarks** (logical timestamps) to determine when to close time windows and execute aggregations.

3. Lambda vs. Kappa Architectures

Architectural patterns for syncing real-time and batch results:

  • Lambda: Dual-track processing. A *Speed Layer* (Flink) provides fast, approximate real-time outputs, while a *Batch Layer* (Spark) runs nightly to generate correct historical data. The client merges both in the database.
  • Kappa: Single-track processing. Removes the batch layer entirely. Everything is a stream. To reprocess history, you re-read messages from a persistent Kafka broker log from offset zero.

4. Streaming Window Types

How unbounded records are grouped to execute aggregations:

  • Tumbling Window: Fixed-size, non-overlapping intervals (e.g. aggregate orders every 5 minutes: 12:00-12:05, 12:05-12:10).
  • Sliding Window: Fixed-size, overlapping intervals (e.g. calculate sum of last 10 minutes, updated every 1 minute).
  • Session Window: Dynamic windows triggered by periods of user inactivity.

How to Answer in an Interview

If asked: "How do you design an analytical processing tier for a scaling e-commerce app?"

Say: "I will structure a Kappa architecture to decouple streaming telemetry. We will write clickstream logs to Kafka partitions. Real-time metrics (like active user carts or promo spikes) will be processed by Apache Flink using tumbling windows. For deep analytics and historical business auditing, we will read the log records asynchronously into an S3 data lake, executing scheduled Apache Spark batch processing jobs to optimize resource cost limits."
