1. CORE FUNDAMENTALS
================================================================================

1.1 SCALABILITY
--------------
Definition: Ability of a system to handle increasing load by adding resources.

Two Types:
- Vertical Scaling (Scale Up):
  - Increase CPU/RAM of single server
  - Easier to implement, no code changes
  - Limited by hardware capacity
  - Single point of failure

- Horizontal Scaling (Scale Out):
  - Add more servers to distribute load
  - Supports massive growth
  - Better fault tolerance
  - Introduces complexity in data consistency
  - Database sharding is a form of horizontal scaling

1.2 RELIABILITY & FAULT TOLERANCE
----------------------------------
Failures are inevitable in distributed systems.

Strategies:
- Replication: Duplicate services/data for failover
- Retry & Backoff: Auto-reattempt failed requests with exponential delays
- Circuit Breakers: Block requests to failing services temporarily
- Rate Limiting: Protect services from being overwhelmed
  Algorithms:
  - Token Bucket: Tokens refilled at fixed rate; each request consumes one. Allows bursts up to bucket capacity.
  - Sliding Window Log: Timestamp-based rolling window. Precise but memory-intensive.
  - Sliding Window Counter: Hybrid of fixed window + sliding log. Smooths edge-case spikes.
  - Leaky Bucket: Constant-rate processing; excess requests are queued or dropped.
  - Fixed Window: Count requests per time unit. Simple but allows burst at window boundaries.
- Graceful Degradation: System continues with reduced functionality
- Idempotency: Ensures repeated identical requests produce the same result.
  - Idempotency Key: Unique key sent by client (e.g., idempotency-key header); server deduplicates
  - Critical for payment, order creation, and any operation with retries
  - Typically implemented via DB unique constraint or cache with TTL
  - Safe methods: GET, PUT, DELETE are naturally idempotent; POST is not
  - Example: Stripe's Idempotency-Key header prevents duplicate charges

1.3 PERFORMANCE
---------------
Two critical metrics:
- Latency: Time for system to respond
- Throughput: Number of requests per second

Optimization strategies:
- Caching to reduce network round-trips
- Database indexing for faster reads
- Database connection pooling: Reuse a pool of pre-established DB connections to avoid connection overhead
  - Reduces latency on each request (no TCP handshake/authentication)
  - Limits concurrent DB connections under load
  - Common pool sizes: 10-30 per app instance (rule of thumb: (core_count * 2) + 1)
  - Tools: HikariCP (Java), PgBouncer (PostgreSQL), ProxySQL (MySQL)
- Data compression
- Batch network calls
- Use gRPC for internal services (smaller payloads, faster serialization)
- WebSockets for real-time connections

1.4 CAP THEOREM
----------------
A distributed data store can only provide 2 of 3 guarantees:

- Consistency (C): Every read receives most recent write
- Availability (A): Every request receives non-error response
- Partition Tolerance (P): System operates despite network failures

In distributed systems, network partitions are inevitable.
Must choose between CP or AP during partitions:

- CP Systems (Banking, Payment):
  Strong consistency, may reduce availability
  Example: PostgreSQL, MongoDB

- AP Systems (Social Feeds, Search):
  High availability, eventual consistency
  Example: Cassandra, DynamoDB

1.5 CACHING
-----------
Caches store copies of data in high-speed memory (RAM).

Types:
- Application Cache: In-memory within service instance, fast but not shared
- Distributed Cache: Shared across nodes (Redis cluster)
- CDN Cache: Static assets from geographically close servers

Caching Strategies:
- TTL (Time-to-Live): Data expires automatically
- Write-Through: Write to cache first, then persist to DB synchronously
- Write-Behind/Write-Back: Write to cache, async persist to DB
- Cache-Aside: Application manages cache invalidation

Cache Invalidation: One of the hardest problems in CS.

1.6 LOAD BALANCING
-------------------
Distributes traffic evenly across servers.

Algorithms:
- Round Robin: Simple rotation of requests
- Least Connections: Route to server with fewest active connections
- IP Hashing: Same client mapped to same server (session persistence)

Types:
- L4 Load Balancer: Works at transport layer (TCP/UDP)
- L7 Load Balancer: Works at application layer (HTTP)

Tools: Nginx, HAProxy, AWS ALB/NLB

1.7 DATABASES
-------------
Relational (SQL): MySQL, PostgreSQL
- ACID properties (Atomicity, Consistency, Isolation, Durability)
- Structured data, schema enforced
- Great for: User profiles, billing, inventory
- JOIN operations, complex queries

Non-Relational (NoSQL): MongoDB, Cassandra, DynamoDB
- BASE model (Basically Available, Soft state, Eventual consistency)
- Flexible schemas, high write throughput
- Great for: Large-scale unstructured data
- Eventual consistency prioritized

1.8 SHARDING & PARTITIONING
-----------------------------
Splitting data across multiple servers based on a key (e.g., UserID).

Sharding Strategies:
- Hash-based: Consistent hashing
- Range-based: Divide by value ranges
- Directory-based: Lookup service maps data to shards

Challenges:
- Complex queries (JOINs across shards)
- Cross-shard transactions
- Rebalancing when adding shards

1.9 CONSISTENT HASHING
-----------------------
A technique to minimize data movement when the number of nodes changes.

Problem: In hash-based sharding (hash(key) % N), adding/removing a node changes N → nearly all keys remap.

How it works:
- Hash both keys and nodes onto a circular ring (0 to 2^32 - 1)
- Each key is assigned to the next clockwise node on the ring
- Adding/removing a node only affects keys in its immediate vicinity
- Only ~1/N of keys need re-mapping on node change (vs nearly all with modulo)

Virtual Nodes:
- Each physical node maps to multiple positions on the ring
- Improves load distribution (prevents hot spots)
- Handles heterogeneous hardware (assign more vnodes to powerful machines)

Applications:
- Distributed caches (Memcached, Redis Cluster)
- DynamoDB, Cassandra (partitioning + replication)
- Load balancers (consistent hashing for sticky sessions)

1.10 REPLICATION
----------------
Primary-Replica: Writes to single node, reads from multiple nodes
Multi-Master: Writes to any node, improves availability, complicates conflict resolution

1.11 INDEXING
--------------
Optimizes data retrieval without scanning entire database.

Types:
- B-tree Indexes: Standard for relational DBs, great for range queries
- Hash Indexes: Exact-match lookups, not for range queries
- Inverted Indexes: Backbone of search engines (Elasticsearch)
- Trie Structures: Used in autocomplete systems
- Bitmap Indexes: Efficient for low-cardinality columns (gender, status)

Advanced Concepts:
- Composite Indexes: Index on multiple columns. Column order matters (leftmost prefix rule).
  Example: INDEX(city, status, date) covers queries on city, city+status, city+status+date.
  Does NOT cover queries on status or date alone.
- Covering Indexes: Index contains all columns needed by a query.
  Query reads entirely from index, no table lookup (index-only scan).
  Significantly faster than non-covering indexes.
- Index Selectivity: Ratio of unique values to total rows.
  High selectivity (e.g., user_email) = efficient filter.
  Low selectivity (e.g., is_active boolean) = poor filter; index may be ignored.
- Partial Indexes: Index only a subset of rows (WHERE clause). Smaller, faster.
- Clustered Indexes: Data rows stored in index order. One per table (e.g., InnoDB primary key).

Trade-offs:
- Indexes speed up reads but slow down writes (INSERT/UPDATE/DELETE must update each index)
- Each index consumes disk space and memory (buffer pool)
- Over-indexing can degrade write performance significantly

1.12 MESSAGE QUEUES & ASYNCHRONOUS PROCESSING
----------------------------------------------
Decouples services for async communication.

Use Cases:
- Background tasks (email, video encoding)
- Load buffering
- Event-driven architectures

Tools: Kafka, RabbitMQ, SQS, Redis Pub/Sub

1.13 OBSERVABILITY & MONITORING
--------------------------------
Three Pillars:
- Metrics: Aggregated data over time (Prometheus, Grafana)
- Logs: Discrete events (ELK Stack)
- Tracing: Track single request across microservices (Jaeger, Zipkin)

Practical Tracing Example (debugging a slow request):
1. A user request hits API Gateway → notification is generated with TraceID
2. TraceID propagates via HTTP headers (x-trace-id) to each downstream service
3. Each service emits spans (service name, operation, start/end timestamps, parent span ID)
4. Jaeger collects spans and reconstructs the full request tree
5. Identifying bottleneck: If auth-service span takes 800ms but db-query span takes 50ms,
   the bottleneck is likely in auth-service business logic (cache miss? external API call?)
6. Integration: correlate TraceID with structured logs for deeper debugging

1.14 DATA PIPELINES & STREAMING
--------------------------------
- Batch Processing: Periodic aggregation (Hadoop, Spark)
- Stream Processing: Real-time events (Kafka, Flink)
- Lambda Architecture: Combines batch + stream layers

1.15 SECURITY & AUTHENTICATION
-------------------------------
Securing services, user identities, and data transit.

Core Concepts:
- OAuth 2.0 & OIDC:
  - Authorization Code Flow with PKCE: Standard for Single Page Apps (SPAs) and mobile apps.
  - Client Credentials Flow: For machine-to-machine backend communication.
- Authentication State:
  - Stateful Session Cookies: Server-side sessions. Secure (HttpOnly, Secure, SameSite flags) and easy to revoke, but harder to scale (requires sticky sessions or shared store like Redis).
  - Stateless JWTs (JSON Web Tokens): Signed tokens containing claims. Scales easily but hard to revoke before expiry.
    - Revocation strategies: Blacklisting/denylisting token IDs in Redis (negates stateless advantage for check operations), or short TTL + sliding expiration with Refresh Tokens.
- API Security:
  - CORS (Cross-Origin Resource Sharing): Browser safety mechanism restricting resource sharing across origins. Uses preflight (OPTIONS) requests.
  - CSRF (Cross-Site Request Forgery): Prevented using SameSite cookie attributes, Anti-CSRF double-submit cookies, or requiring custom headers.
  - Gateway security: Rate-limiting, WAF (Web Application Firewall), and payload validation.

1.16 CLIENT-SERVER DATA FETCHING & SYNCHRONIZATION
--------------------------------------------------
Optimizing query boundaries and UI consistency between frontend and backend.

Data Fetching Paradigms:
- REST APIs: Resource-centric endpoints. Simple and browser-cacheable, but prone to over-fetching (getting fields you don't need) or under-fetching (requiring N+1 network requests).
- GraphQL: Graph-centric single endpoint. Client queries exactly what it needs. Solves over-fetching and under-fetching, but introduces complex backend caching (requires persisted queries or dataloader pattern to prevent database N+1s) and security risks (unbound query depth).
- gRPC: Contract-first binary RPC (HTTP/2). Highly efficient, proto schemas. Great for server-to-server microservices; needs proxy (Envoy) for browser clients.

State Sync & Performance:
- HTTP Caching: Cache-Control headers. `stale-while-revalidate` serves stale cache immediately while background fetching updates.
- Optimistic Updates: UI immediately renders success state before server response completes (e.g., chat message sent, post liked). Rollback on server failure.
- Offline State Sync: Store transactions in IndexedDB/SQLite on client. Sync with server when connection is restored using timestamp/vector clock reconciliation.

3.3 MULTI-STEP PROCESSES (DISTRIBUTED TRANSACTIONS)
----------------------------------------------------
Handling data consistency across multiple microservices without locking resources.

Strategies:
- Two-Phase Commit (2PC):
  - Coordinator prompts all participants to prepare, then commit.
  - Strong consistency, but blocking (if coordinator fails, resources remain locked) and high latency. Not recommended for high-scale microservices.
- Saga Pattern:
  - A sequence of local transactions. Each transaction updates database and publishes an event. If a step fails, Saga executes compensating (rollback) transactions in reverse order.
  - Eventual consistency and non-blocking.
  - Two Implementation Types:
    - Choreography: Decentralized. Services react to events from other services. Simple for small workflows, but hard to trace and debug as complexity grows (spaghetti events).
    - Orchestration: Centralized. A dedicated orchestrator class/service manages transaction states and explicitly tells participants what to execute. Easier to define, test, and monitor, but creates a central coordinator dependency.

- Multi-AZ deployment within region (Tokyo has 3+ AZs per provider)

- LINE Login SDK (OAuth 2.0 + OpenID Connect)
- LINE Messaging API for notifications and customer support (bots)

JAPAN-SPECIFIC SCALING CONSIDERATIONS:
- Rakuten Super Sale / Amazon Prime Day-like traffic spikes (10x+ normal load)
- New Year's (nenga) traffic patterns
- Golden Week and Obon holiday surges
- Anime/game limited-time campaign launch patterns
  - Pre-warm caches before campaign start
  - Queue high-value operations (orders) during peak
  - Sliding window rate limiting at API gateway per-user
- Mobile-first optimization (Japan has ~70% smartphone penetration)

7. TRADE-OFFS TO KNOW
================================================================================

| Concern                          | Trade-off                                      |
|----------------------------------|------------------------------------------------|
| Latency vs Freshness            | Cached data is fast but may be outdated        |
| Consistency vs Availability     | Strong consistency = higher latency, less available |
| Cost vs Scalability             | More replicas = higher infrastructure cost     |
| Complexity vs Maintainability   | Microservices harder to debug than monoliths   |
| REST vs gRPC                    | REST is readable; gRPC is faster but needs schema mgmt |
| SQL vs NoSQL                    | SQL = ACID, structure; NoSQL = flexibility, scale |
| Sync vs Async                   | Sync = simple but blocking; Async = complex but scalable |
| JWT vs Stateful Session          | JWT scales easily; Session is secure and immediately revocable |
| REST vs GraphQL                  | REST is simple, cacheable; GraphQL solves over-fetching but backend caching is complex |
| 2PC vs Saga                      | 2PC has strong consistency but blocks; Saga is eventual consistency, non-blocking |
