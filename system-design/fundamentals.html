1.1 Scalability

The ability of a system to handle increasing load by adding resources.

Why This Matters Interviewers want to see you identify scalability bottlenecks first. Start every design by asking: "Is this system read-heavy or write-heavy?" and "What's the expected QPS growth?"
What Is "Load"? Understanding the Problem

Before you scale anything, you need to understand what kind of load your system faces. Load is more than just "number of users":

  • Requests per second (QPS): How many requests hit your server each second. A photo-sharing app might have 10,000 image uploads/sec.
  • Concurrent users: Users active at the same time. On a shopping site, Black Friday might have 1M concurrent users but relatively few actual purchases.
  • Read/write ratio: Twitter is read-heavy (most users scroll, few tweet). Sensor data logging is write-heavy. A caching strategy depends heavily on this ratio.
  • Data volume: Storing 100 bytes per user is easy. Storing 100MB per user (photos, videos) is hard. You need blob storage (S3), not a database.
  • Growth rate: 100 users today is different from 100K users next month. Design for the expected growth path.
  • Key question to ask: "Is this system read-heavy or write-heavy? What's the expected QPS and data size per user?"
Stateless vs Stateful — Why App Servers Are Easy

Understanding the difference between stateless and stateful services is the key to knowing what to scale vertically vs horizontally:

  • Stateless services (app servers, API servers, workers): Store no data locally. Any request can go to any server. To scale, just add more servers behind a load balancer. No code changes needed for scaling.
  • Stateful services (databases, caches, file storage): Maintain data that must persist across requests. Scaling requires distributing that data (sharding, replication). Much harder.
  • Best practice: Make as many services stateless as possible. Store session data in Redis, not in the app server's memory. Use object storage (S3) for files. The more stateless your architecture, the easier it scales.
  • Example: A Node.js API server with sessions stored locally cannot scale horizontally (user B goes to server 2 and loses their session). Move sessions to Redis → now it's stateless → add as many API servers as needed.
Vertical Scaling Trade-off
Upgrading a single machine (more CPU, RAM, faster SSD). Simple but hits hardware limits. Used for small-to-medium databases, legacy systems.
  • No code changes needed
  • Limited by max hardware capacity
  • Single point of failure — if the machine goes down, everything goes down
  • Example: Moving from 16GB RAM to 64GB on your DB server
Horizontal Scaling Preferred
Adding more machines to distribute the load. The foundation of all large-scale systems.
  • Near-limitless growth (add servers as needed)
  • Better fault tolerance (lose one server, others pick up)
  • Requires load balancer, shared-nothing or distributed state
  • Database sharding is a form of horizontal scaling
  • Example: A web app behind Nginx routing to 10 app servers
Interview Tip Always start with horizontal scaling for stateless tiers (app servers). Reserve vertical scaling for stateful tiers (databases) where horizontal scaling is harder. Say: "I'd scale the app tier horizontally behind a load balancer, and consider sharding the database when we outgrow vertical."

1.2 Reliability & Fault Tolerance

Failures are inevitable in distributed systems. Build for failure from day one.

Why This Matters The single biggest differentiator between junior and senior engineers is anticipating failures. Interviewers probe this by asking "What happens when your cache goes down?" or "How do you handle a database outage?"
What Is Availability? The "Nines"

Availability measures how often your system is up and serving requests. It's expressed as a percentage of uptime per year:

  • 99% ("two nines"): ~3.65 days downtime per year. Acceptable for internal tools, prototypes.
  • 99.9% ("three nines"): ~8.76 hours downtime per year. Typical for consumer web apps.
  • 99.99% ("four nines"): ~52.6 minutes downtime per year. Expected for production systems, API services.
  • 99.999% ("five nines"): ~5.26 minutes downtime per year. Telco, financial infrastructure, mission-critical.
  • Each additional "nine" costs significantly more in infrastructure and operational complexity.
  • Japan context: Japanese users expect high reliability. Even consumer apps target 99.99% uptime in Japan.
MTBF, MTTR, and Failure Modes
  • MTBF (Mean Time Between Failures): Average time the system runs between failures. Higher is better.
  • MTTR (Mean Time To Recover): Average time to detect, diagnose, and fix a failure. Lower is better.
  • Availability formula: MTBF / (MTBF + MTTR). To improve availability, increase MTBF (make fewer mistakes) or decrease MTTR (fail faster, recover faster).
  • Failure modes:
    • Crash: Process stops entirely. Easy to detect (health check fails).
    • Hang: Process is alive but not responding. Harder to detect (process looks fine but isn't doing work).
    • Slow: Process responds but takes too long. Slow requests accumulate and exhaust resources.
    • Byzantine: Process returns incorrect data. Very hard to detect — needs data validation and cross-checking.
Replication
Duplicating data or services across multiple nodes for failover.
  • Active-Passive: Standby replica takes over when primary fails
  • Active-Active: All replicas serve traffic simultaneously
  • Replication factor: How many copies of your data exist
Retry & Exponential Backoff
Auto-reattempt failed requests with increasing delays.
  • First retry after 100ms, second after 200ms, third after 400ms...
  • Add jitter (randomness) to prevent thundering herd problem
  • Set max retries (3-5) and use circuit breaker for persistent failures
  • Example: AWS SDK uses exponential backoff by default

Rate Limiting Algorithms

Token Bucket Most Common
A bucket holds N tokens. Tokens refill at a fixed rate (e.g., 10/sec). Each request consumes a token.
  • Burst handling: Allows short bursts up to bucket size
  • Strict rate: Long-term rate equals refill rate
  • Use case: API rate limiting per user (e.g., 100 req/min with burst to 200)
  • Trade-off: Memory efficient but allows short bursts
Sliding Window Log / Counter
Two approaches for precise rate tracking:
  • Log: Store timestamps of each request. Count in rolling window. Memory-heavy.
  • Counter: Hybrid of fixed window + sliding log. Smooth edge-case bursts.
  • When to use: When you need precise, fair rate limiting (e.g., per-user 10 req/sec)
Leaky Bucket & Fixed Window
  • Leaky Bucket: Request queue processed at constant rate. Drops overflow. Good for shaping traffic.
  • Fixed Window: Count requests per clock minute. Simple but allows double burst at reset boundary (11:59:59 + 12:00:00).

Idempotency

Idempotency Keys Critical
Ensuring repeated requests produce the same result. Critical for payment, order creation, and any operation with retries.
  • Mechanism: Client sends Idempotency-Key header. Server deduplicates via DB unique constraint or cache with TTL.
  • Safe methods: GET, PUT, DELETE are naturally idempotent. POST is not.
  • Example: Stripe's Idempotency-Key header. Same key = same result, no duplicate charge.
  • Store key+response in Redis with a 24h TTL; on retry, return cached response
Interview Tip For payment or order systems, lead with idempotency. Say: "Every payment request includes an idempotency key so retries don't double-charge the user." This is a must-know for fintech interviews.

1.3 Performance

Measured by latency (response time) and throughput (requests/second). Optimize both.

Connection Pooling Essential
Reuse pre-established database connections instead of creating a new TCP connection per request.
  • Reduces latency (no TCP handshake, SSL negotiation, or authentication)
  • Limits concurrent DB connections under load
  • Pool sizing: (core_count × 2) + 1 is a good starting point
  • Tools: HikariCP (Java), PgBouncer (PostgreSQL), ProxySQL (MySQL)
Key Optimization Strategies
  • Caching: Reduce network round-trips by serving from RAM
  • Database indexing: Speed up reads, slightly slow writes
  • Data compression: Smaller payloads = faster transfer
  • Batch network calls: Combine requests to reduce overhead
  • gRPC: Binary protocol, smaller payloads, faster serialization
  • WebSockets: Persistent connection for real-time updates
Understanding Latency: p50, p95, p99, p999

Average latency hides problems. Percentiles reveal the real story:

  • p50 (median): Half of requests are faster than this. Your "typical" experience.
  • p95: 95% of requests are faster than this. 1 in 20 users experience worse. Often used for SLAs.
  • p99: 99% are faster. 1 in 100 users. This catches real issues.
  • p999: 99.9% are faster. 1 in 1000. The worst-of-the-worst.
  • Why percentiles matter: A system with 100ms average but 5000ms p99 is terrible for 1% of users. In a microservice architecture, a request may call 50 services. If each has 1% slow requests, a user has a ~40% chance of hitting at least one slow service (0.99^50 ≈ 0.60). This is the tail at scale problem.
  • Optimization: Focus on eliminating p99 outliers before optimizing p50. Common causes: GC pauses, network congestion, hot keys, cold caches.
Little's Law — Predicting Queue Size

A fundamental queueing theory formula: L = λ × W

  • L = Average number of requests in the system (concurrency)
  • λ (lambda) = Arrival rate (requests per second)
  • W = Average time a request spends in the system (latency)
  • Example: If 100 requests arrive per second (λ=100) and each takes 200ms (W=0.2), the system has L = 100 × 0.2 = 20 requests in flight at any moment.
  • Practical application: If your DB can handle 50 concurrent connections (L=50) and each query takes 100ms (W=0.1), your max throughput is λ = L/W = 50/0.1 = 500 QPS. To go higher, you either need more connections or faster queries.
  • Pool sizing: If your target is 1000 QPS and each query takes 50ms, you need L = 1000 × 0.05 = 50 connections in the pool (at minimum).
Amdahl's Law — Limits of Parallelism

Adding more servers doesn't always help. Speedup = 1 / ((1-P) + P/N)

  • P = Portion of the work that can be parallelized
  • N = Number of processors/servers
  • (1-P) = Sequential portion — this is the bottleneck
  • Example: If 80% of the work can be parallelized (P=0.8), the maximum speedup even with infinite servers is 1 / (0.2) = 5x. The 20% sequential portion always limits you.
  • Real-world application: Database transactions often have sequential portions (locking, serialization). No matter how many replicas you add, writes to the primary are still single-threaded for many operations. This is why sharding is needed — it's the only way to parallelize writes.
  • Connection pooling insight: If queries have a sequential dependency (transaction isolation), pooling won't help beyond a certain point. Amdahl's Law applies.
Interview Tip When asked about performance, always distinguish between latency and throughput. Say: "Our p50 is 50ms but p99 is 800ms — I'd investigate the tail latency before optimizing the median." Use Little's Law to calculate required pool size and max throughput. Back-of-envelope: 1M DAU ≈ 10–15 QPS.

1.4 CAP Theorem

A distributed data store can only provide 2 of 3 guarantees: Consistency, Availability, Partition Tolerance.

Why This Matters CAP is the first decision you make when choosing a database. Interviewers watch for your reasoning: "In a network partition, do you serve stale data or return an error?"

Consistency (C)
Every read receives the most recent write. All nodes see the same data at the same time.
  • Strong consistency: After a write completes, all readers see it
  • Eventual consistency: Reads may return stale data but will eventually converge
  • CP trade-off: If a node can't guarantee consistency, it refuses to respond
Availability (A)
Every non-failed node returns a response (success or error), no matter what.
  • AP trade-off: During a partition, nodes accept writes on both sides — data may diverge
  • Conflict resolution (last-write-wins, CRDTs) needed for eventual consistency
  • Example: DynamoDB AP mode — you always can write, but two users might see different data temporarily
Partition Tolerance (P)
The system continues to operate despite network failures (dropped packets, latency spikes).
  • In distributed systems, network partitions will happen — you must choose P
  • Real choice is CP vs AP, not CA (CA is impossible in a distributed system)
  • Partition: two nodes can't communicate, but each thinks the other is down
PACELC — CAP's Important Extension

CAP only tells you what happens during a partition. But what about when the network is working normally? PACELC answers that:

  • P (Partition): If a partition occurs, trade between Availability and Consistency (this is CAP).
  • E (Else): When the system is running normally (no partition), trade between Latency and Consistency.
  • Example — DynamoDB: In AP mode, during a partition it prefers availability. In normal operation, it uses eventually consistent reads (low latency, potentially stale) as default. Strongly consistent reads are slower (require quorum) but available as an opt-in.
  • Example — PostgreSQL: In CP mode, during a partition it rejects writes if consistency can't be guaranteed. In normal operation, it prefers strong consistency (higher latency, serializable transactions).
  • Interview value: Mentioning PACELC shows you understand that CAP is not the whole story — the trade-off between latency and consistency in normal operation is just as important.
Consistency Levels — Beyond CAP

Not all "consistency" is the same. Different systems offer different levels:

  • Strong Consistency: After a write, all subsequent reads see that write (linearizability). Highest correctness, highest latency. Example: PostgreSQL primary reads.
  • Eventual Consistency: If no new writes, all replicas will eventually converge. Reads may return stale data during convergence. Example: DNS propagation, DynamoDB default reads, CDN cache updates.
  • Read-After-Write Consistency: A user who writes data will always see their own write. Other users might see stale data. Example: Most social networks — you see your own post immediately, friends may see it after a slight delay.
  • Monotonic Reads: A user who reads once will never see older data in subsequent reads. Time moves forward, not backward in what you see. Prevents the "disappearing data" problem.
  • Casual Consistency: Related events are seen in the correct order. If Alice comments on Bob's post, no one sees the comment without also seeing Bob's post.
Real-World Analogy Like a restaurant menu: you can have fast, cheap, or good — pick two. CAP is the same: Consistency, Availability, Partition Tolerance — pick two (and P is mandatory).
Interview Tip Immediately establish that P is non-negotiable in distributed systems. Then reason: "If we choose CP, we sacrifice availability during partitions. For a payment system, that's acceptable. For a social feed, we choose AP for user experience." For extra depth, mention PACELC: "During normal operation, I'd trade consistency for lower latency using eventual consistency, with read-after-write guarantees for the user's own data."

Cache Types
  • Application Cache: In-memory within the service instance. Fastest but not shared across nodes.
  • Distributed Cache: Shared across nodes (Redis cluster, Memcached). Consistent but adds network hop.
  • CDN Cache: Static assets from geographically close edge servers. Best for images, JS, CSS.
Cache Invalidation
One of the hardest problems in computer science. Three main approaches:
  • TTL: Automatically expire data after a time window. Simple, possible stale data.
  • Write-Through: Update cache synchronously on write. Consistent but slower writes.
  • Write-Behind: Update cache immediately, DB asynchronously. Fast but risk of loss.
Cache Hit Ratio — Measuring Effectiveness

Cache hit ratio = cache hits / (cache hits + cache misses). A 90%+ hit ratio means the cache is effective; <90% suggests the cache is too small, TTL is too short, or the cache key design is wrong.

  • Cold start: When a new cache node comes online, hit ratio starts at 0%. It climbs as data is populated. Pre-warming (loading popular data into cache before going live) helps.
  • Working set: The set of data that's accessed frequently. If your working set fits in cache, you get high hit ratios. If it's larger than cache, you get cache churn — data evicted before it's re-read.
  • Improving hit ratio: Increase cache size, use smarter eviction (LRU over TTL-only), pre-warm, or redesign cache keys for better locality.
Eviction Policies — What Gets Removed

When the cache is full, something must go. Different policies for different patterns:

  • LRU (Least Recently Used): Remove the item that hasn't been accessed the longest. Best for most general-purpose workloads. Used by Redis's allkeys-lru, Memcached.
  • LFU (Least Frequently Used): Remove the item accessed the fewest times. Best when popularity matters (e.g., content recommendation). Prevents one-hit-wonders from polluting cache.
  • FIFO (First In, First Out): Remove the oldest item regardless of access. Simple but can evict hot data. Rarely used alone.
  • TTL-only: No explicit eviction — data expires based on time. Can combine with LRU (Redis: volatile-lru).
  • Redis offers 8 eviction policies. Choose based on whether you want to evict only TTL-set keys or any key.
Cache Stampede (Thundering Herd)

The problem: A popular cache key expires. 1000 concurrent requests all miss the cache and hit the database simultaneously. The database melts down.

  • Solution 1 — Early Recompute: Before the key expires (at 80% TTL), proactively recompute it in the background. A single worker does the work; other requests get the stale-but-not-expired value.
  • Solution 2 — Mutex/Locking: On a cache miss, the first request acquires a lock and recomputes. Other requests wait briefly or get a stale value. Only one DB query.
  • Solution 3 — Stale-While-Revalidate: Serve the stale value immediately but trigger a background refresh. Used by CDNs (Cloudflare) and Varnish.
  • Solution 4 — Jittered TTL: Add random variation to TTLs (e.g., 300s ± 30s). Prevents all keys from expiring at the same time.
CDN — Caching at the Edge

CDNs cache static (and sometimes dynamic) content at servers close to the user.

  • PoP (Points of Presence): CDN servers in hundreds of locations worldwide. A user in Tokyo gets content from a Tokyo PoP, not your origin server in us-east-1.
  • Origin Pull vs Push: Pull = CDN fetches from your origin on first miss (lazy, simpler). Push = you upload assets to CDN ahead of time (proactive, for predictable catalog).
  • Cache rules: Use Cache-Control headers (max-age, s-maxage, stale-while-revalidate). Set aggressive cache for images/fonts (1 year + versioned URLs), minimal for HTML/API responses.
  • Cache invalidation: Most CDNs support purging (instant removal) but it takes time to propagate to all edge nodes. Versioned URLs (bundle.v2.js) avoid invalidation entirely.
Interview Tip Name the cache strategy explicitly. Say: "I'd use Cache-Aside with a Redis cluster set to allkeys-lru eviction and a cache stampede protection using early recompute at 80% TTL." Mention CDN for static assets with versioned URLs to avoid invalidation problems. These details separate senior-level answers from generic ones.

1.6 Load Balancing

Distributes incoming traffic evenly across a pool of servers to maximize throughput and reliability.

Algorithms
  • Round Robin: Simple rotation — request 1 to server A, 2 to B, 3 to C. Works for uniform servers.
  • Least Connections: Route to server with fewest active connections. Best for varying request duration.
  • IP Hashing: Same client IP routes to same server (session persistence / sticky sessions).
L4 vs L7
  • L4 (Transport layer): Routes based on IP + TCP/UDP port. Faster, less intelligent. Ideal for simple TCP traffic.
  • L7 (Application layer): Routes based on HTTP headers, URL path, cookies. Slower but enables advanced routing.
  • Example: Route /api/users to user-service, /api/orders to order-service.
Health Checks — Keeping Dead Servers Out

A load balancer is only useful if it stops sending traffic to broken servers.

  • Passive health check: LB monitors responses. If a server returns 5xx errors or times out repeatedly, the LB marks it as unhealthy and stops routing to it. Good for catching unexpected failures.
  • Active health check: LB periodically pings a dedicated /health endpoint on each server (e.g., every 5 seconds). If the endpoint doesn't return 200, the server is removed from rotation. Proactive — catches issues before users hit them.
  • Graceful shutdown: When scaling down, the server tells the LB "I'm going away" (deregister), finishes in-flight requests, then shuts down. Prevents dropped connections during deploys.
DNS Load Balancing — The First Line of Defense

Before traffic even reaches your load balancer, DNS can distribute it:

  • Round-robin DNS: A single domain name resolves to multiple IP addresses. Different users get different IPs. Simple but no health checking — if one IP goes down, some users get errors.
  • Geographic routing: Users in Asia resolve to the Tokyo data center, users in US to us-east-1. Reduces latency. Used by CDNs.
  • Weighted routing: Route 90% of traffic to v2, 10% to v1 for canary deployments.
  • DNS caching (TTL): DNS responses are cached by ISPs and browsers. A low TTL (60 seconds) means changes propagate fast but increases DNS query volume. High TTL (300+) means slower failover.
  • Best practice: Use DNS to route between regions, and an internal LB within each region.
Session Persistence (Sticky Sessions)

Some apps store session data on the server (in-memory). If the next request goes to a different server, the session is lost.

  • Cookie-based: LB sets a cookie identifying the server. All subsequent requests with that cookie go to the same server. Simple but breaks if the server goes down.
  • IP hash: Consistent hashing of the client IP → same server every time (as long as the server pool doesn't change).
  • Better alternative: Externalize sessions. Store session data in Redis or a database. Any server can serve any request. This is the recommended approach — servers become stateless, sticky sessions are unnecessary.
Interview Tip Mention L4 for internal traffic and L7 for API gateway routing. Say: "I'd use an L7 load balancer at the edge for path-based routing and an L4 internally for TCP traffic to database replicas." Mention health checks and graceful shutdown. For sessions, advocate for stateless services with externalized session storage (Redis).

1.7 Databases

Relational (SQL) and Non-Relational (NoSQL) databases serve different needs. Choose based on schema, scale, and consistency requirements.

ACID Properties Explained Simply

ACID guarantees make SQL databases predictable even when things go wrong:

  • Atomicity: A transaction either completes fully or is rolled back entirely. No "half-charged, not delivered" scenarios. Like a bank transfer — money leaves one account and arrives in another, or neither happens.
  • Consistency: The database is always in a valid state. Constraints, triggers, and rules are enforced. You can't save an order without a user_id.
  • Isolation: Concurrent transactions don't interfere with each other. Two simultaneous purchases from the last item in stock — only one succeeds, the other gets "out of stock."
  • Durability: Once committed, data survives crashes, power outages, and reboots. Written to disk, not just memory.
  • In contrast, BASE (NoSQL) prioritizes availability: "database is always writable, but reads might be stale."
Storage Engines: How Data Is Physically Stored

The storage engine is the low-level component that actually writes bytes to disk. Two dominant types:

  • B-Tree (InnoDB, MySQL default): Data is stored in a balanced tree structure. Excellent for reads with range queries. Writes modify pages in place (random I/O). Better for read-heavy workloads, small-to-medium datasets.
  • LSM-Tree (LevelDB, RocksDB, Cassandra, ScyllaDB): Writes go to an in-memory buffer first, then flushed to immutable SSTables. Merged/compacted in the background. Excellent for write-heavy workloads. Reads need to check multiple levels (bloom filters help). Better for time-series, high-ingestion.
  • PostgreSQL uses a variant of B-Tree (no LSM option). MySQL supports both: InnoDB (B-Tree) and MyRocks (LSM-Tree via RocksDB).
  • Real-world: Use B-Tree for user profiles and orders. Use LSM-Tree for logs, analytics, time-series, and any high-write scenario.
ORM Pitfalls with Connection Pooling

ORMs (Sequelize, Prisma, Hibernate, Django ORM) abstract database access but introduce common issues:

  • N+1 query problem: ORM loads a list of posts, then fires one query per post to load comments. 1 + N queries instead of 1 JOIN. Use eager loading (.include() or .prefetch()).
  • Connection pool exhaustion: ORM libraries often hold connections longer than raw SQL. Each transaction occupies a connection. Long-running transactions + high concurrency = pool exhausted, app hangs.
  • Pool sizing: ORM pools need careful tuning. Common default (10 connections) might be too low for web apps. Monitor pool usage under load.
  • Auto-commit vs explicit transactions: ORM defaults to auto-commit unless wrapped in transaction. Unintended long transactions block other queries.
Interview Tip Don't just say "I'll use PostgreSQL." Explain why: "I need ACID for orders and complex JOINs for inventory reports. I'll put read-heavy data behind a cache layer and consider read replicas before sharding." For large-scale systems, consider hybrid: SQL for transactional, NoSQL for high-throughput analytics. Mention storage engines to show depth (B-Tree for reads, LSM-Tree for writes).

1.8 Sharding & Partitioning

Splitting data across multiple servers based on a key (e.g., UserID). Also called horizontal partitioning.

Hash-Based Common
hash(key) % N → assigns each key to a shard.
  • Even distribution across shards
  • Problem: Adding/removing a shard changes N, remapping nearly all keys
  • Solution: Consistent hashing (see 1.9)
Choosing a Shard Key Critical

The shard key is the most important decision in sharding. A bad shard key destroys performance:

  • High cardinality: The key should have many unique values (user_id is great, gender is terrible).
  • Even distribution: Each shard should get roughly equal data. user_id % N distributes evenly. Date-based sharding often creates hotspots (one shard gets all of January's data).
  • Supports the main query pattern: If most queries are "get orders for this user," shard by user_id. Avoid keys that force cross-shard queries.
  • Avoid monotonically increasing keys: Sharding by created_at or auto-increment ID means new data all goes to one shard (the "hot tail" problem).
  • Compound shard keys: Some DBs (Cassandra, DynamoDB) support multi-column keys. E.g., (customer_id, order_date) — partition by customer, sort by date.
The Hotspot Problem

Even with good sharding, some shards can get overwhelmed:

  • Celebrity/user peak: A influencer posts and millions of requests hit their user shard. That one shard melts while others are idle.
  • Time-based spikes: Black Friday creates 100x normal traffic. If sharded by product category, electronics shard gets hammered.
  • Solutions:
    • Use consistent hashing + virtual nodes to distribute load more evenly
    • Cache aggressively for hotspot keys (popular user's data cached in Redis)
    • Split the hot shard further (sub-sharding)
    • Read replicas for read-heavy hotspots
Rebalancing — Adding/Removing Shards

What happens when you need to add a shard because data is growing?

  • Offline rebalancing: Stop writes, move data, restart. Simple but causes downtime. Acceptable for scheduled maintenance.
  • Online rebalancing: Move data in the background while serving traffic. Complex but no downtime. Reads go to old or new shard; writes go to both during migration.
  • Consistent hashing advantage: Only ~1/N of keys move when adding a shard. With modulo hashing, nearly all keys would need to move.
  • Cost of rebalancing: Moving 1GB of data over a network takes time. For large datasets (TB+), rebalancing can take hours or days. Plan for it.
Range-Based
Divide key ranges across shards. E.g., users with IDs 0-1000 → shard 1, 1001-2000 → shard 2.
  • Supports range queries within a shard
  • Risk of hotspots if key distribution is uneven
  • Example: Date-based sharding (Jan → shard 1, Feb → shard 2)
Directory-Based
A lookup service maps each key to its shard.
  • Most flexible — can move data between shards dynamically
  • Single point of failure and performance bottleneck (the lookup service)
  • Used when data distribution policy changes frequently
Interview Tip When discussing sharding, immediately address the challenges: "JOINs across shards are expensive, cross-shard transactions need distributed coordination (Saga pattern), and rebalancing is painful. I'd use consistent hashing to minimize data movement when scaling."

A technique to minimize data movement when the number of nodes changes, used in distributed caches, databases, and load balancers.

Why This Matters Consistent hashing is the backbone of DynamoDB, Cassandra, Redis Cluster, and CDNs. Interviewers frequently ask: "How would you design a distributed cache?" — consistent hashing is the answer.

The Problem — Why Not Modulo?

Most people first think of hash(key) % N to distribute keys across N servers. It works great when N never changes. But when you add or remove a server, every key's mapping changes:

  • 3 servers: hash(X) % 3 = 2 → server 2
  • Add one: hash(X) % 4 = 1 → server 1 (different!)
  • Result: All keys need to be moved. With 1TB of cached data, that's 1TB of network traffic — slow and expensive.
  • Consistent hashing solves this: Adding a node only moves ~1/N of the keys (with 4 nodes, only ~25%).
Walk Through: How the Ring Works

Imagine a clock face instead of a line:

  • Take both your servers (nodes) and your data items (keys) and hash them to numbers on a circle (0 to 232 - 1).
  • Place them on the circle based on their hash.
  • Rule: Each key belongs to the next node you encounter going clockwise.
  • Example: Node A at position 10, Node B at 30, Node C at 60. Key X at position 25 → goes to Node B (the next clockwise node).
  • Add Node D at 45: Key X still goes to Node B (25 → 30). Key at position 35 would have gone to Node C but now goes to Node D. Only keys between 30-45 moved. Keys at 0-30 and 45-60 and 60+ are unaffected.
How It Works
  • Both nodes and keys are hashed onto a circular ring (0 to 2³² − 1)
  • Each key is assigned to the next clockwise node on the ring
  • Adding a node only affects keys in its immediate vicinity
  • Only ~1/N of keys need remapping (vs nearly all with modulo)
Virtual Nodes Important
Each physical node maps to multiple positions on the ring.
  • Improves load distribution (prevents hot spots from uneven hashing)
  • Handles heterogeneous hardware (assign more vnodes to powerful machines)
  • Example: 3 physical nodes × 100 vnodes = 300 positions on the ring
  • Used by: DynamoDB, Cassandra, Redis Cluster
Applications
  • Distributed caches: Memcached, Redis Cluster — minimize cache misses on scale changes
  • Databases: DynamoDB, Cassandra — partitioning + replication together
  • Load balancers: Consistent hashing for sticky sessions
Interview Tip When asked about distributed caching or database partitioning, lead with consistent hashing. Say: "I'd use consistent hashing with virtual nodes so scaling only moves ~1/N of data, and heterogeneous machines can have more weight." Mention the ring, virtual nodes, and the clockwise assignment.

Primary-Replica Standard
All writes go to the primary node. Replicas serve read traffic.
  • Simple, predictable consistency model
  • Scales read throughput linearly with replica count
  • Primary is single point of failure for writes (failover needed)
  • Replication lag can cause stale reads
  • Example: PostgreSQL streaming replication
Multi-Master
Any node accepts both reads and writes. Changes propagate between nodes.
  • Higher write availability (no single point of failure)
  • Complicated conflict resolution (last-write-wins, CRDTs)
  • Network latency between masters affects consistency
  • Example: DynamoDB Global Tables, MySQL Group Replication
Synchronous vs Asynchronous Replication

The trade-off between data safety and write speed:

  • Synchronous: Primary waits for all replicas to confirm the write before acknowledging the client. Guarantees no data loss if the primary fails. Slower writes (higher latency). Used for financial transactions, write-ahead logs.
  • Asynchronous: Primary acknowledges the client immediately. Replicas catch up in the background. Fast writes but data loss possible if the primary fails before replication completes. Used for most web apps (fast, acceptable trade-off).
  • Quorum replication: Wait for a "majority" (W + R > N). Example: 3 nodes, write waits for 2 acks, read queries 2 nodes. Balances safety and speed. Used by Cassandra, etcd, Consul.
Split-Brain Problem

What happens when two nodes both think they're the primary?

  • The scenario: A network partition isolates the primary from replicas. Replicas promote a new primary. When the partition heals, there are two primaries accepting writes. Data diverges.
  • Solutions:
    • Quorum: Only a node that holds the majority of votes can be primary (Paxos / Raft). The isolated node can't form a quorum and steps down.
    • Lease-based: Primary has a time-limited lease. If it can't renew the lease, it abdicates.
    • Fencing: A "fencing token" (monotonically increasing counter) — older tokens are rejected. The old primary's writes are invalid.
  • Split-brain is why you should never rely on just two nodes for a database. Use an odd number (3, 5) for reliable quorum.
Read-Your-Writes Consistency

A common problem with replica reads: A user writes data, then immediately reads from a replica that hasn't received the update yet. They see their own data as missing!

  • Solution 1: Read from primary for the user's own data. Writes → primary reads → primary for "my profile," replicas for "all profiles."
  • Solution 2: Read-after-write consistency with version stamps. The write returns a timestamp; subsequent reads only accept data >= that timestamp.
  • Solution 3: Sticky sessions — the same user always hits the same replica. Fragile but simple.
Interview Tip Start with Primary-Replica for simplicity. Say: "I'll use primary-replica replication — all writes go to the primary, replicas handle reads. Asynchronous replication for speed. If primary fails, promote a replica. For higher data safety, I'd use quorum-based replication (write to 2 of 3, read from 2 of 3)." Mention split-brain risk and read-your-writes consistency to show experience.

1.11 Indexing

Data structures that speed up data retrieval without scanning the entire table.

Why This Matters The most common performance bottleneck in databases is missing or wrong indexes. Interviewers ask: "Your query is slow — what do you check?" Indexes are always the first answer.
Composite Indexes Crucial
An index on multiple columns. Column order matters due to the leftmost prefix rule.
  • Example: INDEX(city, status, date)
  • Covers queries on: city, city+status, city+status+date
  • Does NOT cover: status alone, date alone, status+date
  • Put the most selective column first for best performance
Covering Indexes
An index that contains all columns needed by a query. The DB reads entirely from the index without touching the table.
  • Dramatically faster than non-covering indexes (no table lookup)
  • Known as "index-only scan" in PostgreSQL
  • Trade-off: More columns = larger index = slower writes
Index Selectivity & Partial Indexes
  • Selectivity: Ratio of unique values to total rows. High selectivity (e.g., email) = good filter. Low selectivity (e.g., is_active boolean) — DB may ignore index.
  • Partial Index: Index only a subset of rows matching a WHERE clause. Smaller, faster. Example: CREATE INDEX ON orders(status) WHERE status = 'pending';
  • Clustered Index: Data rows stored in index order. One per table (InnoDB primary key).
How B-Trees Work (Simply)

Think of a B-tree like a library's card catalog:

  • Internal nodes: Like the "A–D, E–H, I–L" signs on cabinets. They tell you which cabinet to look in, but don't have actual books.
  • Leaf nodes: The actual cards pointing to the shelf location. In a DB, leaf nodes contain the actual row location (or the row data for clustered indexes).
  • The tree is balanced: All leaf nodes are at the same depth. This means finding any row takes the same number of steps (O(log n)).
  • Fan-out: Each internal node can point to hundreds of children. A 4-level B-tree can index billions of rows because each level multiplies capacity.
Table Scan vs Index Scan — Cost Comparison

The DB chooses between scanning the whole table or using an index based on cost:

  • Table Scan (Sequential Scan): Read every row, check the condition. Fast when the table is small or when most rows match. Efficient for bulk operations.
  • Index Scan: Traverse the B-tree to find matching rows, then fetch each row from the table. Good when few rows match (high selectivity).
  • Index-Only Scan: All needed columns are in the index. No table fetch needed. The fastest option.
  • Rule of thumb: If more than ~5-10% of rows match, a table scan may be faster than an index scan (because random I/O is slower than sequential).
When Indexes Hurt Performance

Indexes aren't free. Each index adds cost:

  • Write slowdown: Every INSERT/UPDATE/DELETE must update every index on the table. A table with 5 indexes is ~5x slower to write to than an unindexed table.
  • Storage: Each index consumes disk space. A table's indexes can be larger than the table itself.
  • Memory: Indexes are cached in memory (buffer pool). Too many indexes = less memory for caching actual data.
  • Over-indexing anti-pattern: Adding an index on every column "just in case" kills write performance. Only add indexes for known query patterns.
Interview Tip When optimizing a slow query, always check: (1) Is the right index present? (2) Is it selective enough? (3) Is there a covering index possible? Show depth by discussing composite indexes and the leftmost prefix rule in detail. Use EXPLAIN/EXPLAIN ANALYZE to check if the DB is using your index.

Decouple services by sending messages between them asynchronously via a queue or stream.

Use Cases
  • Background tasks: Email sending, video encoding, report generation
  • Load buffering: Queue spikes in traffic — consumers process at their own pace
  • Event-driven architectures: Emit events (order_created, payment_received) → multiple services react
  • Decoupling: Producer doesn't need to know about consumers
Kafka vs RabbitMQ
  • Kafka: Event streaming platform. Persistent logs, replay capability, high throughput. Best for event sourcing, data pipelines, analytics.
  • RabbitMQ: Message broker. Smart queuing, routing, immediate delivery. Best for task queues, RPC-like patterns.
  • Redis Pub/Sub: Lightweight, in-memory. Best for real-time notifications, chat. No persistence.
  • SQS: AWS-managed, fully serverless. Best for simple worker queues.
Delivery Semantics

How many times will a message reach the consumer? This is a critical design decision:

  • At-most-once: Message sent once, no retry if it fails. Fast but data loss possible. Use for: non-critical analytics, sensor readings where fresh data replaces stale.
  • At-least-once: Message retried until acknowledged. No data loss but duplicates possible. Consumer must be idempotent. Use for: order processing, email sending. Most common choice.
  • Exactly-once: Deduplication built into the system (Kafka transactions + idempotent producer + transactional consumer). Highest complexity. Use for: financial transactions, inventory counts.
  • In practice, most systems use at-least-once + idempotent consumers. It's simpler and covers the same ground.
Dead Letter Queue & Backpressure
  • Dead Letter Queue (DLQ): When a message can't be processed after max retries, move it to a separate "dead letter" queue. A human or automated script can inspect and fix DLQ messages later. Prevents messages from getting stuck in loop.
  • Backpressure: When consumers can't keep up with producers. Strategies:
    • Rate limiting: Slow down or reject new producer requests
    • Buffer overflow: In Kafka, increase partition count or retention; in RabbitMQ, increase queue length limits
    • Drop messages: Accept that some messages will be lost (for non-critical data)
    • Scale consumers: Add more consumer instances to increase processing capacity
Interview Tip Introduce queues when you need reliability or decoupling. Say: "I'd put a message queue between the order service and email service. If email is slow, orders aren't blocked. If email fails, messages stay in the queue for retry." Choose Kafka for high-throughput event streaming, RabbitMQ for task queues. Always mention at-least-once delivery + idempotent consumers as the standard approach.

1.13 Observability & Monitoring

Understanding what's happening inside your system through Metrics, Logs, and Tracing (the three pillars).

Distributed Tracing in Practice
Debugging a slow request:
  1. Request hits API Gateway → TraceID is generated
  2. TraceID propagates via HTTP headers (x-trace-id) to each downstream service
  3. Each service emits spans: service name, operation, timestamps, parent span ID
  4. Jaeger collects spans and reconstructs the full request tree
  5. If auth-service span takes 800ms but db-query span takes 50ms → bottleneck is auth-service business logic (cache miss? external API call?)
  6. Correlate TraceID with structured logs for deeper debugging
Alerting Strategy
  • Latency alerts: p99 latency > 500ms for 5 minutes
  • Error rate alerts: 5xx errors > 1% for 5 minutes
  • Infrastructure alerts: CPU > 80%, disk space < 20%
  • Business alerts: Order success rate < 95%
  • Use SLIs (indicators), SLOs (targets, e.g., 99.9% uptime), and SLAs (contractual)
RED vs USE Method

Two complementary frameworks to decide what to measure:

  • RED Method (for services/requests): Rate (requests/sec), Errors (failed requests), Duration (latency). Request-centric. Best for user-facing services.
  • USE Method (for infrastructure/resources): Utilization (% busy), Saturation (queue length), Errors (failure count). Resource-centric. Best for servers, databases, network.
  • Together they give a complete picture: RED tells you if users are unhappy, USE tells you why (which resource is saturated).
Health Checks & Dashboards

Every service should expose endpoints for the infrastructure to check:

  • /healthz (liveness): Is the process alive? Returns 200 OK if running. Simple check (process start).
  • /readyz (readiness): Is the service ready to serve traffic? Returns 200 OK only after DB connection pool is warm, migrations are done, cache is populated.
  • Dashboards vs Alerts: Dashboards are for exploration (what happened?). Alerts are for notification (wake someone up). Dashboard your SLIs, alert on your SLOs. Too many alerts = alert fatigue.
Interview Tip Wrap up your design by mentioning observability. Say: "I'd add structured logging with TraceID propagation across all services, Prometheus for metrics using the RED method (Rate, Errors, Duration), and Jaeger for distributed tracing. /healthz and /readyz endpoints for every service. Alert on SLO violations, dashboard everything else."

Batch Processing

Process large chunks of data on a schedule. Think of it like doing your weekly laundry — you wait until you have a full load, then wash everything at once.

  • When it runs: Hourly, daily, or weekly (scheduled)
  • Latency: Minutes to hours — not for real-time decisions
  • Accuracy: High — you can process all data and correct mistakes
  • Common tools: Hadoop, Spark, Airflow
  • Example: Generate daily sales report at 2:00 AM processing all orders from yesterday
Stream Processing

Process events as they happen, in real-time. Like sorting mail as it arrives rather than waiting for the whole week's mail to pile up.

  • When it runs: Continuously — processes each event when it arrives
  • Latency: Milliseconds to seconds — real-time decisions
  • Accuracy: Approximate — trades precision for speed. Late-arriving data may be missed if the window closes
  • Common tools: Kafka, Flink, Spark Streaming
  • Example: Show real-time order count dashboard. Each order immediately increments the counter
  • Windowing: Define time buckets for aggregation — tumbling (fixed non-overlapping), sliding (overlapping window), session (group by activity)
Lambda vs Kappa Architecture

Two main approaches to building data pipelines:

  • Lambda Architecture: Runs batch and stream in parallel. Batch gives you correctness (reprocess everything), stream gives you speed. A merge service combines both views at query time. Example: Show last hour's revenue in real-time + historically accurate daily totals.
  • Kappa Architecture: Stream-only. Everything is a stream — even batch is just a "replay" of stream data. Simpler to operate (one pipeline instead of two). Relies on Kafka's ability to replay events from a point in time.
  • In practice, Kappa is gaining popularity. Lambda still used when historical reprocessing is very heavy (e.g., ML model training).
Kafka Fundamentals

Understanding Kafka is key to streaming systems:

  • Topic: A named stream of records (like a table in a DB)
  • Partition: A topic is split into partitions for parallel processing. Each partition is ordered. Messages within a partition have a sequential offset.
  • Offset: A unique ID for each message within a partition. Consumers track which offset they've read (like a bookmark).
  • Consumer Group: A set of consumers that divide partitions among themselves. One partition is consumed by exactly one consumer in a group.
  • Retention: Kafka persists messages even after consumption (configurable retention — e.g., 7 days). Enables replay.
  • At-least-once delivery: Kafka guarantees no data loss, but duplicates are possible if a consumer crashes after processing but before committing offset.
Interview Tip Mention data pipelines in systems that need analytics (e.g., "dashboard for business metrics"). Say: "I'd use Kafka for event ingestion, Flink for real-time processing, and Spark for daily batch aggregates. For simpler systems, Kappa Architecture avoids maintaining two separate pipelines." Mention partitioning and consumer groups to show depth.
