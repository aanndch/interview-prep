URL Shortener

Level 1 — foundational design · 45 min interview format

Your experience: You've built full-stack applications with NestJS and PostgreSQL. The URL shortener touches read-heavy caching (Redis), generating unique IDs, and REST API design — patterns you already use daily. This is the simplest design problem and a safe opener for any system design interview.

Quick Refresher

Scope: TinyURL-scale — 100M DAU, 1B shortenings/day, 100B redirects/day. Architecture: API gateway → URL service (stateless) → Redis cache (hot URLs) → PostgreSQL (source of truth). Short codes generated via base62 encoding of a distributed unique ID (snowflake-style).

Redirect: 301 (permanent) for production use — cached by browsers, reduces load. 302 (temporary) when you need analytics — every request hits the server. We use 302 first, upgrading to 301 after recording the click.

Caching: Hot URLs in Redis (LRU eviction, ~90% hit rate). 100B reads/day with 99% cache hit = 1B DB reads/day (~11K DB reads/sec). Two-level cache: local in-memory (L1) + Redis (L2).

Key trade-offs: Pre-generate code batches vs on-demand generation (batch is faster). Base62 vs MD5 hash (base62 avoids collisions, MD5 needs collision detection). Time-based partitions for analytics data. Write-behind cache for click event buffering.

Part 1 — Interview Core (45-min Walkthrough)

1. Requirements Gathering

Start by clarifying scope. The URL shortener looks simple but has enough depth to test your design thinking.

Functional Requirements

  • Shorten a URL: Given a long URL, return a short code (e.g., short.com/abc123).
  • Redirect: When a user visits the short URL, redirect to the original long URL.
  • Custom alias: Allow the user to specify their own alias (e.g., my-link).
  • Expiration: Links can have an optional TTL (default: never expire).
  • Analytics: Track click count, referrer, browser, location per short URL.

Non-Functional Requirements

  • High availability: The system must be up 99.99% — redirects are critical path.
  • Low latency: Redirect should complete in < 50ms (p99).
  • Durability: Once a URL is shortened, it should never be lost.
  • Read-heavy: Expect 100:1 read-to-write ratio (lots of redirects, fewer creations).
  • Scalability: Support 100M+ DAU, billions of redirects per day.
Interview tip: Say: "Let me confirm the scope. Are we building a simple internal tool or a public service like TinyURL? I'll assume public-facing with 100M DAU, 1B shortenings/day, 100B redirects/day. I'll focus on the core shortening and redirect — custom aliases and analytics are stretch goals I'll cover if we have time."

Components

API Gateway: Entry point for all requests. Handles rate limiting (10 req/s per IP for shortening), SSL termination, and request validation. Routes to URL Service instances.

URL Service (stateless): Core business logic. Handles shortening (generate code, store mapping) and redirect (lookup code, return URL). Runs multiple instances behind the gateway.

Redis Cache: Caches short_code → long_url mappings. TTL-based or LRU eviction. Absorbs the vast majority of redirect traffic.

PostgreSQL: Source of truth for URL mappings. Indexed on short_code for fast lookups. Separates write traffic from read replicas.

Kafka (analytics buffer): Click events are published asynchronously. The analytics processor consumes from Kafka and writes to the analytics DB. Decouples redirect path from analytics processing.

7. Key Generation Strategies

Generating unique, short codes is the most interesting design decision. Here are the options:

StrategyHow it worksProsCons
MD5 hash + truncationHash the long URL, take first 7 chars of base64Deterministic — same URL always gets same codeCollisions are possible (need retry + additional check)
Base62 of auto-increment IDUse DB sequence, convert to base62Simple, guaranteed unique, shortest codesSequential — predictable. Single point of contention.
Snowflake-style IDDistributed ID generator (timestamp + worker + seq)Distributed, no single point, good for shardingRequires clock sync, codes not shortest possible
Pre-generated batchesKey service generates batches of codes offlineSuper fast allocation, no contentionExtra moving part, codes are fixed-size

Recommended: Snowflake-ID → Base62

// Generate a unique 64-bit ID (snowflake-style) function generateId(workerId) { const timestamp = currentTimestamp() - EPOCH; return (timestamp << 22) // 41 bits — millisecond timestamp | (workerId << 12) // 10 bits — worker ID (1024 workers) | (sequence << 0); // 12 bits — sequence (4096 per ms) } // Encode to base62 (0-9, a-z, A-Z) = unique short code function toBase62(num) { const chars = '0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ'; let result = ''; while (num > 0) { result = chars[num % 62] + result; num = Math.floor(num / 62); } return result.padStart(7, '0'); // always 7 chars } // Result: "abc1234" — 7 chars, 62^7 ≈ 3.5 trillion combos
Interview tip: "I'd use a snowflake-style distributed ID generator encoded to base62. This gives me unique, non-sequential codes without a single point of contention. A 7-character code gives 3.5 trillion combinations — more than enough for our 1B URLs/year. For even shorter codes, I could go with 6 chars (56 billion combos) since we won't exhaust that in years."

8. Caching Layer

Caching is the most critical component — 1.15M reads/s cannot hit the database. The goal is 99%+ cache hit rate on redirects.

Two-Level Cache

// L1 — Local in-memory cache (each URL Service instance) // Small, fast, no network hop lru_cache = new Map(maxSize: 10_000, TTL: 60s) // L2 — Redis cluster (shared across all services) // Larger, slower (1-2ms), survives instance restarts redis.get("short:{code}") // Lookup flow: 1. Check L1 cache (local) → hit: ~0.1ms, return 2. Check L2 cache (Redis) → hit: ~1ms, populate L1, return 3. Query PostgreSQL → miss: ~5-10ms, populate L2 and L1, return

Cache Strategy

ParameterValueRationale
L1 size10K entries per instanceSmall enough to keep GC healthy. Covers the hottest URLs.
L1 TTL60 secondsShort enough to pick up mapping changes, long enough for cache benefit.
L2 total size100M entries (with LRU)Holds the most frequently accessed ~10% of all URLs.
L2 TTL7 days (sliding)URLs that are popular once tend to stay popular for a while.
Write-throughOn URL creation, populate cache immediatelyFirst redirect skips the DB read.
Cache invalidationOn URL update/delete, evict from both L1 and L2Consistency after user modifies their link.
Expected performance: "With a 90% L1 hit rate and 9% L2 hit rate, only 1% of requests hit the database. At 1.15M QPS, that's 11.5K DB reads/s — well within a single PostgreSQL replica's capacity."

10. Custom Aliases & Expiration

Custom Aliases

Users want short.com/black-friday-deal instead of short.com/abc123.

  • Validation: Alias must match ^[a-zA-Z0-9_-]{3,32}$. Check availability before creating.
  • Storage: Same short_code column — custom aliases have custom_alias = TRUE.
  • Lookup: Same path as generated codes — just another row in the same table.
  • Risk: Users squandering good aliases. Mitigation: aliases expire after 30 days of no clicks.

Expiration (TTL)

  • Default: never expires (or 1 year for anonymous users to reclaim storage).
  • Expired mappings: marked with expires_at < NOW(). Redirect returns 410 Gone.
  • Background job: scans for expired rows daily, archives to cold storage, reclaims short codes.
  • Short code reuse: possible after expiration + grace period (30 days to avoid confusing users).

11. Scaling & Trade-Offs

Single Server → Distributed

ScaleArchitectureLimitation
1M DAUSingle NestJS server + PostgreSQL + RedisSingle point of failure, RAM limits for cache
10M DAUMulti-instance URL Service + Redis cluster + DB read replicasWrite throughput on primary DB, cache warm-up on new instances
100M DAUAdd Kafka for analytics, partition URL mappings by short_code hash, CDN for redirectsCross-region latency, consistency for custom aliases

Key Bottlenecks & Solutions

Cache miss storms: If Redis goes down, all traffic hits the database. Solution: (1) Redis Sentinel/cluster for HA. (2) L1 cache absorbs initial burst. (3) Circuit breaker — if DB response time exceeds 100ms, serve stale cache entries.

ID generation contention: If using a DB sequence for IDs, every shorten request needs a sequence increment. Solution: pre-generate batches of IDs in a dedicated service.

CDN for redirects: Popular short URLs can be cached at the CDN level (CloudFront/Cloudflare). The CDN returns the 301 redirect without hitting our servers. This reduces server load by 80-90% for viral links.

Partitioning the analytics table: 20 TB/day of click events. Partition by date — each day is a separate partition. After 90 days, move to cold storage (S3 + Athena for queries).

Multi-Region Deployment

  • Active-active in US, EU, APAC. Each region has its own URL Service + Redis + PostgreSQL read replicas.
  • Write primary: One region (e.g., US) handles writes. Other regions replicate asynchronously.
  • Redirect: Each region serves from local cache/replica. 99% of redirects are served locally.
  • Consistency: Short codes are globally unique — no cross-region conflicts. Eventual consistency is fine for redirects (5-second delay is acceptable).
Your context: Your NestJS apps already use Redis for rate limiting and session caching, and RabbitMQ for async message processing. The URL shortener scales these same patterns — Redis becomes a two-level cache handling 1.15M QPS, and Kafka (or RabbitMQ) buffers analytics writes. The core architecture isn't new to you; it's your existing stack at 100x scale.

13. Security & Rate Limiting

Rate Limiting Strategy

// Token Bucket algorithm (Redis implementation) // Per user:10 tokens max, refill10 tokens/second function isAllowed(userId) { key = `ratelimit:${userId}` tokens = redis.get(key) if (tokens === null) { redis.set(key, 9, "EX", 1) // consume1, expire in1s return true } if (tokens > 0) { redis.decr(key) return true } return false // rate limited } // Limits: // - Authenticated:10 req/s per API key // - Anonymous:5 req/s per IP address // - Custom alias creation:10/hour per user

URL Validation & Abuse Prevention

  • Malicious URLs: Check against Google Safe Browsing API on creation. Block known phishing/malware domains.
  • URL length: Reject URLs > 2048 characters (HTTP limit). Return413 Payload Too Large.
  • Scheme validation: Only allow http:// and https://. Block javascript:, data:, file: URIs.
  • Squatting prevention: Custom aliases expire after30 days of no clicks. Block reserved words (admin, api, health).
Interview answer: "Rate limiting uses a token bucket in Redis —10 tokens max, refilling at10/s per authenticated user. For anonymous users, I'd use IP-based limiting at5/s. On the redirect path, I'd skip rate limiting since it's read-only and caching absorbs the load. The shorten endpoint is where I'd apply strict limits to prevent abuse."

Analytics Events0 (Kafka retention)1 hour (reprocess from Kafka)

Multi-Region Failover

  • Setup: Active-passive. US (primary) handles all writes. EU (secondary) has read replicas.
  • Replication: PostgreSQL streaming replication. Lag < 1 second typically.
  • Failover trigger: Manual. If US region is unreachable for > 5 minutes, promote EU to primary.
  • DNS swap: Update Route53/CloudFlare to point short.com to EU load balancer.
  • Cache rebuild: EU Redis cold-starts from PostgreSQL. First few minutes have higher DB load.

  • URL mappings: Dailypg_dump to S3. Retain30 days. Point-in-time recovery enabled.
  • Analytics: Hot (0-30 days) in PostgreSQL. Warm (30-90 days) in S3 Parquet. Cold (90+ days) in Glacier.
  • Redis: RDB snapshots every5 minutes to S3. AOF enabled for durability.
Interview answer: "For disaster recovery, I'd use active-passive multi-region with US as primary. RPO is5 minutes — we accept that5 minutes of URL creations could be lost in a catastrophic failure. D1. Base62 Encoding — How It Works

Base62 uses 62 characters: 0-9 (10), a-z (26), A-Z (26). Each character position represents a power of 62, just like decimal uses powers of 10.

// Convert decimal 1,234,567 to base62: 1,234,567 ÷ 62 = 19,912 remainder 23 → char[23] = 'n' 19,912 ÷ 62 = 321 remainder 10 → char[10] = 'a' 321 ÷ 62 = 5 remainder 11 → char[11] = 'b' 5 ÷ 62 = 0 remainder 5 → char[5] = '5' "5ban" = 1,234,567 in base62 // Character count math: 62^1 = 62 // 1 char: 62 codes 62^2 = 3,844 // 2 chars: 3.8K codes 62^3 = 238,328 // 3 chars: 238K codes 62^4 = ~14M // 4 chars: 14M codes 62^5 = ~916M // 5 chars: 916M codes 62^6 = ~56B // 6 chars: 56B codes 62^7 = ~3.5T // 7 chars: 3.5 trillion codes

Why 7 characters? 62^6 = 56 billion — enough for years at 1B URLs/year. 62^7 gives us headroom for a decade+ without increasing length.

D2. Snowflake-Style ID Generation

Snowflake IDs are 64-bit integers that encode time + worker + sequence. Originally developed by Twitter for distributed ID generation without a central coordinator.

// Snowflake bit layout (64 bits total): 0 - 0000000000 0000000000 0000000000 0000000000 0 - 0000000000 - 000000000000 | └───────────────────────── 41 bits ─────────────────────────┘ └─ 10 ─┘└─ 12 ─┘ sign timestamp (ms since custom epoch) worker sequence // Custom epoch: 2024-01-01T00:00:00Z // 41 bits → ~69 years before wrap-around // 10 bits → 1024 unique workers (servers/containers) // 12 bits → 4096 IDs per ms per worker // Max throughput per worker: 4096 × 1000 = 4.1M IDs/second // Total: 1024 × 4.1M = 4.2B IDs/second — overkill for 11.5K writes/s

Clock Skew Handling

  • NTP sync: All workers sync via NTP. If clock jumps backward, the worker waits until the previous timestamp passes.
  • Worker ID assignment: Via ZooKeeper or a simple DB table on startup. Manual config is fine for smaller deployments.
  • Sequence overflow: If sequence exceeds 4095 within the same ms, the worker waits for the next ms — extremely unlikely at our volume.

D3. Multi-Level Caching Strategy

Warm-Up

On a new URL Service instance deployment:

  • L1 cache is empty — first requests from this instance will miss L1 but hit L2 (Redis).
  • No special warm-up needed for URL shortener — Redis handles the majority of reads.
  • Over ~30 seconds, L1 cache fills with the most frequently accessed URLs.

Write-Behind vs Write-Through

OperationStrategyWhy
URL creationWrite-through: DB + Redis (synchronous)First redirect should be fast. Consistency matters — user expects the URL to work immediately.
URL updateWrite-through: DB + Redis (invalidate)User changed the destination. Must propagate immediately or redirect 404s.
Click eventWrite-behind: Kafka (fire-and-forget)Analytics can tolerate 5-second delay. No need to slow the redirect for it.

Cache Eviction

  • L1 (local): TTL-based eviction (60s). No explicit invalidation needed — stale entries expire naturally.
  • L2 (Redis): LRU eviction when memory limit is reached + explicit invalidation on URL update/delete.
  • Invalidation propagation: On URL update, publish a Redis pub/sub message to all URL Service instances to evict their L1 cache entry. Without this, L1 could serve stale data for up to 60 seconds.

D4. Analytics Pipeline at Scale

Kafka Topic Design

// Topic: "click_events" — 32 partitions (configurable) // Partition key: short_code (all clicks for same code go to same partition) // This ensures per-code ordering of events (not critical but nice) // Message value (Protocol Buffers or Avro): { short_code: string, referrer: string, user_agent: string, ip_address: string, timestamp: int64, metadata: map[string]string // extensible }

Consumer Design

  • Batch size: 1000 events or 5 seconds, whichever comes first.
  • Batch write to PostgreSQL: INSERT INTO click_events (short_code, ...) VALUES (...), (...), ... — single statement.
  • Exactly-once semantics: Kafka offsets committed only after successful DB batch write. Re-processing a batch is safe because click events are idempotent (append-only).
  • Rollup: After batch insert, update analytics_daily with INSERT ... ON CONFLICT DO UPDATE.

Storage Strategy

  • Hot data (last 30 days) in PostgreSQL (partitioned by day).
  • Warm data (30-90 days) on S3 as Parquet files, queryable via Athena.
  • Cold data (90+ days) in S3 Glacier — rarely queried, kept for compliance.
  • Materialized views for common dashboard queries: clicks by day, top URLs, top referrers.

D5. Partitioning Strategies

URL Mappings Table

At ~1 TB/year, the URL mappings table is small enough to live on a single well-provisioned PostgreSQL instance for years. Partitioning is optional but useful for:

  • By date range: url_mappings_2024, url_mappings_2025. Useful for archiving old mappings.
  • By hash of short_code: Spreads writes across physical shards. Each shard handles a subset of read/write traffic.

Click Events Table

At 20 TB/day, partitioning is mandatory. Best approach:

// Partition by day using PostgreSQL declarative partitioning: CREATE TABLE click_events ( id BIGSERIAL, short_code VARCHAR(10) NOT NULL, clicked_at TIMESTAMP NOT NULL, ... ) PARTITION BY RANGE (clicked_at); CREATE TABLE click_events_2024_07_16 PARTITION OF click_events FOR VALUES FROM ('2024-07-16') TO ('2024-07-17'); // Benefits: // - Queries with date filters scan only relevant partitions // - Old partitions can be detached and archived without downtime // - Each partition is a manageable size (~20 GB/day)

D6. Custom Alias Handling

Collision Scenarios

ScenarioHow We Handle It
Two users request same alias concurrentlyINSERT with unique constraint → one succeeds, the other gets 409 Conflict. Application-layer check is a SELECT ... FOR UPDATE skip locked for extra safety.
Alias matches a generated short codeGenerated codes use only [0-9a-z] (lowercase). Custom aliases accept [A-Z_-] too. Risk of collision is negligible but handled by the unique constraint.
User requests an offensive aliasCheck against a blocklist on the application layer. Store blocklist in Redis for fast lookups.
Alias is already taken and expiredGrace period of 30 days after expiration before the alias is released. Avoids confusing users who clicked an old link.

Rate Limiting for Custom Aliases

Custom aliases are more expensive than generated codes (availability check + potential contention). Apply stricter rate limits:

  • Authenticated users: 10 creations/hour with custom alias (vs 1000/hour with generated codes).
  • Anonymous: custom aliases not allowed (to prevent squatting).
