Structured Interview Blueprint — Comprehensive First-Principles Reference Guide

Why This Guide Exists

This handbook is built by synthesising the interview delivery execution of Hello Interview, the deep first-principles engineering concepts of the System Design Handbook, and the structured topical checklist of Roadmap.sh, customized with operational constraints for the Japan tech market.

Step 1: Clarifying Requirements

To demonstrate senior-level execution in a 45-minute loop, use the first 3 minutes to ask targeted queries rather than assuming system boundaries.

Dialogue: Scoping & Constraints

Candidate: "Before sketching, I will clarify the scale and functional boundaries of this system:

* Functional Scope: I assume the primary features are URL shortening (receiving a long URL and returning a unique short code) and HTTP redirection. Do we need custom aliases, TTL link expiration, and Open Graph previews for messenger links?

* Traffic Scale: What is the target volume? I'll assume 10 Million new URLs created per day. Under a 100:1 read-to-write ratio, that translates to 1 Billion daily redirects. Latencies at p99 must be under 50ms."

Functional Requirements

  • Shorten URL: Encodes a destination long link into a unique 7-character string.
  • Redirect: Resolves the short code to target destinations via HTTP redirects.
  • Optional Aliases: Supports user-specified strings (e.g. sho.rt/custom-promo).

Non-Functional Requirements

  • High Availability: 99.99% system uptime (no dead links).
  • Low Latency: Redirections completed in under 50ms (p99).
  • Active Disaster Recovery: Geo-distributed multi-region availability in Japan.

Step 2: Scale Estimations

Work through the calculations step-by-step out loud to demonstrate numerical rigor:

  • Writes QPS: $10,000,000 \text{ writes/day} \div 86,400 \text{ seconds} \approx 116 \text{ writes/sec average QPS}$. Under peak (2x), this is **~232 writes/sec**.
  • Reads QPS (Redirects): $1,000,000,000 \text{ reads/day} \div 86,400 \text{ seconds} \approx 11,574 \text{ redirects/sec average QPS}$. Under peak (2x), this is **~23,148 redirects/sec**.
  • URL Metadata Record Size: ~600 bytes.
    • `id` (Bigint / Snowflake ID) = 8 bytes
    • `short_code` (VARCHAR) = 10 bytes
    • `long_url` (TEXT / Target link) = 500 bytes average
    • `created_at`, `expires_at` (Timestamps) = 16 bytes
    • `metadata` = 66 bytes
  • Total Database Write Load (1 Year): $10^7 \text{ writes/day} \times 365 \text{ days} \times 600 \text{ bytes/record} \approx \mathbf{2.19\text{ TB/year}}$.
  • Memory Cache Allocation (80/20 Rule): We hold the top 20% of daily redirect traffic in memory. $$20\% \times 1,000,000,000 \text{ daily requests} = 2 \times 10^8 \text{ hot keys}$$ $$2 \times 10^8 \text{ keys} \times 600 \text{ bytes} \approx \mathbf{120\text{ GB RAM}}$$

Step 5: First-Principles Deep Dives

1. Key Generation Mechanics

To convert an auto-incrementing integer key into a short code, we use **Base62 Encoding**. Base62 maps integers to characters [0-9, a-z, A-Z], reducing key sizes significantly:

// Decimal to Base62 Conversion Division Proof
1,234,567 / 62 = 19,912 remainder 23 -> Char maps to 'n'
   19,912 / 62 =   321 remainder 10 -> Char maps to 'a'
      321 / 62 =     5 remainder 11 -> Char maps to 'b'
        5 / 62 =     0 remainder  5 -> Char maps to '5'
Read bottom-to-top: 5 -> b -> a -> n -> "5ban"

Snowflake ID generation: To avoid central SQL sequence locks, we generate 64-bit Snowflake IDs on web instances. The bit layout allocates 41 timestamp bits, 10 worker node bits, and 12 sequence bits, shifting them into a 64-bit integer: id = (timestamp << 22) | (workerId << 12) | sequence.

UUID v4 Index Fragmentation: Avoid using UUIDs as primary database keys. Because UUIDs are randomly distributed, inserting records causes random writes across index pages in a B-Tree. When a database page fills up, the engine must perform a **Page Split**, splitting the page and reorganizing disk blocks, which increases disk I/O and latency. Snowflake IDs, being chronologically ordered, insert sequentially at the end of the B-Tree, preventing splits.

2. Cache Stampede Prevention (XFetch)

Under heavy peak traffic, when a highly popular link expires from the cache, all concurrent threads will encounter a cache miss and query PostgreSQL read replicas in parallel. This saturates read connections and crashes the database. We protect the database using the probabilistic **XFetch algorithm**:

$$R \leftarrow \beta \cdot \delta \cdot \ln(\text{rand}())$$

$$if \quad (\text{TTL\_remaining} - R < 0)$$

If this conditional math yields true, a background worker thread asynchronously fetches the latest record from the database and updates Redis *before* the key actually expires, neutralizing DB read storms.

Step 6: Japan-Specific System Adaptations

1. Multi-Region Active-Active DR (Tokyo & Osaka)

Japan's high earthquake risk makes standard active-passive database failovers dangerous (due to data loss during recovery windows). We run an **Active-Active multi-region** database between Tokyo (`ap-northeast-1`) and Osaka (`ap-northeast-3`):

  • Tokyo nodes generate Snowflake keys in worker range `1-512`, and Osaka nodes generate in range `513-1024`, guaranteeing **zero primary key write conflicts**.
  • We replicate database rows asynchronously. Write conflicts on updates are resolved using **Last-Write-Wins (LWW)** rules alongside NTP time synchronization, or by maintaining multi-value CRDT structures to reconcile metadata conflicts.

2. Commuter Train Peaks (Tokyo Transit Surges)

Tokyo's rail network creates massive surges of users clicking links concurrently during train commutes (8:00–9:00 AM and 6:00–7:00 PM).

  • DB Thread Protection: Standard thread-per-connection database models suffer under high connection counts. We use **PgBouncer** connection poolers in transaction mode to multiplex thousands of active client connections into a tiny pool of actual database worker threads.
  • CDN Edge Offloading: Public links are cached on local CDN edge nodes (located in Tokyo and Osaka) to resolve redirects without touching the application server layer.

3. In-App Messaging Scraper Throttling (LINE OG Bot Protection)

When short links are shared on messaging platforms (e.g. LINE), background crawler bots immediately scan the destination page to render Open Graph (OG) title cards (generating card titles and preview images). This generates heavy transient read peaks on the shortener backend.

  • We implement a dedicated **Crawler Cache** in Redis. When User-Agent headers match messaging crawlers, we return cached Open Graph payloads immediately, blocking requests from hitting the application servers.

7. Production Operations

  • SLA Monitoring: Latency target of < 50ms at p99, and availability target of 99.99%.
  • Observability: Trace propagation headers (W3C standard) are passed down all request chains.
  • Storage Archival: Click logs are stored on hot SSDs for 30 days, compressed to Apache Parquet formats on Amazon S3 warm tiers, and archived to S3 Glacier after 90 days.
