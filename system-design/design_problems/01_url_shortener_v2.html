Part 1 — The Interview (~30 min + Q&A)

This is exactly what you present in a 45-minute interview. ~30 minutes of structured walkthrough, then 15-20 minutes of interviewer-driven follow-up questions. Each section has a "What to say" script and a "What the interviewer is thinking" note.

0. Mental Model — Think Before You Draw

Before touching the whiteboard, internalize three facts. These shape every decision:

Fact 1: It's read-heavy

For every URL you create, people will click it hundreds of times. The read:write ratio is 100:1 or higher. This means your caching layer is the most important component. A cache miss on the read path is a disaster; a slow write path is fine.

Fact 2: Reads are simple, writes are complex

A redirect is just: "give me the long URL for this short code." One key lookup. But creating a short URL requires: generate a unique code, validate the input, check for conflicts, store it, and optionally run analytics. Optimize for the simple path.

Fact 3: The code space is huge

A 7-character code using base62 (a-z, A-Z, 0-9) gives you 627 ≈ 3.5 trillion possible codes. At 1 billion URLs per year, you won't run out for 3,500 years. You don't need complex hash-collision handling. Just generate a unique ID and encode it.

Think like an engineer: Most interviewers aren't testing if you know TinyURL's architecture. They're testing if you can reason about trade-offs. When you say "I'd use a cache because this is read-heavy," that's reasoning. When you say "I'd use Redis," that's just naming a tool. Always lead with the why.

The Redirect Flow (301 vs 302)

When a user visits GET /Kx8mB2p, the request flows: CDN → LB → URL Service → L1 cache → L2 (Redis) → PostgreSQL. But which HTTP status code do you return?

301 Permanent302 Temporary
Browser behaviorCaches redirect — future visits skip our serverEvery visit hits our server
Server loadVery low (first request only)High (every request)
AnalyticsCan't track (browser doesn't call us)Can track every click
When to useAnalytics not needed, or URL is permanentYou need click tracking
Interview answer: "I'd use 302 for the first few requests to build an analytics baseline, then switch to 301 so browsers cache it. If analytics aren't needed, start with 301. The CDN can cache 301 redirects at the edge — for viral URLs, 80% of traffic never reaches our servers."
Key insight: The choice of 301 vs 302 is a perfect example of a trade-off. There's no "right" answer — it depends on whether analytics matter more than server load. State the trade-off, pick a side, and explain why.

4. Key Generation (5 min)

This is the most interesting design decision. How do you generate a unique 7-character code?

Option Comparison

ApproachHow it worksProsCons
MD5 hash + truncate Hash the long URL, take first 7 chars of base64 Deterministic (same URL → same code) Collisions possible. Need retry logic.
DB auto-increment → base62 Get next ID from DB sequence, convert to base62 Simple, guaranteed unique Sequential (predictable). DB is bottleneck.
Snowflake ID → base62 Distributed ID generator (timestamp + worker + seq) Distributed, no single point, time-sortable Requires clock sync.
Pre-generated key service Offline service generates batches of codes Ultra-fast allocation. No contention. Extra moving part.

Recommended: Snowflake → Base62

// Step 1: Generate a unique 64-bit ID function generateId(workerId) { const timestamp = Date.now() - EPOCH; // ms since 2024-01-01 return (timestamp << 22) // 41 bits → ~69 years | (workerId << 12) // 10 bits → 1024 workers | (sequence); // 12 bits → 4096/ms } // Step 2: Encode to base62 function toBase62(num) { const chars = '0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ'; let result = ''; while (num > 0) { result = chars[num % 62] + result; num = Math.floor(num / 62); } return result.padStart(7, '0'); // always 7 chars } // Result: "Kx8mB2p" — unique, non-sequential, 7 chars
Why not MD5? MD5 truncation gives you a 7-char code, but you must handle collisions. Snowflake IDs are unique by construction — no collision handling needed.

6. Caching Strategy (5 min)

Caching is the most critical component. Let's design the two-level cache properly.

Two-Level Cache Architecture

// Request flow for a redirect: 1. L1 (in-memory, per instance) → 0.1ms → 90% hit rate 2. L2 (Redis, shared cluster) → 1-2ms → 9% hit rate 3. PostgreSQL (read replica) → 5-10ms → 1% hit rate (cold URLs) // Weighted average latency: 0.90 × 0.1ms + 0.09 × 1.5ms + 0.01 × 7ms = 0.09 + 0.135 + 0.07 = 0.295ms average // At 1.15M QPS, only 1% hits the DB = 11.5K DB reads/sec // A single PostgreSQL replica handles ~100K reads/sec easily

Cache Parameters

ParameterL1 (In-Memory)L2 (Redis)
Size per instance10K entries (~6MB)100M entries (~60GB cluster)
EvictionLRULRU + TTL (7 days)
Populate onL2 hit or DB hitDB hit or URL creation
Invalidate onURL update/deleteURL update/delete
Survives restartNo (rebuilds from L2)Yes (persistent)

Write Strategy

// URL Creation (write-through): 1. Write to PostgreSQL (source of truth) 2. Write to Redis (L2 cache) 3. Return response // On URL update/delete (cache invalidation): 1. Update/delete in PostgreSQL 2. Delete from Redis (L2) 3. Broadcast invalidation to all instances (clear L1) → Use Redis pub/sub: each instance subscribes to "cache_invalidation" channel

7. Bottlenecks & Scaling (3 min)

Every system has a weakest link. Find it and fix it.

Scaling Progression

ScaleArchitectureNew Bottleneck
1K DAUSingle server + PostgreSQL + RedisSingle point of failure
10M DAUMulti-instance + Redis cluster + DB replicasWrite throughput on primary DB
100M DAUCDN + Kafka + time-partitioned analytics DBAnalytics storage (20 TB/day)
1B DAUSharded PostgreSQL + multi-regionCross-region consistency

Key Bottlenecks & Solutions

Cache miss storm — If Redis goes down, 1.15M QPS hits PostgreSQL directly. That's 100× what it can handle.

Fix: Redis Sentinel for automatic failover. L1 cache absorbs the first wave. Circuit breaker: if DB latency > 100ms, serve stale L1 entries.

ID generation contention — If using a DB auto-increment, every POST /shorten needs a sequence lock.

Fix: Snowflake IDs are generated locally on each instance (no coordination).

Hot URLs — A viral link (e.g., a tweet with 1M clicks) overwhelms a single DB shard.

Fix: CDN caching handles this naturally. A 301 redirect cached at the edge means 1M clicks = 0 server requests.

Interview answer: "The biggest risk is a cache miss storm. If Redis fails, we need three defenses: (1) Sentinel for fast failover, (2) L1 cache absorbs the initial spike, (3) circuit breaker protects the database."

A4. Load Balancing

L4 vs L7 — Which One?

Layer 4 (TCP)Layer 7 (HTTP)
Operates onIP address + portHTTP headers, URL path, cookies
SpeedFaster (no HTTP parsing)Slower (parses HTTP)
RoutingBy IP + port onlyBy URL path, header, cookie
SSL terminationNo (passes through)Yes (decrypts, offloads from backend)
URL shortener useNot idealBest choice

Why L7 for URL shortener: You need to route GET /{code} (redirect) differently from POST /api/v1/shorten (create). L7 can inspect the URL path and apply different rate limits.

Algorithm

Least connections — route to the server with fewest active connections. Better than round-robin because redirect requests are fast (~5ms) but POST /shorten is slower (~50ms).

Sticky sessions? Not needed. URL Service is stateless — any instance can handle any request.

A5. Concurrency & Race Conditions

Race Condition 1: Custom Alias Collision

// Two users try to claim "my-link" at the same time: Request A: POST /shorten { "long_url": "...", "custom_alias": "my-link" } Request B: POST /shorten { "long_url": "...", "custom_alias": "my-link" } // Without protection: A: SELECT WHERE short_code = 'my-link' → not found B: SELECT WHERE short_code = 'my-link' → not found A: INSERT short_code = 'my-link' → success B: INSERT short_code = 'my-link' → success (DUPLICATE! 💥) // Fix: Use the UNIQUE constraint as the safety net A: INSERT short_code = 'my-link' → success B: INSERT short_code = 'my-link' → UNIQUE violation → 409 Conflict

Race Condition 2: Cache Stale Read

// User updates their URL destination, but a redirect reads stale cache: 1. User A updates /abc123 → new-destination.com 2. Request B reads /abc123 from L1 cache → old-destination.com (stale!) // Fix: Write-through + cache invalidation 1. Update PostgreSQL (source of truth) 2. Delete from Redis (L2) 3. Broadcast L1 invalidation via Redis pub/sub 4. All instances clear their L1 entry for this key // Window of staleness: ~50-100ms — acceptable for URL shortener

Race Condition 3: Analytics Double-Count

// Kafka consumer crashes after DB write but before offset commit: 1. Consumer reads batch of 1000 click events 2. Writes to PostgreSQL successfully 3. Crashes before committing Kafka offset 4. Restarts, re-reads the same batch → double-counts 1000 clicks // Fix: Click events are idempotent (append-only) // Double-counting is acceptable for analytics — it's approximate anyway

Idempotency on Shorten Endpoint

// Problem: Client sends POST /shorten, connection times out. // Client doesn't know if it succeeded. Retries. Now two URLs exist. // Fix: Client-generated request ID (idempotency key) POST /api/v1/shorten Headers: Idempotency-Key: "req-uuid-123" Body: { "long_url": "..." } // Server side: 1. Check if idempotency-key exists in Redis 2. If exists → return cached response (deduplicated) 3. If not → process, store response in Redis with 24h TTL

A8. Disaster Recovery

RTO / RPO Targets

ComponentRPO (Data Loss)RTO (Recovery Time)
URL Mappings5 minutes (async replication)30 minutes (failover + DNS swap)
Analytics Events0 (Kafka retention)1 hour (reprocess from Kafka)
Redis Cache5 minutes (RDB snapshots)5 minutes (rebuild from DB)

Failure Scenarios

Single URL Service instance fails: Load balancer removes it from the pool. Clients retry and hit a healthy instance. L1 cache lost, but L2 (Redis) survives. Recovery: ~30 seconds.

Redis cluster fails: Sentinel promotes a replica. Traffic temporarily hits PostgreSQL (cache miss storm). L1 caches absorb the initial spike. Circuit breaker protects the DB. Recovery: ~10 seconds.

PostgreSQL primary fails: Promote read replica to primary. Update connection strings. RPO: ~5 min (async replication lag). RTO: ~30 min.

Full region failure: DNS failover to secondary region. Cold-start Redis from PostgreSQL. RPO: ~5 min. RTO: ~2-5 min.

Backup Strategy

  • URL mappings: Daily pg_dump to S3. Retain 30 days. Point-in-time recovery enabled.
  • Analytics: Kafka retains events for 7 days. Can reprocess everything if needed.
  • Redis: RDB snapshots every 5 minutes to S3. AOF enabled for durability.

D1. Base62 Encoding Explained

Base62 uses 62 characters: 0-9 (10), a-z (26), A-Z (26). It's like decimal (0-9) but with 62 symbols instead of 10.

// Convert decimal 1,234,567 to base62: 1,234,567 ÷ 62 = 19,912 remainder 23 → char[23] = 'n' 19,912 ÷ 62 = 321 remainder 10 → char[10] = 'a' 321 ÷ 62 = 5 remainder 11 → char[11] = 'b' 5 ÷ 62 = 0 remainder 5 → char[5] = '5' Read remainders bottom-to-top: "5ban" = 1,234,567 in base62

Capacity Math

Code LengthCombinationsEnough for
5 chars625 ≈ 916MLess than 1 year at 1B URLs/year
6 chars626 ≈ 56B56 years at 1B URLs/year
7 chars627 ≈ 3.5T3,500 years at 1B URLs/year

D2. Distributed ID Generation (Snowflake)

Snowflake IDs are 64-bit integers that encode time + worker + sequence. No central coordinator needed.

// Bit layout (64 bits total): 0 - [41 bits timestamp] - [10 bits worker] - [12 bits sequence] | | | | sign ms since epoch worker ID counter (~69 years) (1024 workers) (4096/ms) // Max throughput per worker: 4096 IDs/ms = 4.1M IDs/sec // Total cluster: 1024 workers × 4.1M = 4.2B IDs/sec // We need 11.5K writes/sec — overkill by 365,000×

Clock Skew Handling

If a worker's clock jumps backward (NTP sync), the worker detects the anomaly and waits until the previous timestamp catches up. This prevents ID collisions from time reversal.

D3. Two-Level Caching In Depth

Cache Stampede Prevention

// Problem: Hot URL expires from L1 simultaneously on all instances // 1000 instances all miss L1 → all hit Redis → Redis overloaded // Solution 1: Mutex lock (one instance rebuilds, others wait) if (l1.has(key)) return l1.get(key); if (mutex.tryLock(`rebuild:${key}`)) { const value = await redis.get(key); l1.set(key, value, TTL); mutex.unlock(`rebuild:${key}`); } else { await sleep(50); // wait for the rebuild return l1.get(key); // try again } // Solution 2: Probabilistic early expiration if (TTL_remaining / TTL_total < Math.random()) { refresh(key); }

L1 Invalidation Across Instances

// On URL update: 1. Update PostgreSQL 2. Delete from Redis (L2) 3. Publish: PUBLISH cache_invalidation "short:Kx8mB2p" // Each URL Service instance subscribes: SUBSCRIBE cache_invalidation on message: l1.delete(message.key)

D4. Analytics Pipeline Architecture

Kafka Topic Design

// Topic: "click_events" — 32 partitions // Partition key: short_code (all clicks for same code go to same partition) // Message value (Avro or Protobuf): { short_code: string, referrer: string, user_agent: string, ip_address: string, timestamp: int64 }

  • Exactly-once: Kafka offsets committed only after successful DB write. Re-processing is safe (append-only).
  • Rollup updates: After batch insert, update analytics_daily with INSERT ... ON CONFLICT DO UPDATE SET click_count = click_count + N.

D5. Database Sharding at Scale

When to Shard

PostgreSQL on a single node handles ~11K writes/sec and ~100K reads/sec. At 100M DAU, writes (11.5K/sec) are at the limit. If you grow beyond that, you need sharding.

Sharding Strategy for URL Mappings

// Hash-based sharding on short_code shard = hash(short_code) % NUM_SHARDS // Example with 4 shards: hash("Kx8mB2p") % 4 = 2 → Shard 2 hash("abc1234") % 4 = 0 → Shard 0 // Each shard is a separate PostgreSQL instance with 1-2 read replicas

Why hash-based, not range-based? URL codes are uniformly distributed (snowflake IDs are random-looking). Hash sharding gives even distribution.

Rate Limiting

// Token Bucket in Redis (Lua script for atomicity) local key = KEYS[1] // "ratelimit:user:123" local limit = tonumber(ARGV[1]) // max tokens local refill = tonumber(ARGV[2]) // tokens per second local tokens = redis.call("GET", key) or limit if tonumber(tokens) > 0 then redis.call("DECR", key) return 1 // allowed else return 0 // rate limited end // Limits: // - POST /shorten: 10 req/s per authenticated user // - POST /shorten: 5 req/s per anonymous IP // - Custom alias: 10/hour per user

URL Validation

  • Malicious URLs: Check against Google Safe Browsing API on creation.
  • URL length: Reject URLs > 2048 characters (HTTP limit).
  • Scheme: Only allow http:// and https://. Block javascript:, data:, file:.
  • Squatting: Custom aliases expire after 30 days of no clicks. Block reserved words (admin, api, health).
Interview Scripts & Follow-Up Questions

How to Open the Problem

Opening (first 2 minutes)

"I'll design a URL shortening service like TinyURL. Let me confirm the requirements:

Functional: Shorten URLs, redirect to original, support custom aliases, optional expiration, click analytics.

Non-functional: Highly available (redirects must work), low latency (<50ms p99), read-heavy (100:1 read:write), 100M DAU, 1B shortenings/day, 100B redirects/day.

Scale: 1.15M redirects/sec average. We need a strong caching layer.

Architecture: API gateway for rate limiting, stateless URL service, Redis for caching, PostgreSQL for persistence, Kafka for async analytics."

Transition to Deep Dive (after high-level design)

"The core flow is straightforward — the interesting questions are around caching and key generation. Want me to dive into the caching strategy first, or the key generation approach?"

Common Follow-Up Questions

QuestionHow to Answer
"What happens if Redis goes down?""Three defenses: (1) Redis Sentinel for automatic failover in ~10s. (2) L1 cache absorbs the initial burst. (3) Circuit breaker: if DB latency > 100ms, serve stale L1 entries. Better a slightly stale redirect than a 503."
"How do you handle duplicate short codes?""Generated codes via snowflake ID are unique by construction. No collision possible. For custom aliases, the UNIQUE constraint is the safety net."
"Why not just use a hash of the long URL?""MD5 truncation: (1) collisions require retry logic, (2) same URL from different users gets same code (privacy issue), (3) MD5 is 32 hex chars — truncating to 7 increases collision probability. Snowflake IDs avoid all three."
"How do you update a URL's destination?""PUT /{short_code}. On update: write to DB, delete from Redis (L2), broadcast L1 invalidation. Short code stays the same."
"How would you support a viral link (1M clicks/min)?""CDN caching handles this naturally. A 301 redirect cached at the edge means 1M clicks = 0 requests to our servers."
"What about GDPR / privacy?""Click analytics store IP hashes, not raw IPs. Users can request deletion. Analytics data older than 90 days is purged."
"How do you monitor this system?""Three pillars: metrics (Prometheus — latency, error rate, cache hit), logs (structured JSON), traces (distributed tracing for write path). SLI is redirect latency p99. SLO is <50ms."
"What if the primary database goes down?""Promote the read replica to primary (RPO ~5min from async replication lag). Update connection strings via service discovery. DNS failover to secondary region for regional outages."
Japan-specific tip: Japanese interviewers often ask about disaster recovery. "If the primary database region goes down, I'd promote the read replica to primary (RPO ~5min, RTO ~30min). DNS failover to the secondary region handles regional outages."
