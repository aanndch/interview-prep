Real-Time Chat / Messaging System

Level 1 — directly matches your resume · 45 min interview format

Your experience: You built a P2P texting system serving 70%+ of clients with 1M+ weekly messages using NestJS, RabbitMQ, and WebSockets. This problem scales that to WhatsApp-level (hundreds of millions of users). Every concept here connects to code you've already written.

Quick Refresher

Scope: WhatsApp-scale — 100M DAU, 5B msg/day, 58K msg/s avg (116K peak), 5TB storage/day. Architecture: WebSocket pool behind L7 LB (sticky sessions) → Chat servers → RabbitMQ for async routing + PostgreSQL (partitioned by time) for persistence + Redis for presence/sessions.

Delivery: At-least-once via RabbitMQ acks + idempotency via client_message_id (unique constraint + LRU dedup cache) = effectively exactly-once. Message ordering via single RabbitMQ queue per conversation (FIFO) + client-side sequence numbers.

Offline: Message persisted to DB, push notification sent (FCM/APNS). On reconnect, sync from last_read_at. Groups: Fan-out on write for ≤256; shared stream for larger.

Key trade-offs: RabbitMQ over Kafka (per-message ack suits chat better than raw throughput), time-based partitioning (easy archiving), read receipts as soft state (lost ack is not a data issue). E2EE via Signal Protocol — server is a dumb relay.

1. Requirements Gathering

Start every system design interview by clarifying what you're building. Don't assume — ask.

Functional Requirements

  • One-on-one chat: User A sends a message to User B. B receives it in real-time.
  • Group chat: Messages delivered to all members of a group (up to 256 participants).
  • Online/offline delivery: If recipient is offline, store message and deliver when they reconnect.
  • Read receipts: Sender sees when their message was delivered and read.
  • Typing indicators: Show when the other person is typing.
  • Message history: Users can scroll back through conversation history.
  • Multi-device: Same user on phone, desktop, and web — all receive messages.

Non-Functional Requirements

  • Low latency: End-to-end delivery under 200ms for 99th percentile.
  • High availability: System should never lose a message (at-least-once delivery).
  • Ordering: Messages within a conversation must appear in sending order.
  • Scalability: Support 100M+ DAU, billions of messages/day.
  • Durability: Messages survive server failures. Persist to disk.
Interview tip: Say: "Before I design, let me confirm the scope. Are we building WhatsApp-scale (100M+ DAU) or a smaller team chat? For this interview, I'll assume WhatsApp-scale with 100M DAU, 10B messages/day. I'll focus on one-on-one and small group chats — we can skip the broadcast/channel use case for now unless you want to discuss it." This shows you're thinking about scope.

Components

Chat Server (stateless WebSocket handler): Maintains persistent WebSocket connections with clients. Routes messages to the correct recipient via an internal lookup. Multiple instances behind a load balancer with sticky sessions.

Gateway / Load Balancer: Terminates WebSocket connections at L4 or L7. Uses IP hash (or a session cookie) for sticky sessions — ensures a reconnecting client reaches the same server that has their state.

Redis: Stores online presence (user_id → server_id mapping), session data, and typing indicator state. Fast lookups (< 1ms) are critical for routing.

RabbitMQ: Acts as a message buffer. Producers (chat servers) publish messages to a queue. Consumers (the same or other chat servers) consume and deliver to the recipient's WebSocket connection.

PostgreSQL (partitioned): The source of truth for all messages and conversation metadata. Queried for message history and offline message retrieval.

Media Service (separate): For image/video/file uploads. Not in scope for this design, but mention it exists.

4. Database Schema

Core Tables

-- Users (straightforward)
CREATE TABLE users (
    id          UUID PRIMARY KEY,
    username    VARCHAR(50) UNIQUE NOT NULL,
    display_name VARCHAR(100),
    password_hash VARCHAR(255),
    created_at  TIMESTAMP DEFAULT NOW()
);

-- Conversations (group or 1-on-1)
CREATE TABLE conversations (
    id          UUID PRIMARY KEY,
    type        SMALLINT  -- 1 = direct, 2 = group
    name        VARCHAR(100),        -- null for direct messages
    created_at  TIMESTAMP DEFAULT NOW()
);

-- Conversation participants
CREATE TABLE conversation_participants (
    conversation_id UUID REFERENCES conversations(id),
    user_id         UUID REFERENCES users(id),
    joined_at       TIMESTAMP DEFAULT NOW(),
    last_read_at    TIMESTAMP,          -- for read receipts
    PRIMARY KEY (conversation_id, user_id)
);

-- Messages (source of truth)
CREATE TABLE messages (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id UUID NOT NULL REFERENCES conversations(id),
    sender_id       UUID NOT NULL REFERENCES users(id),
    content         TEXT NOT NULL,
    client_message_id VARCHAR(64) NOT NULL,  -- for idempotency
    created_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Index for fast conversation history queries
CREATE INDEX idx_messages_conv_created
    ON messages (conversation_id, created_at DESC);

-- Idempotency index
CREATE UNIQUE INDEX idx_messages_client_id
    ON messages (sender_id, client_message_id);

Why This Schema Works

Design ChoiceWhy
UUID primary keysClient can generate message IDs without talking to the server. Avoids auto-increment contention. Works offline.
client_message_idDeduplication. If client sends the same message twice (e.g., network hiccup), the unique constraint prevents duplicates.
last_read_at on participantsSingle timestamp per user per conversation is enough to compute unread counts and read receipts.
Denormalized created_at indexScroll-back queries (SELECT ... WHERE conversation_id = X ORDER BY created_at DESC LIMIT 50) are covered by this index.
Your context: Your NestJS + TypeORM setup would map these cleanly — @Entity() for each table, @Index() on the composite.

6. WebSocket Protocol & Connection Management

Connection Lifecycle

// 1. Client authenticates via REST (gets a JWT) POST /auth/login → { token: "jwt..." } // 2. Client opens WebSocket with JWT in query params ws://chat.example.com/ws?token=jwt... // 3. Server validates JWT, registers connection redis.set("presence:user:{user_id}", "server:{server_id}", TTL=60s) redis.sadd("user_sessions:{user_id}", "session:{socket_id}") // 4. Heartbeat — every 30s to detect disconnection Client → { type: "ping" } Server → { type: "pong" } // 5. On disconnect: remove from presence, queue offline messages redis.del("presence:user:{user_id}") // Messages queued in RabbitMQ for this user will be stored on delivery failure

Sticky Sessions — Why They Matter

WebSocket connections are stateful. Once a client connects to chat server A, it must keep talking to A — you can't route to a different server mid-session. Load balancers handle this via:

  • IP hash: Same IP always hits the same server. Simple but breaks if users share an IP (NAT).
  • Cookie-based: LB sets a cookie on the first HTTP upgrade response; subsequent requests include it.
Your context: Your NestJS WebSocket gateways use @WebSocketGateway() with a Redis adapter for horizontal scaling. The Redis adapter uses pub/sub to broadcast messages across servers — so even if client A is on server 1 and client B is on server 2, they can communicate. In this design, RabbitMQ handles that cross-server routing instead.

7. Message Ordering & Delivery Guarantees

The Ordering Problem

Messages within a conversation must appear in the order the sender intended. In a distributed system with multiple chat servers, this is surprisingly hard:

  • User A sends message M1 from phone, then M2 from desktop — M2 might arrive at the server first.
  • Two users in a group chat send messages simultaneously — whose appears first?

Solution: Client-Side Sequence Numbers + Server-Side Ordering

// Client tracks a sequence number per conversation { type: "send_message", payload: { conversation_id: "conv-123", content: "Hello", client_message_id: "uuid", sequence_number: 42 // monotonic per conversation }} // Server side: use the sequence number for display order // But use server-created_at for chronological order // For group chat: one RabbitMQ queue per conversation // All messages for conv-123 go to the same queue consumer // Single consumer preserves FIFO order

Key insight: There are two different "orders". The display order (what the user sees) uses the created_at timestamp. The delivery order (which message gets processed first) is handled by routing all messages in a conversation through a single RabbitMQ queue with a single consumer — this preserves FIFO within that queue.

Delivery Guarantees

GuaranteeHow We Achieve It
At-least-onceRabbitMQ publisher confirms + consumer acknowledgments. If consumer crashes before sending ack, message is redelivered.
No duplicatesclient_message_id unique constraint in PostgreSQL + in-memory dedup set (LRU cache of recent IDs). If same ID received twice, second is silently dropped.
In-order deliverySingle queue per conversation. One consumer processes messages sequentially. Out-of-order from multi-device is resolved by client-side sequence numbers.
Delivery receiptsWhen server delivers to recipient's WebSocket, it publishes a message_ack event back to the sender's queue.

8. Offline Messages & Store-and-Forward

How It Works

// 1. Sender sends message Client A → Chat Server → RabbitMQ → Consumer // 2. Consumer checks presence recipient = redis.get("presence:user:{user_id}") if recipient == null: // Recipient offline — message stays in DB // Notification push (APNS/FCM) is triggered triggerPushNotification(user_id, message_preview) else: // Route to the chat server holding their WebSocket server_id = parseServerId(recipient) publishToServer(server_id, message)

Offline Message Retrieval

When a user comes online:

  1. They authenticate and open a WebSocket.
  2. The server checks last_read_at for each conversation.
  3. It queries SELECT * FROM messages WHERE conversation_id = X AND created_at > last_read_at ORDER BY created_at ASC LIMIT 50.
  4. Pending messages are streamed over the WebSocket.
  5. The client inserts them into the local store and renders.
Interview tip: "For the offline case, I'd push a notification via FCM/APNS with a message preview and a conversation_id deep link. When the user opens the app, we sync from last_read_at. For large backlogs (user was offline for days), we'd batch the sync in pages of 50 to avoid overwhelming the client."

10. Read Receipts & Typing Indicators

Read Receipts

Three states: Sent → Delivered → Read.

// Sent → Delivered: server acks when message persists to DB Server → Client A: { type: "message_ack", client_message_id: "...", status: "delivered" } // Delivered → Read: client sends mark_read event Client B → Server: { type: "mark_read", conversation_id: "conv-123", up_to: "2024-06-15T10:30:00Z" } // Server updates last_read_at and notifies sender UPDATE conversation_participants SET last_read_at = "..." WHERE conversation_id = "conv-123" AND user_id = "user-B"; Server → Client A: { type: "message_ack", client_message_id: "...", status: "read" }
Performance note: Read receipts are soft state. We don't need strong consistency here. If a read receipt is lost, the sender sees "delivered" instead of "read" — not a data integrity issue. This is a good place to use optimistic updates and eventually consistent delivery.

Typing Indicators

Typing indicators are ephemeral — they're not stored in the database. They're delivered directly via WebSocket with a short-lived state in Redis:

// Client sends typing_start (debounced — max once per 3 seconds)
{ type: "typing_start", payload: { conversation_id: "conv-123" }}

// Server sets a short TTL in Redis
redis.set("typing:{conversation_id}:{user_id}", "true", "EX", 5)

// Server broadcasts to all OTHER conversation participants
Server → OtherClients: { type: "typing", payload: { conversation_id, user_id, is_typing: true }}

// Client sends typing_stop (on send, or after 3s of inactivity)
{ type: "typing_stop", payload: { conversation_id: "conv-123" }}

// If TTL expires without a stop, the indicator disappears automatically
// This handles the case where the client disconnects while typing

11. End-to-End Encryption Overview

This is a common follow-up question. You don't need to design the full crypto, but you should know the Signal Protocol concepts:

How It Works (High Level)

  • Public key registration: Each device generates a key pair (public + private). The public key is registered with the server. Private key never leaves the device.
  • Pre-keys: Devices upload a batch of "pre-keys" (signed + one-time) to the server. When Alice wants to send Bob a message, she fetches one of Bob's pre-keys to establish a session.
  • Ratchet: Each message generates a new key using a Diffie-Hellman ratchet. Even if a key is compromised, previous and future messages are protected (forward secrecy).
  • The server is a dumb relay: It stores and forwards encrypted blobs. It cannot read the content.
Interview answer: "For E2EE, I'd use the Signal Protocol — it's the gold standard used by WhatsApp and Signal. The server never sees plaintext. Each message is encrypted with a per-message key from a Diffie-Hellman ratchet. The server's role is just to store and forward encrypted payloads. Key management is the hard part — I'd integrate with an existing library like libsignal rather than implementing the crypto myself."

12. Scaling & Trade-Offs

Single Server → Distributed

ScaleArchitectureLimitation
1K concurrent usersSingle NestJS server + PostgreSQL + in-memory stateSingle point of failure, RAM limits
100K concurrentMultiple chat servers + Redis (presence) + RabbitMQ (message queue)RabbitMQ throughput, DB write bottleneck
10M concurrentAdd message partitioning, read replicas, CDN for mediaCross-region latency, consistency challenges
100M+ DAU (WhatsApp)Custom Erlang stack, per-conversation sharding, in-house message storeOperational complexity, infrastructure cost

Key Bottlenecks & Solutions

WebSocket connections per server: A single machine can handle ~1M concurrent connections (with epoll/kqueue and enough RAM). At 100M DAU with 30% concurrency = 30M connections, we need ~30-50 chat servers.

Database writes: 116K writes/sec at peak. A single PostgreSQL instance handles ~5-10K writes/sec. Solution: partition messages by conversation_id hash across multiple DB instances.

RabbitMQ throughput: A well-tuned RabbitMQ cluster handles ~50K msg/s. At 116K peak, we'd need multiple queues or use Kafka for higher throughput.

Same-conversation ordering: All messages in one conversation must go through the same queue consumer. This means a single consumer thread per busy conversation. If a conversation has very high throughput, it becomes a bottleneck. Mitigation: batch writes, use server-side timestamps for ordering.

+ RabbitMQ vs Kafka — which is better for chat at scale? → See Deep Dive D6

Multi-Region Deployment

For a global service (WhatsApp serves 180+ countries):

  • Regional clusters: Deploy chat servers in each region (US, EU, APAC). Users connect to the nearest region.
  • Cross-region messages: If user A in US sends to user B in EU, the message is queued in the US cluster, replicated to EU via a cross-region RabbitMQ bridge, and delivered by the EU cluster.
  • Database: Each region has its own DB. User data is partitioned by home region. Cross-region reads use async replication.
  • Consistency: Messages are local to the sender's region. The recipient reads from their region's replica (eventually consistent cross-region).

14. Security & Rate Limiting

Rate Limiting Strategy

// Token Bucket (Redis implementation) // Per user:30 msg/min, burst10 msg/s function isAllowed(userId, action) { key = `ratelimit:${userId}:${action}` tokens = redis.get(key) if (tokens === null) { redis.set(key, 29, "EX", 60) // consume1, expire in60s return true } if (tokens > 0) { redis.decr(key) return true } return false // rate limited → return429 } // Limits: // - Message sending:30 msg/min per user // - Group creation:5 groups/hour // - File upload:100 MB/hour per user // - WebSocket connections:3 concurrent per user

WebSocket Abuse Protection

  • Max message size: 64KB (text), 100MB (media via presigned URL — never through WebSocket)
  • Connection timeout: 30s idle → disconnect. Heartbeat every30s keeps connection alive.
  • Reconnection backoff: Exponential: 1s, 2s, 4s, 8s, max30s. Prevents thundering herd on restart.
  • IP-based limiting: Max100 new connections/minute per IP. Prevents bot farms.

Spam & Abuse Prevention

  • URL scanning: Check links against Google Safe Browsing API on send. Block known phishing domains.
  • Message frequency analysis: ML-based spam detection. Flag accounts sending identical messages to many recipients.
  • Report mechanism: Users can block/report contacts. Reported accounts flagged for review.
  • Account verification: Phone number verification required. Limits anonymous abuse.

Content Moderation Under E2EE

  • Server cannot read content: E2EE means moderation relies on metadata and user reports.
  • Metadata analysis: Message frequency, connection patterns, group join/leave behavior.
  • User reports: Trigger manual review. Reported messages decrypted locally on device for review (with user consent).
  • Legal compliance: Content reported by users can be decrypted for review. Unreported content remains private.
Interview answer: "Rate limiting uses a token bucket in Redis —30 msg/min per user, with burst allowance of10 msg/s. For WebSocket abuse, I'd enforce30s idle timeout with heartbeat, and exponential backoff on reconnect. Content moderation under E2EE is challenging — the server can't read messages, so we rely on user reports and metadata analysis for abuse detection."

RTO/RPO Targets

Multi-Region Failover

  • Active-active: US, EU, APAC regions. Each has full chat server stack + local Cassandra.
  • Routing: GeoDNS directs users to nearest region. Failover on health check failure.
  • Cross-region: Async replication for user metadata. Messages local to sender's region.
  • Consistency: Eventual consistency for cross-region reads (5-second delay acceptable).

Failover Scenarios

// Scenario 1: Single chat server failure 1. LB detects health check failure (3s) 2. Clients disconnect, detect failure 3. Clients reconnect with exponential backoff (1s, 2s, 4s, max30s) 4. New server looks up presence in Redis 5. Pending messages delivered from RabbitMQ (un-acked messages redelivered) // Total recovery: ~30 seconds // Scenario 2: Full region failure 1. GeoDNS failover to nearest healthy region (30s TTL) 2. Users reconnect, authenticate with new region 3. Message history: cross-region Cassandra replication 4. Presence: rebuilt from client connections // Total recovery: ~2-5 minutes

Backup Strategy

  • Cassandra: Incremental backups every6 hours to S3. Point-in-time recovery enabled.
  • Redis: RDB snapshots every5 minutes + AOF for durability. Snapshots to S3.
  • RabbitMQ: Quorum queues with disk-based persistence. Messages survive broker restart.
  • Retention: Hot (0-7 days) in Cassandra, Warm (7-90 days) in S3 Standard, Cold (90+ days) in Glacier.
Interview answer: "For disaster recovery, I'd use active-active multi-region with GeoDNS. RPO is0 for messages (sync Cassandra replication) and5 minutes for presence. RTO is30 seconds for single server failure (client reconnect) and2-5 minutes for full region failover. The key insight is that WebSocket state is ephemeral — clients naturally reconnect, so the real recovery is just reconnecting them to a healthy server."

QuestionHow to Answer
How do you handle duplicate messages?"client_message_id with a unique constraint in PostgreSQL plus an LRU dedup cache in Redis. Since we use at-least-once delivery, duplicates are inevitable — idempotency is the solution, not prevention."

How does multi-device sync work?"Each device maintains its own WebSocket connection. When a message is sent, it goes to RabbitMQ and fans out to all devices of the recipient. Offline devices sync from last_read_at on reconnect. The client_message_id scheme works across devices — each device generates its own IDs."
How would you support message editing/deletion?"For editing, I'd add an edited_at column to the messages table and send a message_edit event over WebSocket. For deletion ('delete for everyone'), I'd set a deleted_at column — clients hide messages with deleted_at IS NOT NULL rather than actually deleting rows."
How do you handle media messages (images, voice)?"Media is uploaded to a separate service (S3 + CDN) via a presigned URL. The message only contains the URL and a thumbnail. The client renders the attachment from the URL. This keeps our message pipeline fast — we're only passing metadata."

Part 2 — Deep Dives & Advanced Topics

These sections provide the reasoning behind Part 1 decisions. Read them to understand why each choice was made — this depth is what separates good answers from great ones.

D1. Why RabbitMQ, Not Direct Delivery?

You might think: "If both clients are connected to chat servers, why not just look up the recipient's WebSocket and deliver directly?" That works for online delivery, but breaks down for:

  • Offline delivery: Recipient is disconnected — message must be queued.
  • Multi-device: Recipient on phone + desktop — one message must reach both.
  • Server crash: If the delivering server crashes mid-delivery, the message is lost. RabbitMQ persists it and redelivers to another server.
  • At-least-once guarantees: RabbitMQ's message acknowledgments give us automatic retry on failure.

Flow: Chat server A receives message → publishes to RabbitMQ → chat server B (or the same server) consumes → delivers to recipient's WebSocket → sends ack back to RabbitMQ.

D2. Partitioning the Messages Table

Messages is the fastest-growing table (5TB/year). Strategies:

  • By time (monthly): messages_2024_01, messages_2024_02. Easy to archive old partitions. Queries with time ranges go to the right partition.
  • By conversation_id hash: Spreads writes evenly. But scroll-back queries might fan out across shards.
  • Hybrid: Partition by month, then shard each partition by hash on conversation_id. Complex but scales horizontally.

Recommended for interview: "I'd start with time-based partitioning for the messages table — partition by month. Old partitions (older than 6 months) move to cold storage (S3 + Glacier). Active messages live in the hot partition, which I might shard further if write throughput exceeds a single node."

D3. Connection State vs Stateless Design

A common interview question: "WebSockets are stateful — how do you handle server restarts?"

Answer: Chat servers are designed to be mostly stateless regarding message data. The only state is the open WebSocket connection. All message data lives in RabbitMQ and PostgreSQL. If a server crashes:

  1. The load balancer detects the health check failure and stops routing to that server.
  2. Clients detect disconnection and reconnect (with exponential backoff).
  3. The new server looks up presence in Redis, subscribes to the user's message queue, and delivers pending messages from history.
  4. Messages that were in-flight on the crashed server remain un-acked in RabbitMQ and get redelivered to another consumer.

Result: The system tolerates server failures without losing messages. Users see a brief reconnection (1-2 seconds).

D4. At-Least-Once + Idempotency = Exactly-Once

True exactly-once delivery is very expensive (distributed transactions, two-phase commit). The practical approach is:

At-least-once delivery + idempotent consumers = effectively exactly-once.

Here's how it works for message storage:

// Consumer receives message from RabbitMQ
async function handleMessage(msg) {
    try {
        // Check dedup: has this client_message_id been processed?
        const exists = await dedupCheck(msg.client_message_id);
        if (exists) {
            channel.ack(msg);  // Already processed, just ack
            return;
        }

        // Insert into PostgreSQL (unique constraint catches duplicates)
        await db.insert('messages', {
            id: msg.message_id,
            conversation_id: msg.conversation_id,
            content: msg.content,
            client_message_id: msg.client_message_id
        });

        // Mark dedup
        await markProcessed(msg.client_message_id);

        // Deliver to recipient's WebSocket
        if (recipientOnline) {
            await wsSend(recipientSocket, msg);
        }

        channel.ack(msg);  // Only ack AFTER successful processing
    } catch (err) {
        channel.nack(msg);  // Reject — RabbitMQ redelivers
    }
}

If the consumer crashes between the DB insert and the ack, the message is redelivered. The dedup check catches it. The unique constraint on (sender_id, client_message_id) is the safety net.

Your context: Your RabbitMQ-based messaging already uses this pattern. The important addition for the interview is the dedup cache (in-memory LRU or Redis) — without it, every duplicate message hits PostgreSQL, which is slower.

D5. Handling Large Groups (Telegram Case)

For groups > 256, fan-out on write becomes expensive. Telegram uses a hybrid approach:

  • For groups ≤ 200: full fan-out on write.
  • For groups > 200: write once to a shared mailbox. Clients poll periodically.

Interview answer: "I'd start with fan-out on write for groups up to 256 — it keeps reads simple (no polling lag) and write overhead is manageable at our scale. If we needed to support mega-groups with 10K+ members, I'd switch to a shared stream model where clients poll for new messages."

D6. RabbitMQ vs Kafka for Chat

RabbitMQ advantages for chat:

  • Per-message acknowledgments — exactly what we need for at-least-once delivery
  • Dead-letter queues — failed messages go to a DLQ for inspection
  • Topic exchanges — route messages to the right consumer
  • Mature and simple — you already use it

Kafka advantages:

  • Higher throughput — millions of msg/s
  • Message replay — consumers can re-read from any offset
  • Built-in partitioning — messages with the same key go to the same partition (FIFO guaranteed)
  • Log compaction — could keep only the latest state per key

Recommendation: "At our scale (58K msg/s average, 116K peak), RabbitMQ with multiple queues per shard would work. If we grew 10x, I'd consider Kafka for its higher throughput and built-in partitioning. But RabbitMQ's per-message ack semantics are a better fit for our delivery guarantees."

D7. Message Search Under E2EE

A common follow-up: "How do users search their message history?" Under E2EE, this is architecturally constrained — the server holds only ciphertext, making server-side full-text search structurally impossible.

Client-Side Search (WhatsApp/Signal Approach)

// Client maintains local SQLite FTS5 index // Built incrementally as messages arrive and are decrypted // On message receive: 1. Decrypt message (E2EE happens on device) 2. Insert into local SQLite messages table 3. Update FTS5 index: INSERT INTO messages_fts VALUES (content) 4. Media: index via on-device OCR (images) or metadata (filenames) // On search query: 1. Query local FTS5: SELECT * FROM messages_fts WHERE content MATCH 'search term' 2. Return matching messages with highlights 3. No network call — instant results

Limitations

  • No cross-device search: Phone's index has no knowledge of desktop messages (unless synced)
  • No server-side search: Cannot search across all users (needed for compliance, support)
  • Storage cost: FTS5 index doubles local storage per device
  • Backup complexity: User must backup local DB to restore search after reinstall

The Architectural Fork

Interview insight: "If asked to 'add search to WhatsApp,' the correct answer starts with: 'This requires choosing between client-side only search with no cross-device support, or removing server-side E2EE for messages that need to be searchable.' That architectural fork is the answer. A standalone Elasticsearch cluster is not."

When Search is a Product Requirement

If the interviewer insists on server-side search, you must acknowledge the E2EE trade-off:

  • Option A: Disable E2EE for searchable messages (like Telegram's optional E2EE)
  • Option B: Search only metadata (sender, date, group) — not content
  • Option C: On-device indexing with periodic encrypted backup to server (user holds decryption key)
